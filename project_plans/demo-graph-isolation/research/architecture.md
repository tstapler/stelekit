## Q1: `GraphManager.addGraph()` / `switchGraph()` and `IN_MEMORY` Backend

### `addGraph(path)` — does NOT create a database

`addGraph` (lines 230–272) only:
1. Computes `graphId = sha256(canonicalPath).take(16)` via `ContentHasher`
2. Resolves `displayName` via `fileSystem.displayNameForPath`
3. Constructs a `GraphInfo` and inserts it into `_graphRegistry` StateFlow
4. Calls `saveRegistry()` to persist as JSON in `platformSettings`
5. Fires a `launch(IO)` coroutine for git-root detection

No database file is created in `addGraph`.

### `switchGraph(id)` — creates the database via lazy initialization

1. Creates a per-graph `CoroutineScope(SupervisorJob() + Default)`
2. Launches an IO coroutine that:
   - Calls `driverFactory.getDatabaseUrl(id.value)` → on JVM: `"jdbc:sqlite:~/.config/stelekit/stelekit-graph-{hash}.db"`
   - Constructs `RepositoryFactoryImpl(driverFactory, dbUrl, graphId)`
   - Calls `factory.createRepositorySet(backend = defaultBackend, ...)` — **this line creates the database**

### `IN_MEMORY` is clean for demo graphs — IF properly threaded through

In `RepositoryFactoryImpl`, the `database: SteleDatabase` property is `lazy`. When `backend == IN_MEMORY`, every `create*Repository()` call returns an `InMemory*Repository()` instance and **never accesses `database`**. No `driverFactory.createDriver()` is called, no SQLite file is written.

Telemetry driver creation is gated on `backend == GraphBackend.SQLDELIGHT && graphId != null` — also skipped for `IN_MEMORY`.

**The problem**: `switchGraph` is hardcoded to `backend = defaultBackend` (line 400:
`createRepositorySet(backend = defaultBackend, ...)`). There is no per-graph override. To use
`IN_MEMORY` for demo graphs, `switchGraph` must inspect `graphInfo.isDemo` and substitute
`GraphBackend.IN_MEMORY` when true. No other structural change is needed.

## Q2: `GraphLoader.loadGraph()` on JVM — Cannot Read Classpath Directories

### The load path

`loadGraph(graphPath)` (line 490) does:
1. `fileSystem.directoryExists(graphPath)` — early return if missing
2. Constructs `"$graphPath/pages"` and `"$graphPath/journals"` paths
3. Calls `sanitizeDirectory(pagesDir)` — calls `fileSystem.listFiles(path)` internally
4. Calls `loadDirectory(pagesDir, ...)` and `loadDirectory(journalsDir, ...)` in parallel

`loadDirectory` (line 1116) calls `fileRegistry.scanDirectory(path)` which in turn calls
`fileSystem.listFiles(path)` — on JVM this uses `java.io.File.listFiles()`.

### The classpath problem

`kmp/src/commonMain/resources/demo-graph/` contains the complete demo content (pages/,
journals/ subdirs with `.md` files). On JVM these resources are:
- Readable via `getResourceAsStream("demo-graph/pages/About SteleKit.md")`
- **Not listable** via `java.io.File.listFiles()` when packaged inside a JAR

Running from Gradle (development) the resources are unpacked to the class output directory,
so `File.listFiles()` would work — but this fails in a distributed app.

### Required approach: JvmDemoFileSystem shim

The WASM platform already solved this with `DemoFileSystem` (generated by
`:kmp:generateDemoFileSystem` from the resource files). The same pattern works on JVM:

**Option A (recommended)**: Generate a `jvmMain` `JvmDemoFileSystem` class with the same
generator task. The generator already outputs a `FileSystem` implementation that hardcodes file
contents as string literals and provides `listFiles()` / `readFile()` without touching disk.
The generator would emit a second output file (or the single output could be in `commonMain` as
a class that reads via `Res.readBytes()` — KMP compose-resources). This is consistent with the
existing WASM pattern and requires no runtime temp-dir extraction.

**Option B (simpler, but impure)**: On JVM, extract classpath resources to a system temp
directory at demo-graph creation time, then point `graphPath` at the temp directory. This works
with the existing `FileSystem` impl but leaves temp files on disk and is harder to clean up.

**Option C**: `expect/actual` `ClasspathDemoFileSystem` — `commonMain` declares the interface,
`jvmMain`/`wasmJsMain` each provide the actual. The WASM actual is the existing `DemoFileSystem`;
the JVM actual reads via `this.javaClass.classLoader.getResourceAsStream(...)` and uses a
manifest of known paths (generated or hardcoded).

**Recommendation**: Extend the `:kmp:generateDemoFileSystem` Gradle task to emit a second file in
`jvmMain` named `JvmDemoFileSystem`. It implements `FileSystem` exactly like `DemoFileSystem` but
replaces date-dynamic journal generation with a fixed "today" journal seeded from the resource
content. The virtual `graphPath` used for demo on JVM should be a sentinel like `"/demo"` that
can never match a real filesystem path.

## Q3: `loadRegistry()` / `saveRegistry()` — Serialization Safety and Strip Logic

### Storage mechanism

Registry is persisted as a JSON string via:
```kotlin
platformSettings.getString("graph_registry", "")
platformSettings.putString("graph_registry", json.encodeToString(registry))
```

The `Json` instance uses `ignoreUnknownKeys = true` (line 61 of GraphManager.kt).

### Adding `isDemo: Boolean = false` — backward-compatible

Because the `Json` instance ignores unknown keys AND `isDemo` has a default value:
- **Deserializing old JSON** (no `isDemo` field): works, defaults to `false`
- **Serializing new JSON**: includes `isDemo: false` for real graphs, `isDemo: true` for demo
- **Old app reading new JSON**: `ignoreUnknownKeys = true` silently drops `isDemo` — safe

No migration required. The addition is a non-breaking schema change.

### Display name refresh in `loadRegistry()` — one caution

`loadRegistry()` iterates every graph and calls `fileSystem.displayNameForPath(graph.path)`
(line 113) to refresh stale display names. For a demo graph with `path = "/demo"`, this call
must return `"Demo Graph"` (or any non-crashing string). On JVM, `displayNameForPath` resolves
via `java.io.File("/demo").name` → returns `"demo"`, which is fine. The `isDemo = true` flag
or the hardcoded `displayName = "Demo Graph"` in `addDemoGraph` will take precedence anyway
since `loadRegistry` only updates if `freshName != graph.displayName`.

**However**: after strip logic is added, `loadRegistry()` won't see demo entries at all on
next launch, so this is moot.

### Strip logic location

The stripping should happen inside `loadRegistry()` immediately after deserialization, before
setting `_graphRegistry.value`:

```kotlin
val stripped = refreshed.copy(
    graphs = refreshed.graphs.filter { !it.isDemo },
    activeGraphId = if (refreshed.graphs.firstOrNull { it.id == refreshed.activeGraphId }?.isDemo == true)
        null else refreshed.activeGraphId
)
```

This ensures:
1. Demo graphs never survive a restart
2. If the demo was active when the user closed the app, the `activeGraphId` is cleared to null
   so the user sees the graph picker / onboarding screen, not a broken load

## Q4: Exact Path from "Load Demo Graph" Click to Content on Screen

1. **`Onboarding.kt:162-164`** — Click handler hardcodes `demoPath = "deps/graph-parser/test/resources/exporter-test-graph"` and calls `onGraphSelected(demoPath)`

2. **`App.kt:1151-1162`** — `onGraphSelected` lambda:
   - `graphManager.addGraph(path)` → creates `GraphInfo`, persists to `platformSettings`, fires git detection coroutine, returns `graphId`
   - `viewModel.setGraphPath(path)` → updates `_uiState.currentGraphPath` and immediately calls `viewModel.loadGraph(path)`
   - `graphManager.switchGraph(graphId)` → launches IO coroutine: `RepositoryFactoryImpl.createRepositorySet(backend = SQLDELIGHT)` → **creates `~/.config/stelekit/stelekit-graph-{hash}.db` on disk**

3. **`StelekitViewModel.loadGraph(path)`** → `graphLoader.loadGraphProgressive(path, ...)` → `fileSystem.directoryExists("deps/graph-parser/test/resources/exporter-test-graph")` — succeeds only if run from the project root

4. **`GraphLoader.loadGraphProgressive`** → `loadDirectory("$path/pages", ...)` + `loadDirectory("$path/journals", ...)` → `fileSystem.listFiles(...)` → reads each `.md` file → feeds parsed blocks/pages through `pageRepository`/`blockRepository`

5. **Compose UI** — `LaunchedEffect` in `GraphContent` collects from the repository `Flow`s; once `switchGraph` sets `_activeRepositorySet.value`, the UI re-renders with graph content

**Where the SQLite file is created**: Step 2, inside `switchGraph`'s IO coroutine, when `RepositoryFactoryImpl.database` lazy is first accessed by `createBlockRepository(SQLDELIGHT)`. The file path is `driverFactory.getDatabaseUrl(graphId.value)` = `"jdbc:sqlite:{dbDir}/stelekit-graph-{hash}.db"` where `dbDir` is `~/.local/share/stelekit` (Linux JVM).

With the proposed `addDemoGraph()` fix, step 2 uses `IN_MEMORY` backend: `database` lazy is never accessed, no file is created, and no JDBC URL is opened.

## Q5: Which Other `GraphManager` Methods Need `isDemo` Awareness

### `switchGraph` — must check `isDemo`

Must substitute `GraphBackend.IN_MEMORY` for demo graphs:
```kotlin
val backend = if (graphInfo.isDemo) GraphBackend.IN_MEMORY else defaultBackend
val repoSet = factory.createRepositorySet(backend = backend, ...)
```

### Git detection coroutine — must skip

`addGraph` fires `launch(IO) { detectGitRoot(expandedPath) }` unconditionally. For `addDemoGraph`, skip this: a virtual path like `"/demo"` has no git root. If `addDemoGraph` bypasses `addGraph` entirely (recommended), this is automatic.

### `checkGitignoreForDatabase` — must skip

Called inside `addGraph`'s `withContext(IO)` block. For demo graphs, no real directory exists; `fileSystem.fileExists("$graphPath/.gitignore")` would either return false (benign log) or throw. Skip by not calling `addGraph` from `addDemoGraph`.

### `renameGraph` — should guard against demo graphs

`renameGraph` updates `displayName` in the registry and persists. For demo graphs this is useless (stripped on next launch) and misleading. Add a guard:
```kotlin
if (registry.graphs.firstOrNull { it.id == id }?.isDemo == true) return false
```

### `removeGraph` — OK as-is

Removes by ID from the registry; does not touch the filesystem. Works correctly for demo graphs.

### `loadRegistry()` — needs strip logic (see Q3)

### `saveRegistry()` — no change needed

Demo entries can be written; they'll be stripped on next load. Alternatively, filter `isDemo = true` on save, but that's redundant given the load-time strip.

### `MigrationRunner` inside `switchGraph` — must skip for `IN_MEMORY`

Already gated: `if (defaultBackend == GraphBackend.SQLDELIGHT)` (line 411). With the fix making demo use `IN_MEMORY`, this block is skipped automatically — `writeActor` is null for `IN_MEMORY` anyway.

### `UuidMigration` inside `switchGraph` — also already gated

Same guard: only runs when `defaultBackend == GraphBackend.SQLDELIGHT`.

## WASM Workaround Removal

`kmp/src/wasmJsMain/kotlin/dev/stapler/stelekit/browser/Main.kt` lines 117–121:
```kotlin
if (existingRegistry.contains("/demo")) {
    localStorage.removeItem("graph_registry")
}
```
This checks for stale `/demo` paths as a string match on the raw JSON. Once `isDemo = true` is added and `loadRegistry()` strips demo entries at load time, the registry will never have stale demo entries after any session. The workaround can be deleted — but only after `loadRegistry()` strip logic ships, to avoid a race condition where the WASM app loads the stale registry before the strip code runs.

## Implementation Sketch

### `GraphInfo.kt` change
```kotlin
data class GraphInfo(
    val id: GraphId,
    val path: String,
    val displayName: String,
    val addedAt: Long,
    val isParanoidMode: Boolean = false,
    val detectedRepoRoot: String? = null,
    val detectedWikiSubdir: String? = null,
    val gitDetectionDismissed: Boolean = false,
    val isDemo: Boolean = false,           // NEW — never survives a restart
)
```

### `GraphManager.addDemoGraph()` — new method (does NOT call `addGraph`)
```kotlin
suspend fun addDemoGraph(): GraphId {
    val demoId = GraphId("demo")
    val info = GraphInfo(
        id = demoId,
        path = "/demo",
        displayName = "Demo Graph",
        addedAt = Clock.System.now().toEpochMilliseconds(),
        isDemo = true,
    )
    val registry = _graphRegistry.value
    if (!registry.graphIds.contains(demoId)) {
        _graphRegistry.value = registry.copy(graphs = registry.graphs + info)
        // Do NOT call saveRegistry() — demo graph must not persist
    }
    return demoId
}
```
Then `switchGraph(demoId)` uses `IN_MEMORY` (after the `isDemo` check is added).

### `GraphManager.loadRegistry()` addition
```kotlin
val stripped = refreshed.copy(
    graphs = refreshed.graphs.filter { !it.isDemo },
    activeGraphId = if (refreshed.graphs.firstOrNull { it.id == refreshed.activeGraphId }?.isDemo == true)
        null else refreshed.activeGraphId
)
_graphRegistry.value = stripped
```

### JVM content seeding

Option: extend `:kmp:generateDemoFileSystem` to also emit `jvmMain/.../JvmDemoFileSystem.kt`
with the same `FileSystem` interface. `addDemoGraph()` on JVM wraps the `GraphLoader` with this
`FileSystem` shim for the initial `loadGraph("/demo")` call instead of the platform `FileSystem`.

Simpler option (if demo is truly read-only on JVM): pass `JvmDemoFileSystem()` as the
`fileSystem` argument when constructing the `GraphLoader` for the demo graph only. This requires
`GraphLoader` to be constructable with a different `FileSystem` than the rest of the app — it
already accepts `fileSystem` as a constructor parameter (via `RepositorySet.createGraphLoader`),
so this is clean.

## Summary of Key Invariants

| Concern | Current state | Required change |
|---|---|---|
| SQLite file created for demo | Yes (SQLDELIGHT default) | `isDemo` → `IN_MEMORY` in `switchGraph` |
| Demo persisted in registry | Yes | Strip `isDemo=true` in `loadRegistry()` |
| Demo auto-opens on restart | Yes | Strip clears `activeGraphId` |
| Relative path stored | Yes | Use `/demo` virtual path + don't save |
| Display name unfriendly | Yes (`displayNameForPath` returns dir name) | Hardcode `"Demo Graph"` in `addDemoGraph()` |
| WASM `contains("/demo")` workaround | Yes, Main.kt:118 | Remove after strip logic ships |
| JVM classpath content seeding | Not addressed | `JvmDemoFileSystem` shim needed |
