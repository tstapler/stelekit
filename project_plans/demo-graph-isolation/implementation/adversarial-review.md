# Adversarial Review: demo-graph-isolation
**Date**: 2026-07-05
**Verdict**: CONCERNS

---

## Blockers

- [x] **RESOLVED — `switchGraph()` unconditionally calls `saveRegistry()` (line 446), persisting the demo entry** — Original concern: `switchGraph(DEMO_GRAPH_ID)` at line 446 unconditionally calls `saveRegistry()`, serializing the demo `GraphInfo` and `activeGraphId = "__demo__"` into `platformSettings["graph_registry"]`. Task 2.1.1b fully resolves this: filtering in `saveRegistry()` before the `putString` call strips all `isDemo == true` entries and nulls out `activeGraphId` when it equals `DEMO_GRAPH_ID`. Grep confirms there is exactly ONE `platformSettings.putString("graph_registry", ...)` site in the codebase (line 224 of `GraphManager.kt`, inside `saveRegistry()`); all 11 call sites route through this single function, so no bypass paths exist. The fix covers the `switchGraph()` callsite and every other callsite automatically. `loadRegistry()` stripping (planned as defense-in-depth for production databases written before this fix) is not redundant — it handles existing dirty registries — and is not broken by 2.1.1b. Acceptance criterion met once Task 2.1.1b is implemented: `platformSettings.getString("graph_registry", "")` will not contain `"__demo__"` even after `switchGraph(DEMO_GRAPH_ID)` runs.

## Concerns

- [ ] **WASM demo-fallback path (lines 75–107 of `Main.kt`) is not updated to use `addDemoGraph()`** — The plan only removes lines 117–121 (the else-branch path-substring workaround). The fallback branch itself (lines 96–107) still persists `lastGraphPath = graphPath` (the demo path from `DemoFileSystem.getDefaultGraphPath()`) via `settings.putString("lastGraphPath", graphPath)` at line 104. It never calls `addDemoGraph()`, so `graphManager.getActiveGraphInfo()?.isDemo` is always `false` during WASM fallback sessions. Consequences: (a) `DemoBanner` will not show, (b) the active graph pill will not say "Demo Graph", (c) the `effectiveFileSystem` derivation in Phase 4.1.3a relies on `activeGraphInfo?.isDemo == true` and will not fire — but this is harmless because `fileSystem` is already `DemoFileSystem()` at the callsite. The concern is that the success metric "Demo appears as 'Demo Graph' in registry" is unmet for WASM fallback sessions, and `lastGraphPath = "/demo"` is persisted to settings. The transition to OPFS (when it becomes available on the next load) is safe because line 116 of the OPFS branch overwrites `lastGraphPath` before `GraphManager.init` runs, but the WASM fallback experience diverges from the JVM demo experience without acknowledgment. Recommend: either update the WASM fallback to call `addDemoGraph()` after `GraphManager` construction, or document the divergence explicitly as accepted scope.

- [ ] **`_graphRegistry.value` read-modify-write in `addDemoGraph()` is not atomic** — The idempotency check pattern is `val registry = _graphRegistry.value; if (!contains) _graphRegistry.value = registry.copy(...)`. `MutableStateFlow` does not provide CAS semantics for direct `value =` assignment; two concurrent callers that both pass the `contains` check before either updates `value` will produce duplicate demo entries. The existing `addGraph()` has the same pattern (line 254–261), so this is not a new anti-pattern, but `addDemoGraph()` is the first function where the same ID can be registered multiple times without harm and the idempotency check is the only guard. `_graphRegistry.update { r -> if (!r.graphIds.contains(demoId)) r.copy(...) else r }` uses `StateFlow.update`'s internal CAS and is the correct fix.

- [ ] **`graphWriter` still uses the original `fileSystem`, not `effectiveFileSystem`** — Phase 4.1.3a updates `graphLoader`'s `remember` key and argument to use `effectiveFileSystem`. But `graphWriter = remember(fileSystem, repos, graphLoader, sidecarManager) { ... fileSystem ... }` (App.kt line 565–567) still receives the platform `fileSystem`. When a user edits a block in the demo session, `BlockStateManager` debounces and calls `GraphWriter.saveBlock()`, which attempts a write to `fileSystem.writeFile("/demo/pages/...", ...)`. On JVM, `/demo` does not exist; the write will fail. `IN_MEMORY` backend keeps the in-memory state correct so the user sees their edit, but `GraphWriter` failures may emit error logs or trip any `onError` callback wired to the writer. The `DemoBanner` says "changes won't be saved," so the behavior is disclosed, but the write errors are an uncontrolled failure path not mentioned in the plan. Recommend updating the `graphWriter` `remember` block to use `effectiveFileSystem` the same way `graphLoader` is updated.

- [ ] **`onboardingCompleted` is reset on every demo-only restart, indefinitely** — `loadRegistry()` sets `onboardingCompleted = false` whenever stripping leaves `graphs.isEmpty()`. A user who loaded the demo, ran through the app, quit, and relaunches sees onboarding again — every time, until they pick a real graph. The requirements say "app opens to onboarding or real graph — never to demo," which matches this, but the UX implication is that a new user exploring the app feels punished for trying the demo. There is no "Continue without saving" path from the demo to the main app without going through onboarding again. The plan does not acknowledge this UX loop. Recommend adding a requirement or explicit out-of-scope annotation, and consider whether `setOnboardingCompleted(true)` (Phase 4.1.2a) should be conditional on the user completing the full onboarding flow rather than triggering immediately on demo selection.

## Minors

- The plan's Risk Control table states "DemoFileSystem is stateless; recreation is safe." This is incorrect — `DemoFileSystem` has a mutable `private val overrides = mutableMapOf<String, String>()` field for session edits. In practice, recreation does not occur within a demo session because `activeGraphInfo` is frozen via `remember` without keys, so `remember(activeGraphInfo?.isDemo)` is effectively `remember` (no key). The behavior is correct, but the stated rationale is wrong and could mislead reviewers.

- Story 5.1.2's suggested `isDemoActive` computation in the plan text is `availableGraphs.size < (availableGraphs + demo).size` — this is nonsensical. The plan's own inline comment says "or simply compute from `activeGraphInfo?.isDemo` in `LeftSidebar`," which is correct. The bad suggestion in the task text should be removed before implementation to avoid confusion.

- `removeGraph()` has no `isDemo` guard. While the active-graph check (`registry.activeGraphId == id → return false`) prevents removal while demo is active, after a graph switch the demo entry would remain in `_graphRegistry.value` (in-memory) and could theoretically be removed. `saveRegistry()` would then be called with the demo removed from the in-memory registry — harmless, since `loadRegistry()` would have already stripped it on the next launch. The symmetry with `renameGraph()` suggests a guard should exist for consistency, even if the code path is unreachable in the current UI.

- The phase sequencing in the dependency diagram shows Phase 6 (WASM cleanup) after Phase 5 (UX). If development is done incrementally, Phase 6 should move earlier (after Phase 2) to avoid any window where both the old workaround and the new `loadRegistry()` stripping are active simultaneously — double-stripping is harmless, but it is unnecessary complexity during the development window.
