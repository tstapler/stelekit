# UX Research: web-git-writeback

**Agent**: Research Agent 5 (UX Research)
**Date**: 2026-07-14
**Scope**: UX implications of implementing `WasmGitWriteService` per ADR-015, closing BUG-005 Phase 2.

## 0. What already exists (read before designing anything new)

Per `ADR-015`'s explicit rationale ("the conflict resolution screen, sync status badge, and commit flow are shared Compose UI... no WASM-specific UI branches are needed"), this feature is UI-*reuse*, not UI-*creation*. The existing components are more mature than the requirements doc implies:

|---|---|---|

Success is a 3-second fading green checkmark; Error routes taps to `onAuthError` when the error is `AuthFailed`; conflict shows an amber "Conflict" chip. |
| Conflict resolution | `ui/screens/git/ConflictResolutionScreen.kt` | Per-file "Keep mine" vs "Use remote" `FilterChip`s, an explanatory subtitle already added ("A background sync pulled remote changes that conflict with your local edits"), an "Abort merge" confirmation dialog that preserves local changes. |
| Algorithmic (journal) merge review | `ui/screens/git/JournalMergeReviewScreen.kt` | **This is the load-bearing precedent for Q3 below.** Even *algorithmically successful* journal merges are NOT applied silently — they route to a 3-panel LOCAL / MERGED (editable) / REMOTE review screen before commit, with warning banners for low-confidence merges or unresolved conflict markers. |

**Design implication**: this project's UX work is narrower than "design a sync UI" — it is (a) confirm the existing `SyncState`/error taxonomy actually covers every failure mode `WasmGitWriteService` can produce (it currently does not — see Section 4), (b) decide what's *web-specific* about credential entry and the "don't close this tab" risk that desktop never had, and (c) decide whether silent non-overlapping auto-merge (new to this project, not journal-merge) should get its own visible confirmation, following the journal-merge precedent.

## 1. Comparable UX patterns (browser apps writing to a git remote)

Sampled from the class of tools SteleKit's web write-back most resembles: browser-based Markdown/wiki editors backed by a GitHub repo (GitHub's own web editor + GitHub Actions-based static site CMSes, Netlify CMS / Decap CMS, TinaCMS, Forestry, Obsidian Git-style community plugins, CodeSandbox/StackBlitz's GitHub sync, Google Docs' offline/reconnect model as an analogy for the local/remote split even though it's not git-backed).

**(a) Sync status display** — the converged pattern across these tools is a **persistent, small, always-visible status chip** (not a modal, not a toast that disappears before being read) with exactly the states SteleKit already models: idle/saved, syncing (spinner), unsynced/dirty (a distinct visual state — often a filled dot or "N unsaved changes"), error (red, actionable), and conflict (amber, actionable). Decap/Netlify CMS and TinaCMS both show a "Saving…" → "Saved to \<branch\>" transition distinct from "Publishing…" → "Published", i.e., they distinguish *local draft save* from *remote publish* as two visually separate steps — directly analogous to SteleKit's OPFS-save vs. git-push split (see Section 2). GitHub's own in-browser editor shows "Uncommitted changes" only at the point of commit, because GitHub's web editor has no separate local-save step; it is *not* a good analog for SteleKit's autosave-then-later-sync model.

**(b) "Don't close this tab" warning** — the standard mechanism is `window.beforeunload` (or the Kotlin/WASM equivalent) firing a native browser "leave site?" confirmation, gated strictly on a boolean "has unsynced work" flag, not on every edit. Overuse (firing when there's nothing at risk) trains users to reflexively dismiss it, which defeats the point the first time it matters. Tools that get this right key the warning off the *same* dirty-set the sync engine itself tracks (in SteleKit's case, the OPFS dirty-file set ADR-015 specifies) — never a separate, potentially-stale "hasUnsavedChanges" flag. Some tools (Notion, Google Docs) additionally show a small icon-level indicator ("saving…", cloud-with-slash for offline) so the beforeunload dialog is not the user's *first* signal — it should be the last-resort backstop, not the primary communication channel.

**(c) PAT entry UX** — converged pattern: masked input by default with a visibility toggle (SteleKit already has this), inline text naming the exact minimum scopes required (typically `repo` for GitHub classic PATs, or the specific fine-grained-PAT repository permissions: Contents read/write), and a **direct link that pre-fills the token-creation form** with the right scopes/expiration pre-checked (GitHub supports this via URL parameters on `github.com/settings/tokens/new?scopes=repo&description=...`; GitLab's `personal_access_tokens/new` supports `scopes[]` query params similarly). SteleKit's current Step 3 names "Personal access token" with no scope guidance and no deep link — this is a real gap worth closing since it's a one-line addition with outsized trust payoff (see Section 5, emotional job). Best-in-class tools also validate the token against a lightweight API call (e.g., `GET /user`) before letting the user proceed, distinct from SteleKit's existing "Test connection" step which already does something adjacent (a `fetch`) — confirm in planning whether "Test connection" surfaces a *scope* problem (401/403 due to missing `repo` scope) distinctly from a *URL/network* problem, since users cannot tell those apart from a generic failure message.

## 2. User mental model: two decoupled "save" steps

Desktop SteleKit's mental model is single-step: edit → debounced write to markdown file on disk (500ms, per `CLAUDE.md`'s data flow) → that file *is* the source of truth, and git sync (when configured) is a distinct, occasional, explicit action layered on top, but even without git configured the user's data is durably on their own filesystem. **Losing power here is the crux of the UX problem this project must solve for**: on web, "saved" (SQLite/OPFS, effectively instant, but browser-sandboxed storage with no guarantee of surviving a cleared cache, private-browsing session end, or storage-eviction under disk pressure) and "synced" (pushed to the user's actual GitHub/GitLab repo, the only representation that survives browser data loss) are not just two steps — they differ in *durability class*, not just latency. A desktop user has never had to reason about this distinction because local disk and "the file" were the same thing.

The UI must not reuse language that implies parity between these two states. Concretely:
- Do not show a generic "Saved" checkmark for the OPFS write the way desktop implicitly treats a debounced disk write as final — on web that checkmark should read something more like "Saved to this browser" or use a visually distinct (not checkmark-green) treatment, reserving the unambiguous "safe" signal (checkmark, or the existing `SyncState.Success` green check) for the state *after* the remote push succeeds.
- The existing `SyncStatusBadge`'s `Idle` state renders *nothing* (per its own doc comment: "No visible badge when idle"). On desktop this is correct — idle means fully synced and there's nothing to say. On web, reusing `Idle`-renders-nothing for "OPFS-saved but not yet pushed, and git isn't syncing right now" would actively mislead a user coming from the desktop mental model into believing "no badge" still means "fully safe." This is the single highest-leverage UX decision in this project: **web needs a state (or a repurposed use of an existing one, e.g., `MergeAvailable`'s sibling for "local changes not yet pushed") that is visible whenever the OPFS dirty-set is non-empty**, distinct from `Idle`. Flag this explicitly for the planning phase — it may require a `SyncState` addition (e.g., `LocalChangesPending(fileCount: Int)`) since none of the current 12 states cleanly represents "dirty but not currently mid-operation."
- Naming: avoid the word "save" for the git operation entirely in UI copy — use "sync," "push," or "back up to GitHub" so the two concepts never share a verb. This echoes the Decap/TinaCMS convention of "Saving draft" vs. "Publishing."

## 3. Accessibility

**Sync-status indicator (badge + button)**:
- The existing `SyncStatusBadge`'s icon-only regions already ship `contentDescription` (verified in source — e.g., `"Sync now"`, `"Merge conflict"`, `"Vault locked — tap to unlock"`) — this is good and should be preserved for any new state added for Section 2's "local changes pending" gap.
- Compose's default `IconButton`/`Icon` semantics expose these as accessible names, but the **state changes need to be announced**, not just labeled — a screen-reader user tabbing past a static icon after a sync completes in the background won't know it changed unless the region is live. Recommend wrapping the badge's dynamic text/icon in a Compose `liveRegion` semantics modifier (`Modifier.semantics { liveRegion = LiveRegionMode.Polite }`) so state transitions (idle → syncing → success/error) are announced without requiring focus. This is not currently present in `SyncStatusBadge.kt` — worth a planning-phase decision on whether to add it as part of this project or file as a fast-follow, since it's arguably pre-existing debt this project's "local changes pending" state addition would inherit either way.
- Keyboard: the manual "Sync now" `IconButton` is already a standard focusable/activatable target (Enter/Space) — no gap there. The clickable badge regions (`ConflictPending`, `Error`, `CredentialExpired` — built with `Modifier.clickable` on a `Row`, not an `IconButton`) should be checked for proper `Role.Button` semantics and keyboard-activation support; a bare `clickable` on a `Row` gets tap/click but needs explicit `role = Role.Button` merged into semantics to read correctly and respond to Enter/Space via a screen reader or keyboard-only user. Verify during implementation.
- Any new "Retry sync" affordance should follow the same `contentDescription` + `Role.Button` pattern, with a description that states the action, not just repeats the icon's meaning (e.g., "Retry sync" not "Error icon").

**Conflict resolution screen**: the per-file `FilterChip` pair ("Keep mine" / "Use remote") already carries leading icons with `contentDescription = null` (decorative, correct, since the adjacent `Text` label already names the choice) — good. For the **auto-merge-vs-manual-resolve distinction this project introduces**: if auto-merge stays silent (see Q3 answer below, recommendation is *don't* keep it fully silent), whatever surfaces it needs the same treatment `JournalMergeReviewScreen` gets — a dismissible, non-blocking summary, not a screen the user must click through, since by definition nothing needs *deciding* for the non-overlapping case. A toast/snackbar-class or badge-class disclosure (not a full screen) is the right accessibility-and-friction tradeoff — it should be discoverable (announced via the same live-region pattern as Section 3's status badge) but not force a modal interaction for a case where, per `ADR-015`, no user decision is required.

**Answering "silent-and-correct, or visible?"**: silent-and-correct is the *action* (no user decision should be forced), but **not silent-and-unauditable**. The codebase's own `JournalMergeReviewScreen` precedent (Section 0) already establishes that this team's convention is to surface merge outcomes even when they required no manual resolution, specifically because merge correctness is exactly the kind of thing a user needs to be able to check *after the fact* if something looks wrong on the remote. Given this project's origin bug ("my data isn't reaching the external store and I can't tell" — see Section 5), a fully silent auto-merge reproduces the same failure mode this project exists to fix, just one layer deeper: the merge could theoretically not include everything the user expects, and there'd be no trail. Recommend: log every auto-merge (already required per Observability Requirements) AND make it visible via a low-friction UI trace — e.g., a brief non-blocking `SyncState.Success`-style toast that says "Synced (2 files auto-merged with remote changes)" rather than reusing the plain "Synced" success state indistinguishably from a conflict-free push. This is a `SyncState` shape decision for planning (likely: add an `autoMergedCount` field to `Success`, or a distinct state) rather than a new screen.

## 4. Error states

Mapping each requested error state to the current `SyncState`/`GitError` model (`git/model/SyncState.kt`, `error/DomainError.kt`) and flagging gaps `WasmGitWriteService` will newly produce:

| Error state | Existing coverage | Gap for this project |
|---|---|---|
| **PAT invalid/expired mid-session** | `SyncState.CredentialExpired(graphId)` + `GitError.CredentialExpired` exist, with copy "GitHub authentication expired — tap to re-connect" and a tap-through to re-auth. | Covered structurally, but this copy currently reads as OAuth-specific ("re-connect"). For `WasmGitWriteService`'s PAT flow (not OAuth device-flow necessarily — web users may also use raw HTTPS PAT per `GitSetupScreen` Step 3), confirm the re-auth tap target opens the *PAT field*, not an OAuth flow that may not apply. A mid-session 401 from the Git Data API (e.g., PAT revoked between page load and push attempt) should route here, not to the generic `Error`/`AuthFailed` state, which today reads "check your credentials" — less actionable than the dedicated expired-state's "tap to re-connect." |
| **Rate-limited by GitHub/GitLab** | **No dedicated state exists.** `GitError` has no `RateLimited` variant; it would currently fall through to `PushFailed`/`FetchFailed`/`CommitFailed` with a generic "tap to retry" message. | Real risk per `ADR-015`'s own text (60 req/hr unauthenticated, 5000/hr authenticated — a write session touching many files across the 5-step GitHub sequence could plausibly hit secondary rate limits even authenticated, e.g., GitHub's abuse-detection on rapid sequential writes). Recommend a new `GitError.RateLimited(retryAfterSeconds: Int?)` surfaced distinctly: "GitHub rate limit reached — retrying in Xs" with an automatic backoff-retry rather than a manual "tap to retry" (the user tapping immediately will just hit the limit again). This is a planning-phase model addition, not just a copy string. |
| **Network offline during sync** | `GitError.Offline` / copy "Offline — sync will resume when connected" already exists and already implies auto-resume framing (good — matches the pattern in Section 1a). | Confirm `WasmGitWriteService` actually distinguishes "browser reports `navigator.onLine = false`" from "request failed for other reasons" — a generic fetch failure on web can look identical to offline from the client's perspective (DNS failure, CORS misconfiguration, host down) but should not all collapse into the reassuring "will resume when connected" copy if the real cause won't self-resolve. |
| **Partial-failure (some files pushed, sequence failed before ref update)** | **No dedicated state.** This is the "5-step sequence is not atomic" rabbit hole `ADR-015`/requirements.md both flag explicitly. Per ADR-015 Rationale #2, a failure here leaves an *orphaned commit object* (not referenced by any ref) — from the git remote's perspective as observed by GitHub's UI, **nothing changed**; the failure is invisible on the remote, which is actually the safe outcome (no partial/corrupt state visible to collaborators). | The UI-relevant question is purely local: does the client correctly report this as a plain retry-safe failure (since retry re-derives the dirty set from scratch per the ADR) rather than something scarier like "some of your files may have partially synced"? Recommend copy explicitly reassures: "Sync failed partway through — nothing was changed on GitHub. Your local changes are safe; sync will retry." This directly serves the emotional job in Section 5 and should map to the existing `PushFailed`/`CommitFailed` states with this specific copy, not a new state — but copy must not imply partial application occurred remotely, since per the ADR it structurally cannot (only orphaned unreferenced objects, invisible to the user and eventually GC'd). |
| **Successful auto-merge** | See Section 3's answer — recommend surfaced via an enriched `Success` state, not silent. | Same as Section 3. |

**One more gap worth flagging for planning**: none of the current `GitError` variants distinguish "GitHub" from "GitLab" in copy (reasonable, since the desktop/Android path is host-agnostic today) — but `WasmGitWriteService` is the first implementation where GitHub and GitLab take genuinely different code paths (5-step vs. single-call). If a GitLab-specific failure mode surfaces (e.g., the `start_branch` conflict-detection parameter behaving differently than GitHub's ref-SHA check), confirm error copy doesn't accidentally say "GitHub" to a GitLab user or vice versa — check whether existing copy is host-neutral by construction (it currently is: "tap to retry," "Authentication failed," etc. — no hardcoded host name found) and keep it that way.

## 5. Jobs-to-be-done

**Functional job**: "Get my notes backed up to my own git repo without leaving the browser tab I'm already editing in." Success is invisible-by-default (sync just works in the background per the poll-interval/manual-trigger pattern already established for JVM/Android) — the user should not need to think about *this* project's existence when things are working. The functional job is fully satisfied by ADR-015's decision as specified; no UX gap here beyond what's covered in Sections 2–4.

**Emotional job — the load-bearing one for this project**: trust that data isn't silently lost. This is not a generic nicety; it is *literally the bug that started this entire investigation* (per requirements.md's Problem Statement: "pages save to SQLite but never reach an external markdown store... for web users syncing against a git remote, there is currently no way to get local edits back out at all"). Every UX decision in Sections 2–4 should be evaluated against a single test: **would this design have made the original bug detectable to the user sooner?** Under that lens:
- Section 2's "don't let `Idle` mean the same thing on web that it means on desktop" is the single most direct mitigation — the original bug's failure mode was precisely "the UI looked idle/fine while nothing was actually reaching the remote."
- Section 4's insistence on not overloading generic retry copy for rate-limiting and partial-failure cases exists so that a *real*, unresolvable failure doesn't get camouflaged as a transient, self-resolving one — camouflaging failure as normalcy is exactly how the original bug went unnoticed.
- Section 3's "silent-and-correct but not silent-and-unauditable" for auto-merge exists for the same reason at one further remove: even *successful* operations should leave a trace a suspicious user can go check, because the cost of an invisible failure mode (an incorrect auto-merge silently dropping content) is a second instance of the same class of bug this project is closing out.
- The PAT-scope-and-deep-link gap (Section 1c) is a smaller but real trust lever: a user who doesn't understand why they're granting `repo` scope to a third-party-feeling web app is primed to distrust the whole flow; naming exactly what's needed and why (write access to create commits) is cheap and directly addresses "can I trust this."

**Social job**: minor but present — for users sharing a repo with collaborators (the conflict-resolution and auto-merge flows exist precisely because concurrent editors are a real scenario), the job is "don't let my sync silently clobber someone else's work, and don't make me the one who has to explain a broken commit history." This is already the *design intent* behind `ADR-015`'s conflict-detection-before-ref-update and the reused `ConflictResolutionScreen` — no new UX surface needed here beyond ensuring conflict/auto-merge visibility (Section 3) extends to this collaborative case too, since a collaborator's confidence in the shared repo depends on the same auditability the solo user needs for self-trust.

## Summary of concrete recommendations for planning phase

1. Add a `SyncState` variant (or repurpose an existing one) representing "OPFS dirty-set non-empty, not currently syncing" — the current `Idle`-renders-nothing behavior is a design bug on web, not just a missing feature, given the desktop mental-model mismatch in Section 2.
2. Add `GitError.RateLimited(retryAfterSeconds: Int?)` with auto-backoff-retry UX, not manual retry.
3. Enrich `SyncState.Success` (or add a variant) to disclose auto-merged-file count, following the `JournalMergeReviewScreen` precedent of never fully hiding merge outcomes.
4. Write explicit, reassuring copy for the "5-step sequence failed partway" case that states nothing changed remotely (true per ADR-015's atomicity analysis) — do not let it read as a possible partial-corruption warning.
5. Add scope-specific PAT guidance + deep link to token creation (`github.com/settings/tokens/new?scopes=repo`, GitLab equivalent) in `GitSetupScreen` Step 3 — cheap, high trust payoff, directly serves the emotional JTBD.
6. Wire `window.beforeunload`-equivalent "unsynced changes" warning keyed directly off the same OPFS dirty-set `WasmGitWriteService` reads — never a separate/derived flag.
7. Verify (implementation-phase, not just planning) that clickable-but-not-`IconButton` badge regions in `SyncStatusBadge.kt` carry proper `Role.Button` semantics, and add a live-region announcement for state transitions.
8. Confirm the re-auth tap target from `CredentialExpired`/`AuthFailed` states opens the correct credential-entry surface (PAT field vs. OAuth) for whichever auth type is actually configured on web.
9. Cross-check `docs/tasks/git-sync-ux.md`'s still-open items (especially the "Cancel leaves git in MERGING state" data-corruption bug) before assuming the reused conflict UI is defect-free on web — this project inherits that UI as-is.
