## Concerns

- [ ] **Epic 2.2/3.3 — resume/cleanup contract for `pendingCommitSha` across a conflict is underspecified.** Story 3.3.2/Task 3.3.2c introduces `abortMerge` to clear "any in-progress `pendingCommitSha`/`pendingTreeSha` set-aside-for-merge state," but Story 3.3.2's own acceptance criteria also states that `resolveConflictBySide` (unmodified) always calls `commit()` again, and `commit()` (Task 3.1.1e) unconditionally rebuilds and overwrites `pendingCommit`/`pendingTreeSha` whenever the dirty set is non-empty. If `commit()` always runs before any `push()` in `GitSyncService.sync()`'s fixed step order, `abortMerge`'s cleanup role is redundant; if there is some path where `push()` could read a stale `pendingCommitSha` from a prior conflicting/pre-reload attempt without a fresh `commit()` first, that's an unaddressed illegal-state/resume bug. The plan should state explicitly (as a tested invariant, not an implicit assumption) that "`push()` is never called without an immediately preceding successful `commit()` in the same sync attempt," and either drop `abortMerge`'s SHA-clearing responsibility or explain the scenario it actually guards against. **Remediation**: add an explicit acceptance criterion/test to Story 3.1.2 or 3.3.2 that reloads the page (simulated) mid-conflict, calls `sync()` fresh, and asserts the final push uses the freshly-derived commit, never the stale one — and correspondingly document what `abortMerge` is for once that invariant is confirmed.

- [ ] **`WasmGitWriteService` is trending toward a God Object.** The Pattern Decisions table explicitly rejects splitting GitHub/GitLab dispatch into two classes (reasonable — only 2 hosts, no runtime polymorphism benefit), but that reasoning only addresses the *host* axis, not the *responsibility* axis. As planned, one class owns: per-endpoint HTTP + JSON request/response handling for 8+ GitHub/GitLab endpoints (Epics 3.1, 3.2, plus `log()`/raw-content fetch), conflict partitioning and tree-rebuild merge logic (Epic 3.3), retry/backoff/rate-limit policy (Epic 3.4), `GitWriteLock` coordination, dirty-set orchestration via calls into `PlatformFileSystem`, and observability logging (Epic 3.5) — five-plus genuinely distinct concerns bundled into one file/class regardless of the 2-host decision. **Remediation**: keep the single orchestrator (Service Layer pattern is the right call here) but extract collaborators it composes rather than implements inline: a `GitHubDataApiClient`/`GitLabCommitsApiClient` pair owning only wire-level HTTP+JSON per endpoint, and a stateless `ConflictPartitioner`/`GitTreeMerger` object owning `partitionConflicts`/tree-rebuild logic (this pairs with the testability finding below — extracting the pure logic also gets it out of `wasmJsMain` and into `commonMain`, directly testable). `WasmGitWriteService` then orchestrates: `GitHostAdapter` (config) → API client (transport) → `ConflictPartitioner`/`GitTreeMerger` (pure logic) → `GitWriteLock` (coordination) → `PlatformFileSystem` (dirty-set access) → logging. This is a restructuring of Epic 3.1–3.5's internal organization, not a change to any story's acceptance criteria.

- [ ] **`PlatformFileSystem` absorbs git-domain "dirty set" responsibility, coupling a Files-layer class to a Git-domain concern.** Per CLAUDE.md's layering (UI → ViewModel → Repository → Database/Files), `PlatformFileSystem` sits at the Files layer; Epic 2.1/2.2 add `dirtySet`, `getDirtySnapshot()`, `dirtyFileCountFlow`, OPFS marker debounce/persistence, `setPendingCommit()`, and `clearDirtySet()` directly as fields/methods on it. The Pattern Decision correctly rejects reusing `markDirty` (its Android-SAF write-behind semantics genuinely conflict), but the alternative chosen — folding git-sync bookkeeping directly into the file-system actual — creates the opposite problem: a low-level I/O abstraction now understands git commit-staging concepts (`pendingCommitSha`), and `WasmGitWriteService` (git domain) reaches back into `PlatformFileSystem` internals (`applyRemoteContent`, a merge-write path that deliberately bypasses the normal dirty-tracked `writeFile`) — a bidirectional coupling between the two files. **Remediation**: introduce a small `GitDirtyTracker` collaborator (plain class, instantiated and owned by `PlatformFileSystem.wasmJs.kt`, not `commonMain` since it's still OPFS/JSON-marker specific) that owns the dirty map, debounce, and marker serialization; `PlatformFileSystem.writeFile`/`deleteFile` call one narrow `tracker.record(path, op)` hook instead of embedding the map/debounce logic inline. This keeps the hook placement decision (correct — it must live at the `writeFile`/`deleteFile` call site) while giving the git-sync bookkeeping its own named, single-responsibility home that `WasmGitWriteService` depends on directly instead of poking through `PlatformFileSystem`.

- [ ] **Testability split is inconsistent with the plan's own stated principle.** The `GitHostAdapter` Pattern Decision explicitly justifies placing pure logic in `commonMain` specifically because "it's directly unit-testable... instead of the pure-Kotlin 'test double' pattern `WasmSectionSyncServiceTest.kt` needs for genuinely wasmJs-only code" (also stated verbatim in ADR-017's Consequences). Story 6.1.1, however, plans **reimplemented test doubles** in `commonTest` for `partitionConflicts` (Task 3.3.1a — pure `Set` intersect/subtract, zero JS dependency), the retry/backoff branching (Task 3.4.1a/b — the acceptance criteria describes it as a pure function of `suspend (attempt: Int) -> Int?`, i.e., already abstracted away from the real HTTP call), and the size-ceiling guard (Task 3.1.1b — a pure byte-length comparison). None of these three have a `js()`/`JsAny` dependency as described. Testing them via a hand-reimplemented double reintroduces the exact "double can silently drift from production logic" risk the `GitHostAdapter` decision was written to avoid. **Remediation**: move `partitionConflicts`, the size-ceiling guard, and the retry-decision function (not the Ktor `HttpRequestRetry` wiring itself, just the pure "given this status/attempt, what happens next" branching) into `commonMain` alongside `GitHostAdapter`, and have `WasmGitWriteService` call the real functions. Test them directly in `commonTest`, matching Story 6.1.2's already-correct treatment of `GitHostAdapter`.

- [ ] **Primitive obsession on SHA-shaped strings creates a real parameter-swap risk at a named call site.** `baseSha`, `pendingCommitSha`, `pendingTreeSha`, and the `treeSha`/`parentSha`/`commitSha` locals threaded through Epic 3.1 are all bare `String`. Concretely, Task 3.1.1d's `createCommitObject(hostConfig, treeSha, parentSha, message)` takes two SHA-shaped `String` parameters from two different object namespaces (a **tree** SHA just returned by `buildTree`, and a **commit** SHA — the local `baseSha`/parent commit) — the compiler cannot catch an accidental swap of the two, and `buildTree`/`createCommitObject`/`advanceRef`/`fetchCompareDelta` all pass SHAs around in different roles (base, tree, parent, head). **Remediation**: introduce cheap zero-cost newtypes — `value class CommitSha(val value: String)`, `value class TreeSha(val value: String)`, `value class BlobSha(val value: String)` — and use them in the signatures of `buildTree`, `createCommitObject`, `advanceRef`, `fetchRefSha`, `fetchCompareDelta`, and the `DirtySetMarker`/`PendingCommit` fields (ties directly into the Blocker above). This is a small, mechanical addition given the plan hasn't been implemented yet.

- [ ] **PAT is a bare `String` despite being the single highest-security-sensitivity value in this project.** `requirements.md`'s Non-functional Requirements calls the token handling "the highest-security-sensitivity piece of this session's investigation" and mandates it "must never be logged." The plan's only enforcement mechanism for that requirement is Task 3.5.1b, a **manual grep-and-classify audit** run once against the final diff — a process check, not a structural guarantee, and it will not catch a token-logging bug introduced in a later, unrelated change to this file. `GitHostConfig.token` and every function that threads it through (`authHeader`, `resolve`, the HTTP client calls) use plain `String`. **Remediation**: wrap the token in a small value class with a redacting `toString()`/`hashCode`-safe `equals`, e.g. `class GitToken(val value: String) { override fun toString() = "GitToken(***)" }`, used everywhere except the single header-construction call site (`GitHostAdapter.authHeader`) that needs the raw value. This converts "PAT never appears in a log line" from a one-time grep-verified promise into a property that holds for every future `logger.info("...$token...")` typo, since string-interpolating a `GitToken` prints `GitToken(***)` instead of the secret. The grep audit (Task 3.5.1b) can stay as defense-in-depth, but shouldn't be the only line of defense for the plan's own top-named security risk.

## Nitpicks

- The `DirtySetMarker` aggregate bundles two concerns with different lifecycles in one persisted object: `dirtyFiles` (continuously mutated as the user edits) and the pending-commit staging data (exists only within a single write-back attempt, addressed structurally by the sum-type Blocker above). Once that's fixed with `PendingCommit`, consider whether `DirtySetMarker` should more explicitly be read as "checkpoint = working set + optional in-flight staging" in its KDoc, so a future reader doesn't assume both halves share a lifecycle.
- Epic 4.1's `WasmGitRepository` implements several `GitRepository` methods (`isGitRepo`, `hasDetachedHead`, `removeStaleLockFile`, `stageSubdir`) as constant/no-op stubs because the interface is working-tree-shaped and the web backend is checkout-less. This is an accepted, already-reasoned trade-off in ADR-017 point 4 (not re-litigated here), but is worth flagging as latent ISP tension for whoever eventually designs `ADR-016`'s `SyncTransport` — a narrower interface (or a capability-detection method) might let a future transport avoid implementing meaningless no-ops. Out of scope for this plan.
- Host dispatch (`when (hostType)` over Strategy) and the "no diff/merge library" / kotlinx.serialization / Ktor `HttpRequestRetry` / stdlib Base64 choices all check out against `research/build-vs-buy.md` and the design-patterns guide — no changes needed there.
