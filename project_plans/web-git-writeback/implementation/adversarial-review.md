**Date**: 2026-07-14

## Blockers

## Resolved

- [x] **Web Lock scope is architecturally unachievable as specified.** RESOLVED, verified 2026-07-14: the Pattern Decision "Web Lock scope" (plan.md line 71), the new "Web Lock — guarantees and known gaps" note under Epic 2.3 (plan.md lines 394–403), and ADR-017's Consequences section (lines 104–118) now all consistently narrow the lock to wrap *only* `push()`'s write-critical-section (the ref-update PATCH or GitLab commits POST) plus the immediately-following `clearDirtySet` checkpoint write — confirmed achievable with the real callback-scoped `navigator.locks.request()` API and a real single-block `withLock` (Task 2.3.1c), since it's entered and released inside one non-suspending-across-other-calls function call, never spanning `commit()`→`push()`. The plan explicitly documents why the wider scope was rejected (leak risk: `commitLocalChanges()` and `resolveConflictBySide()`'s pre-resolution `commit()` — verified for real at `GitSyncService.kt` lines 291–315 and 375–404 — both call `commit()` with no following `push()` in the same invocation, which would leak an attempt-spanning lock without extra reentrancy machinery). It also honestly enumerates what remains uncovered — the `fetch()`/`merge()`-to-`push()` read-modify-write window across tabs, same-tab double-invocation/reentrancy — and states the residual risk is caught by server-side CAS (`409`/`422` on GitHub, `last_commit_id`/`start_sha` on GitLab), not silently ignored. Task 3.1.2e and Task 3.2.1c both correctly scope lock acquisition to only the "something staged" branch (skipping the lock entirely on a no-op `PendingCommit.None`), which is consistent with the narrowed design. This is a genuine, technically sound redesign, not a relabeling of the old claim — no new gap is introduced by the narrowing since the previously-claimed wider guarantee never actually existed against the real `GitSyncService.sync()` call structure (confirmed directly against `GitSyncService.kt`).

- [x] **GitLab conflict detection (`fetch()`/`merge()`) is never specified.** RESOLVED, verified 2026-07-14: new Story 3.2.2 wires `fetch()`/`merge()` (GitLab) via `GitLabCompareResponse`/`GitLabDiffEntry` (previously-orphaned Task 1.2.2c models) reusing the same host-agnostic `partitionConflicts` GitHub uses, with concrete Given/When/Then acceptance criteria for both the non-overlapping and overlapping cases, and explicit confirmation that GitLab's `merge()` sets the identical `SyncState.ConflictPending` GitHub's does. New Story 3.2.3 gives a concrete, specific (not hand-waved) detection mechanism for the push-time race: GitLab's commits API returns HTTP `400` (not `409`) with a `{"message": string}` body, with a stated conservative classification rule (any `400` not independently attributable to a local request-shape bug is treated as `MergeConflict`) and path-extraction-from-message logic (Task 3.2.3a). The story is honest that this exact status/body is "not yet verified end-to-end" and cites `research/stack.md` §2 as the basis for the assumption — but pairs that with a live-test task (3.2.3c) that forces the exact race (stale `last_commit_id`) and asserts the real response shape, i.e. verification of a stated hypothesis, not "TBD, confirm live" deferral of the mechanism itself. Routing to `ConflictResolutionScreen` is confirmed concretely: Task 3.2.3b specs the actual missing code change (`GitSyncService.sync()`'s push `.onLeft` at line ~222 gains a `MergeConflict` type-check branch setting `SyncState.ConflictPending`) — verified directly against `GitSyncService.kt`: line 222's `push().onLeft` currently only sets the generic `SyncState.Error(err)` with no `MergeConflict` special-case, exactly as the plan describes, confirming both the gap and the fix are real. The story also correctly notes this same latent bug affects GitHub's own push-time 409/422 conflicts (Task 3.1.2d's `MergeConflict` return was never previously routed to `ConflictPending` either) and fixes both hosts in one change — verified as a genuine, non-invented bonus fix, not scope-creep dressing.

- [x] **`SyncState.RateLimited` silently dropped.** RESOLVED, verified 2026-07-14: new Story 1.3.3 adds `SyncState.RateLimited(retryAfterSeconds: Int?)` to the sealed class (Task 1.3.3a) and wires `GitSyncService` to route `GitError.RateLimited` into it at every relevant `.onLeft` site — commit/fetch/merge/push in `sync()`, fetch in `fetchOnly()` (Task 1.3.3b) — mirroring the existing, real `AuthFailed`+`GITHUB_OAUTH` → `CredentialExpired` precedent confirmed at `GitSyncService.kt` lines 153–157. New Story 5.1.2 is the real consumption site fix: `SyncStatusBadge.kt` gains an `is SyncState.RateLimited` branch (Task 5.1.2a) with neutral (non-error) tinting and a "Retrying…" label, and — critically — Task 5.1.2b specs a literal string-content test asserting `!contains("tap to retry", ignoreCase = true)` against the composed content description/label. This makes the "never tap-to-retry" UX requirement a testable acceptance criterion, not prose, directly satisfying the review's bar. (Note: a later pass also added Story 3.4.2, which makes the "Retrying…" copy actually true by scheduling a real auto-retry after `retryAfterSeconds` — pre-mortem P1 #4. That's a separate fix layered on top of this one, not a correction to it.)

- [x] **`writeFileBytes` is never hooked for dirty-tracking.** RESOLVED, verified 2026-07-14: Story 2.1.1's Grounding note and acceptance criteria now explicitly cover `writeFileBytes` — Task 2.1.1e implements real byte-level OPFS I/O for it (there was no override at all before, so paranoid-mode writes previously crashed outright) *and* wires the same `recordDirty` hook `writeFile` uses, into the same dirty map. The acceptance criteria include an explicit case: "Writing raw bytes (paranoid mode) records a `WRITE` dirty entry the same way... the same dirty set `writeFile` writes into, not a separate one." - [x] **PAT expired/revoked mid-session has no dedicated handling.** RESOLVED, verified 2026-07-14: new Story 3.4.3 adds an explicit `401`/`403`-without-`Retry-After` → `DomainError.GitError.CredentialExpired` mapping (Task 3.4.3a's shared `mapHttpFailure` classifier, applied across every GitHub and GitLab HTTP call site), and wires `GitSyncService`'s five `.onLeft` branches to route it to `SyncState.CredentialExpired(graphId)` (Task 3.4.3b) — deliberately *not* reusing `AuthFailed`, since that variant's existing `GitSyncService` routing only fires for `GitAuthType.GITHUB_OAUTH`, which web's session-scoped PAT (`HTTPS_TOKEN`) would never match (Story 3.4.3's own Design section spells this out). Task 3.4.3c adds the missing/empty-token test pre-mortem finding #3 prescribed. Note: this story's copy was itself corrected in this pass (see the fix logged against Task 3.4.3a's acceptance-criteria text) to be host-neutral rather than hardcoding "GitHub," since the same classifier now also handles GitLab's 401s.

- [ ] **Ambiguous push outcome (network drop after the server applies the ref update, before the client sees the response) produces a false conflict on retry, not a clean resume.** STILL OPEN, re-verified 2026-07-14 — no task in the current plan.md adds the "already-synced, not conflicting" check this concern recommends; a targeted search of plan.md for "already-synced"/"byte-identical" finds nothing. If `push()`'s `PATCH .../git/refs/heads/{branch}` succeeds server-side but the response is lost client-side, `clearDirtySet` is never called (correct per the "clear-last" crash-safety design), so the local `baseSha` stays stale and the dirty set stays populated. On the next sync attempt, `commit()` re-derives a fresh commit from the same dirty set, and `fetch()` observes the remote ref has moved (to the *already-applied* version of this exact same edit) — the compare-delta's changed paths will be identical to the local dirty paths, so `partitionConflicts` computes 100% overlap and surfaces a spurious `MergeConflict`/`ConflictPending` for content the user already successfully pushed. This is a real, silent-until-hit edge case — not data loss, but a confusing, avoidable manual-resolution prompt for a no-op. **This pass explicitly accepts it as known risk rather than fixing it** — see `plan.md`'s new "Deferred Technical Debt" section (mirrors `pre-mortem.md` Failure Mode #2, P2 severity) for the accepted-risk framing and pointer. — **Recommendation** (unchanged, still the right fix if this is picked up): after a successful push, if local `baseSha` is stale but the remote-changed-paths are identical (not just overlapping) to the local dirty paths with matching content, treat it as already-synced and clear the dirty set instead of prompting for conflict resolution.

- [ ] **GitLab write-path CORS credentials handling is unverified**, despite `research/pitfalls.md` §2 flagging it by name as the "bigger open risk" and explicitly recommending it be "verified against a live `gitlab.com/api/v4/.../repository/commits` POST during research/implementation," including a specific gotcha: "confirm the exact `fetch()` call must NOT include `credentials: 'include'` or GitLab will reject the preflight outright." STILL OPEN, re-verified 2026-07-14 — this pass added Task 3.2.1e (confirms the Ktor client isn't sending `credentials: 'include'`, config point) and a new Story 6.3.1 acceptance criterion (the GitLab live push must complete without a CORS-preflight rejection, with a documented fallback — explicitly set the JS engine's `fetch` `credentials` option to `"omit"` — if it doesn't) plus a matching note on Task 6.3.1d. That's real progress: the risk now has a concrete task and a concrete live-test assertion, not just a research-doc mention. But it remains genuinely **unverified** until that live test is actually run against real `gitlab.com` — nothing in this pass changes that, since Story 6.3.1 is gated/manual, not CI-blocking. Keeping this item open until that live run happens (or until Story 6.3.1 is run once and its result is recorded here).

- [x] **Network-level failures (connection drop, DNS failure, offline) aren't covered by the retry configuration.** RESOLVED, verified 2026-07-14: new `DomainError.GitError.NetworkFailure` variant (Story 1.3.1) is distinct from both a received-but-erroring HTTP response and from `RateLimited` (which requires an actual `429`/`403` response with a status code to inspect). Task 3.4.1d wraps each call site's request-dispatch step in `try/catch (e: Throwable)` (rethrowing `CancellationException` unchanged), mapping any exception where no HTTP response was ever received — connection refused, DNS failure, offline mid-request, CORS block — to `NetworkFailure`, explicitly *before* any response/status inspection so it can never be conflated with a received `5xx`. This directly closes the gap `research/pitfalls.md` §6 flagged (rejected-Promise failures were previously indistinguishable from HTTP-status failures).

## Minors

- GitLab's single-call commits API bundles multiple files' base64 content into one JSON POST body; `MAX_BLOB_BYTES` (Task 3.1.1b) is scoped to Epic 3.1's per-blob GitHub check only — there's no aggregate body-size guard for the GitLab path, which could hit GitLab's own request-body limits even when every individual file is under the GitHub-derived per-file threshold.
- Task 4.3.1a's sketch reuses `PlatformFileSystem.githubToken` (a GitHub-specifically-named existing field) as the generic token source passed into `GitHostAdapter.resolve()` regardless of resolved host type. Functionally fine if there's only one active token per graph, but the name invites future confusion now that the same field backs both GitHub and GitLab auth — worth a rename or an explicit comment.
- `WasmGitWriteService` is an intentional single ~400–600 line orchestrator (explicit, well-justified Pattern Decision given exactly 2 fixed hosts) — reasonable now, but worth revisiting if retry/observability/auto-merge logic accretes further before this ships.
- The plan correctly incorporates both of requirements.md's Phase-2 corrections (MergeConflict/ConflictPending terminology, GitRepository-actual-as-primary-reachability-path) and correctly resists two scope-creep temptations (the `autoMergedCount` UX enrichment, and hiding "Clone and Add" on web) by explicitly deferring them with reasoning in Unresolved Questions rather than silently expanding scope — no regression to either corrected framing found.
