# Pre-mortem: web-git-writeback
**Date**: 2026-07-14

## Failure Modes

| # | Failure | First Symptom | Prevention | Severity |
|---|---------|--------------|------------|----------|
| 1 | The dirty-set marker write is debounced ~2s (Task 2.1.2b); a crash or tab-close inside that window discards the record that a file needs syncing, even though the file's content is already safely in OPFS. The UI then reports "0 unsynced" for content that was never pushed — the exact original bug, one layer deeper. | User edits a page, closes the tab within a few seconds, reopens later: the sidebar shows no pending-sync indicator and the edit never appears on the remote, with no error anywhere. | Flush the marker synchronously (or near-zero-debounce) on `visibilitychange`/`beforeunload`/`pagehide`, not just on a timer; add a test that calls `writeFile()` then simulates reload *before* the debounce timer fires and asserts the dirty entry survives. | P1 |
| 2 | The Web Lock only wraps `push()`'s write-critical-section (by design, per ADR-017/adversarial review), and GitHub's ref-update response can be lost client-side after the server already applied it. Both gaps mean a losing tab, or a tab that never saw its own push's success response, computes a conflict partition against stale data. | A user (or QA) sees a `ConflictPending` prompt for a file they *just successfully pushed*, with LOCAL and REMOTE showing identical content — or, in the two-tab case, a losing tab's edit is bounced into conflict resolution UI when nothing actually diverged. | Add the "already-synced, not conflicting" check the adversarial review recommended: if local `baseSha` is stale but the remote-changed content is byte-identical to the local dirty content, auto-clear instead of prompting. Add a two-tab integration test exercising the `fetch()`/`merge()`-to-`push()` race window explicitly (not just same-tab lock contention). | P2 |
| 3 | Session-scoped PAT (never persisted) plus no `401`/expired-token → `AuthFailed` mapping anywhere in `WasmGitWriteService` (confirmed gap in adversarial review) means an expired or absent token surfaces as a generic `PushFailed`, not "please re-enter your PAT." | User taps "sync" repeatedly across sessions, sees a generic red error each time with no actionable message; `LocalChangesPending`'s count keeps climbing while the user has no idea *why* sync keeps failing. | Add explicit `401`/`403`-without-`Retry-After` → `AuthFailed` mapping at every `WasmGitWriteService` HTTP call site (blob/tree/commit/ref/GitLab-commit), routed through the existing `AuthFailed`→`CredentialExpired` path in `GitSyncService`. Add a test asserting a missing/empty token (the `PlatformFileSystem.githubToken ?: ""` fallback in Task 4.3.1a's sketch) produces `CredentialExpired`, not a generic error. | P2 |
| 4 | `SyncState.RateLimited`'s badge text says "Retrying…" (mandated by UX requirement to never say "tap to retry"), but no task in Epic 3.4/5.1 actually schedules an automatic re-invocation of `sync()` after `retryAfterSeconds` elapses — retry only happens *inside* one push attempt's fixed Ktor retry budget (4 tries), then the attempt terminates. | The sync badge is stuck showing "Retrying…" indefinitely after a rate-limit hit; nothing pushes until the user notices and manually taps again — the UI actively promises self-healing behavior that doesn't exist. | Either implement a real scheduled auto-retry (a coroutine that re-calls `sync()` after `retryAfterSeconds`, while the tab/composable is alive) or change the badge copy to not claim automatic retry if none is built. Add an acceptance test proving an automatic re-sync actually fires after the stated delay, not just that the state renders with the right text. | P1 |
| 5 | GitLab's conflict classification (Story 3.2.3) treats *any* `400` not attributable to an obvious local validation bug as `MergeConflict` — a conservative rule chosen specifically because the real response shape is "not yet verified end-to-end" against a live API. A genuine request-shape bug (encoding edge case, unusual filename, or the aggregate-body-size gap the adversarial review's Minors flagged as unguarded) would be misclassified the same way. | A GitLab user's sync perpetually shows a conflict on the same file no matter which side (LOCAL/REMOTE) they pick — the "conflict" recurs every single attempt because the real cause is a request bug, not a real conflict, and choosing either side just resends the same malformed request. | Treat Story 6.3.1's gated live GitLab test as a hard prerequisite before shipping the GitLab path (not just nice-to-have), to confirm the `400`-body shape assumption for real. Add a client-side escape hatch: if the same file re-surfaces as `MergeConflict` after two consecutive resolve-and-retry cycles with no intervening remote change, downgrade to a distinguishable "sync failed, see logs" state instead of re-showing the same conflict UI forever. Add the aggregate GitLab request-body size guard the Minors note flagged as missing. | P2 |

## P1 Items (address before implementation)
- [x] #1 — Add an unload-triggered synchronous (or near-zero-debounce) flush of `.stele-dirty-set.json` in Epic 2.1/2.2, plus a crash-before-flush regression test, so a tab close within the debounce window cannot silently drop a dirty-set entry for content that's already safely in OPFS. **Fixed in `plan.md` Epic 2.1, Story 2.1.2 / Task 2.1.2b** — replaced the fixed ~2s trailing debounce with an immediate-start, coalesce-while-busy write scheduler (bounds the crash-loss window to a single in-flight OPFS write's duration instead of indefinitely during active editing), plus a `visibilitychange`-triggered best-effort flush as a belt-and-suspenders guard, plus a new regression test (Task 2.1.2b-test).
- [x] #4 — Either build an actual scheduled auto-retry for `SyncState.RateLimited` in Epic 3.4/5.1, or change the badge's "Retrying…" copy so it doesn't promise behavior the plan doesn't implement; add a test proving whichever choice is made actually holds. **Fixed in `plan.md` Epic 3.4, new Story 3.4.2** — `GitSyncService` gains a `rateLimitRetryJob` field scheduled on its existing owned `scope` (the same scope `shutdown()`/`startPeriodicSync` already use), re-invoking `sync()`/`fetchOnly()` after the `Retry-After` delay, cancelled by a manual sync trigger or by `shutdown()`; verified by Task 3.4.2d's virtual-time test.
