# Pre-Mortem: paranoid-mode

**Date**: 2026-06-24

---

## Failure Mode 1 — User loses access to their encrypted graph

**Incident**: Six months after shipping, a reporter filed a P1 support ticket: "All my notes are gone." The user's laptop had crashed mid-write during a full DEK rotation (triggered from VaultSettingsScreen). The rotation had re-encrypted 1,200 files and was updating the `.stele-vault` header when power cut out. On restart, the header had been partially written — its MAC no longer verified, and the backup header in the reserved area (offset 2061) had never been populated because T2.2.3 was implemented as a stub that wrote zeros. The old DEK was unreachable. All keyslots decrypted to garbage. The user had no passphrase fallback because the OS keychain slot was the only registered provider, and the laptop's keychain was cleared by the repair shop before return. The user lost 3 years of notes. Total data recovery: zero. The incident report identified two independent failures: (1) the backup header path was untested end-to-end (there was no test exercising "corrupt primary, recover from backup"), and (2) the UI did not enforce the T7.5.3 rule requiring at least one non-keychain slot, so the OS keychain was the only registered provider.

**Priority**: P1

**Mitigation**: Three mandatory plan changes. First, add a story to Epic 2 (Story 2.2.X) that explicitly tests primary-header-corrupted recovery: corrupt bytes 0–12 of `.stele-vault`, open the vault, assert that the backup header at offset 2061 is read and unlock succeeds — this test must pass before Epic 2 is considered complete. Second, add an Epic 2 task requiring that the full DEK rotation (T2.4.2) uses a two-phase commit: all re-encrypted files are written to a `.stele-vault.rotation-staging/` directory, the backup header is written and fsync'd, then an atomic rename replaces the primary; the old header is only cleared after the new header verifies. Third, enforce T7.5.3 in the UI at the point of OS-keychain registration: if the keychain slot is being added and no passphrase/keyfile slot exists, block the operation with a required "Add a backup passphrase first" dialog — not a dismissible warning.

## Failure Mode 2 — Hidden volume discovered

**Incident**: A journalist using SteleKit paranoid mode was detained and their laptop seized. The forensic examiner ran `ls -la` on the graph directory and found a directory named `_hidden_reserve/`. They did not need to decrypt anything: the directory name alone was cited in the warrant affidavit as evidence of intentional concealment. Separately, a second examiner noted that every `.stek` file had a modification timestamp of `1970-01-01 00:00:00 UTC` — a recognizable artifact of T7.4.1's fixed-timestamp mitigation — and flagged this as an anomaly consistent with deliberate evidence tampering. The journalist's legal team argued plausible deniability, but the `_hidden_reserve/` directory name, the epoch timestamps, and the presence of 50 MB of unreadable binary data collectively defeated the argument. The feature shipped with these two detectable fingerprints intact because the adversarial review concern about `_hidden_reserve/` (marked as a concern, not a blocker) was not acted on before the Epic 6 deadline.

**Priority**: P1

**Mitigation**: Before Epic 6 is implemented, resolve the two fingerprint concerns from the adversarial review as hard blockers, not concerns. First, rename `_hidden_reserve/` to a non-descriptive identifier: use the first 16 hex characters of `HKDF(outerDEK, salt="reserve-dirname", info="v1")` so the directory name looks like a normal asset cache and changes per-vault. Update all references in the plan (T6.1.1, T3.2.4, T3.3.4, RT-10, HV-03, HV-04, GL-06) to derive this name at runtime from the outer DEK. Second, change T7.4.1 from "set fixed modification timestamp (epoch zero)" to "set modification timestamp to a random value within the last 30 days" — epoch zero is a trivially detectable artifact. The validation plan test RT-10 and the adversarial review concern about this mitigation must both be updated to reflect these changes.

## Failure Mode 3 — Performance regression ships

**Incident**: Three weeks after release, one-star reviews appeared on the desktop app: "Paranoid mode takes forever to open." Investigation revealed that the Argon2id benchmark run at vault creation time (T8.1.2) had been implemented using post-warmup JVM timing, omitting the 200–300 ms BouncyCastle class-loading overhead that only occurs on the first call per JVM process. Users on low-end hardware (4 GB RAM, spinning disk, JVM startup from cold) experienced 8–12 second unlock times against vaults calibrated to 4-second targets. The "Standard" preset had been calibrated on a developer MacBook Pro; it was never tested on the low-end hardware called out in NFR-6. The WASM path was worse: the libsodium.js WASM module initialization added 600–900 ms before any Argon2id work began, making the first unlock in a browser session take 15+ seconds. The WASM gap was known (validation plan §13 notes "WASM tests are manual + pre-release only") but the pre-release WASM performance test was skipped due to time pressure.

**Priority**: P2

**Mitigation**: Two additions to the plan. First, revise T8.1.2 to specify that the Argon2id benchmark at vault creation must be conducted from a cold JVM (no prior BouncyCastle calls in the same process), must run the benchmark three times and take the median, and must be conducted on the CI runner's hardware specification (not a developer machine) to set the "Standard" preset threshold. The plan should add a fast-path: if the cold-start benchmark exceeds 2 seconds, automatically downgrade to the "Fast" preset and surface a one-time informational message. Second, add a mandatory WASM performance gate to the pre-release checklist: PERF-01 must pass in the `wasmJsBrowserTest` environment including libsodium.js initialization time, not excluding it. The WASM §13 gap note must be escalated from "action required before shipping" to a hard CI gate on the release workflow.

## Failure Mode 4 — Key material exposed in a crash dump

**Incident**: A user reported that their paranoid-mode DEK appeared verbatim in a JVM heap dump produced when the app crashed with an `OutOfMemoryError` during a large graph import. The OOM was thrown inside the Argon2id scratch buffer allocation (64 MiB on a 2 GB heap under memory pressure), which left `VaultManager` in an indeterminate state: the DEK `ByteArray` had been written but `lock()` was never called. The OOM propagated uncaught through the coroutine scope, reaching Android's default uncaught-exception handler, which killed the process and wrote a heap dump to external storage. The heap dump contained the DEK as a contiguous 32-byte region identifiable by correlation with keyslot data. The incident was compounded by the passphrase `CharArray` being found in the dump as well: the `CharArray → ByteArray` conversion path (which the adversarial review flagged as unspecified) had gone through a `String` intermediate in a Kotlin stdlib call, and that `String` was interned and not zeroed before the OOM. The plan had acknowledged the JVM GC limitation (T7.1.4, RISK-2) and documented it as "accepted risk," but had no story for the OOM-during-unlock failure path.

**Priority**: P2

**Mitigation**: Add two tasks to Epic 7 (Story 7.1). First, add a task specifying the mandatory `CharArray → ByteArray` conversion path: `charset.newEncoder().encode(CharBuffer.wrap(charArray))` into a `ByteArray`, zero-fill the `ByteArray` after use — never via `String`, `concatToString()`, or any Kotlin stdlib method that produces a `String` intermediate. Add a detekt rule banning `CharArray.concatToString()` and `String(charArray)` in the `vault/` package. Second, add a task wrapping the Argon2id call in `runCatching { }.onFailure { e -> if (e is OutOfMemoryError || e is Error) { zeroDek(); emit(VaultError.ResourceExhausted) } }` — as the CLAUDE.md architecture notes, uncaught `Throwable` kills the Android process, so the guard must be at the `VaultManager.unlock()` call site, not only at the coroutine exception handler level. Add a corresponding `VaultState.ResourceExhausted` UI state and a test that simulates OOM during Argon2id by injecting a mock `CryptoEngine` that throws `OutOfMemoryError` and asserts the DEK reference is zeroed and the UI state is `ResourceExhausted`.

## Failure Mode 5 — WASM vault breaks after browser update

**Incident**: Eight months after shipping, Chrome 140 shipped a change to its WASM memory model that altered how libsodium.js allocates its internal heap during `sodium.ready` initialization. Vaults created before Chrome 140 could no longer be unlocked in the browser: `crypto_pwhash` returned a different output for the same inputs due to a libsodium.js version mismatch between the npm-pinned version used at vault creation and the version loaded by the updated app bundle. The root cause was that libsodium.js had been pinned in `package.json` without a lock file committed to the repo, so a `npm install` after the Chrome update pulled a minor libsodium.js version that had changed the Argon2id output encoding. Users had no recovery path because the vault's keyslot parameters did not record which libsodium.js version had been used for Argon2id, and the JVM path (which used BouncyCastle) produced correct output — confirming the bug was WASM-specific. The cross-platform interoperability test (WASM §13: "generate on JVM, decrypt on WASM") would have caught this if it had been run in CI, but the WASM CI gap (noted in validation plan §13) meant this test had only been run manually once at initial release.

**Priority**: P2

**Mitigation**: Two plan additions. First, commit a `package-lock.json` (or `yarn.lock`) for the WASM npm dependencies and pin the exact libsodium.js version in CI; add a Dependabot or Renovate rule that requires a manual review and cross-platform interoperability test before any libsodium.js version bump is merged. Second, add a cross-platform round-trip test to the release CI workflow: a jvmTest produces an encrypted file at known Argon2id parameters, the output bytes and encrypted ciphertext are written to a fixture file, and the WASM `wasmJsBrowserTest` suite loads that fixture and asserts decryption produces the expected plaintext — failing if libsodium.js behavior diverges. This test must be a hard CI gate on the release branch, not optional.

## Failure Mode 6 — Hidden volume capacity exhaustion

**Incident**: A power user who had been using the hidden volume for 18 months hit a capacity wall. They had set `reserveSizeMb=50` at vault creation (the default) and had been storing embedded images and large reference documents in their hidden notes. When the hidden volume's single `.stek` archive filled the 50 MB reserve, `HiddenVolumeManager` returned `VaultError.HiddenAreaWriteDenied` — the same error used when the outer graph tries to write into `_hidden_reserve/`. The UI displayed a generic "write denied" error with no explanation. The user, unaware the hidden volume had a capacity limit, assumed the app was broken and filed a bug report. The page save appeared to have succeeded in the UI (the debounced write completed without surfacing the error to the block editor's local state) — the error was emitted asynchronously through `GraphWriter` and was lost in a `.onLeft { logger.error(...) }` call that had no UI propagation path. The user's last several hours of notes to the hidden volume were silently lost.

**Priority**: P1

**Mitigation**: Two mandatory changes. First, `HiddenVolumeManager` must expose a `Flow<HiddenVolumeCapacity>` that emits `(used: Long, total: Long)` on every hidden-volume write, and `GraphWriter` must surface a `VaultState.HiddenVolumeNearCapacity` warning at 80% and a `VaultState.HiddenVolumeFull` error at 100%. The "write denied" error path in `GraphWriter` must propagate to the ViewModel as a dismissible alert — it cannot be swallowed in a logger call. Second, the plan already flags RISK-7 ("silent data loss if hidden volume fills") but the resolution ("capacity warning before writes reach 90%") is described as "required" without being assigned to any Epic or Story. Assign this explicitly: add Story 6.3 to Epic 6, "Hidden volume capacity management," with tasks for the capacity flow, UI warning, and a `HiddenVolumeCapacityTest` asserting that a write attempt at 100% capacity (a) fails with a user-visible error within 500 ms and (b) does not corrupt previously-written hidden-volume data.

## Failure Mode 7 — Implementation slips past deadline due to JVM ChaCha20-Poly1305 behavior differences

**Incident**: Epic 1 (Crypto Foundation) was estimated at two weeks and took six. The primary blocker was RISK-3 (adversarial review): the JDK `ChaCha20-Poly1305` implementation appends the 16-byte Poly1305 tag inside the `doFinal` output buffer rather than returning it separately, and the output size returned by `getOutputSize()` is the ciphertext-plus-tag length. This was not validated against the STEK binary format spec (plan §2a) before implementation began — the format spec assumed the tag was a separate field at a fixed offset, but the JDK's `doFinal` returns `ciphertext || tag` as a single blob. Correcting the format spec mid-Epic required updating T1.2.1, T1.2.2, the STEK offset table, CL-01 through CL-09, and the WASM interop declarations, because libsodium.js separates the tag from the ciphertext in its API. The WASM interop prototype (RISK-1) had been deferred past Phase 1 start, so the API impedance mismatch between JVM and WASM was not discovered until Phase 3, requiring a partial rewrite of `WasmCryptoEngine` and a format bump. The overall schedule slipped by 10 weeks and Epic 6 (Hidden Volume) was cut from v1.

**Priority**: P2

**Mitigation**: Before any Phase 1 code is written, complete two concrete validation tasks. First, write a 30-line standalone JVM program (not inside the KMP module — just a plain `main()`) that calls `Cipher("ChaCha20-Poly1305").doFinal()` and prints the output length, the first 12 bytes, and the last 16 bytes; confirm the STEK offset table against actual JDK behavior and update the format spec if needed. Second, complete the RISK-1 WASM prototype as a required Phase 0 gate — specifically, implement `encryptAEAD` in `WasmCryptoEngine`, call it from a test, and verify that a file encrypted on JVM can be decrypted by the WASM engine using the same key and nonce. This cross-platform smoke test must pass before Phase 1 is considered started. Add these two items as explicit pre-Phase-1 gate tasks in the Implementation Sequence table (plan §5), not as risks to be resolved "sometime before Phase 3."
