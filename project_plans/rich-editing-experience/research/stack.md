(Constraint in requirements.md says "no new dependencies" anyway — moot for planning, but worth noting for implementation-phase file targeting.)

**Does BasicTextField2 / TextFieldState (new Compose Foundation text API) apply?** No — not on this stack, and not worth chasing. Compose 1.10.3 already promotes what used to be called `BasicTextField2` internals to the *stable* `BasicTextField(state: TextFieldState, ...)` overload, but this repo uses the **classic** `BasicTextField(value: TextFieldValue, onValueChange: (TextFieldValue) -> Unit, ...)` overload exclusively (`BlockEditor.kt:99`, `BlockItem.kt` x4, confirmed zero references to `TextFieldState`/`rememberTextFieldState` anywhere in `commonMain`). Migrating to `TextFieldState` would be a real rewrite (different state-hoisting model, different selection/undo API) with no clear benefit for this project's scope (toolbar affordances, shortcuts, journey efficiency) — recommend **not** touching this in-scope; flag as a separate future modernization ADR if ever pursued.

## 2. Keyboard handling patterns already in this repo (reuse, don't reinvent)

- **`Modifier.onPreviewKeyEvent { }`** is the established pattern for intercepting hardware-keyboard events before they reach the text field's IME handling. Used in `BlockEditor.kt:166` (wraps the `BasicTextField`) and `Modifier.onKeyEvent { }` in `CommandPalette.kt:101` (attached to the palette's outer `Column`, not the search `TextField` itself — deliberate, since the palette needs Up/Down/Enter/Escape to work while the `TextField` still receives normal typing).
- **Modifier check both `isCtrlPressed` and `isMetaPressed`** together (`BlockEditor.kt:304`, `App.kt:1763`) — this is the repo's existing convention for cross-platform Ctrl(Win/Linux)/Cmd(macOS) parity. **Continue this convention for any new shortcut** — do not hardcode one or the other.
- **`event.type == KeyEventType.KeyDown`** gating is used everywhere to avoid double-firing on KeyUp — follow this for new handlers.
- Central shortcut table in `BlockEditor.kt`'s private `handleKeyEvent()` (lines ~219-465): ordered cascade of independent `if` blocks (arrow/enter navigation → Ctrl+Enter → Tab → Escape → Ctrl/Cmd+format-letter → `[` bracket-wrap → Ctrl+Left/Right word nav → Home/End → plain Enter/Backspace/Tab/Up/Down). New shortcuts should be added as another guarded `if` block in this cascade, consistent with existing style, rather than a new dispatch mechanism.
- **Command palette (`CommandPalette.kt`)** already exists and is wired to a live command source — do not build a second one. Its `Command` data class (`AppState.kt:196`: `id`, `label`, `shortcut: String?`, `action: () -> Unit`) is populated in `StelekitViewModel.updateCommands()` (`StelekitViewModel.kt:2017`) by **bridging** the richer `EditorCommand` type from `editor/commands/CommandManager` into the simpler UI `Command` type. **Adding a new palette-visible action = registering a new `EditorCommand` in `EssentialCommands.kt` (or a new file in `editor/commands/`) and letting the existing bridge surface it** — no CommandPalette.kt changes needed unless the UI itself (row layout, shortcut-badge rendering) needs to change.
- The palette shows `command.shortcut` as a display-only badge (`CommandItem`, `CommandPalette.kt:208-223`) — it is **not** currently cross-wired so that the badge text is guaranteed to match a live keyboard binding (i.e., no single source of truth enforcing "the shortcut shown in the palette == the shortcut that actually fires"). Worth flagging as an audit finding: any toolbar/shortcut overhaul should establish one canonical shortcut table that both `handleKeyEvent()` and the `EditorCommand.shortcut` strings read from, to prevent drift.

## 3. Orphaned "rich" command/editor infrastructure — do not build on top of it without a decision

There are **two parallel command systems** in this codebase:

1. **Live system**: `ui/screens/*.kt` (`FormatAction` enum + `applyFormatAction()` in `BlockEditor.kt:491`), `MobileBlockToolbar.kt`, `CommandPalette.kt`, `AppState.Command`. This is what actually runs today across all platforms.
2. **Unused/legacy system**: `kmp/src/commonMain/kotlin/dev/stapler/stelekit/editor/` — a much larger, more "enterprise" command framework (`CommandManager`, `CommandRegistry`, `CommandSystem`, `CommandTypes`, `EssentialCommands.kt` (776 lines), `SlashCommandHandler.kt`, `RichTextEditor.kt`, `editor/components/RichTextEditor.kt`, `editor/performance/PerformanceOptimizedEditor.kt`, plus README.md and UNDO_REDO_README.md documenting an `EnhancedCommandPalette` and `/slash-command` UX that **does not exist in the live UI** — no `/` trigger exists anywhere in `BlockEditor.kt`/`BlockItem.kt`).

Only a thin slice of #2 is actually live: `StelekitViewModel` instantiates `CommandManager.create(...)` (`StelekitViewModel.kt:420`) purely to source the list of `EditorCommand`s that get bridged into the palette's `Command` list (`updateCommands()`). The `SlashCommandHandler`, `RichTextEditor` composables, and `EnhancedCommandPalette` referenced in the README are **not called from any `ui/` file, `jvmMain`, `androidMain`, `iosMain`, or `jsMain` source set** — confirmed via repo-wide grep. They do have `businessTest` coverage (`ImageAttachCallbackContractTest`, `LinkInsertionTest`), so they are exercised in isolation but not wired to the running app.

**Implication for planning**: the requirements call for toolbar/shortcut overhaul across "all editing surfaces." Before extending the command system, Phase 3 planning must explicitly decide: (a) extend the live, thin `EssentialCommands.kt`/`CommandManager` bridge (low risk, matches what's actually running), or (b) resurrect `RichTextEditor.kt`/`SlashCommandHandler`/`EnhancedCommandPalette` and wire them into `BlockEditor.kt` for the first time (higher risk — untested integration path, and the README's documented API surface — e.g. `CommandConfig(async, debounceMs)`, `CommandResult.Partial` — has never been exercised against the real `BasicTextField`/`TextFieldValue` in production). Recommend (a) unless slash-commands specifically are a requirement — nothing in requirements.md asks for a `/` command syntax, so resurrecting (b) is likely scope creep.

## 4. Existing markdown decoration/rendering — reuse for new toolbar affordances

`ui/components/MarkdownEngine.kt` (584 lines) is the one markdown-to-`AnnotatedString` engine in the repo, and it is **already reused broadly**:
- `parseMarkdownWithStyling(...)` — the render-mode path (bold/italic/strikethrough/highlight/wiki-links/tags/task-markers/inline-code styled via `SpanStyle`) — is called from `BlockViewer.kt`, `PageView.kt`, `HeadingBlock.kt`, `SearchDialog.kt`, `ReferencesPanel.kt`, `ImportScreen.kt`, `SnippetText.kt`, `GlobalUnlinkedReferencesScreen.kt`, and LLM preview screens (`NewPagePreview.kt`, `BlockDiffView.kt`). **This is the correct, established reuse point for any new "preview" affordance** (e.g. showing what a table or code block will look like) — do not write a second markdown-to-styled-text renderer.
- `applyMarkdownStylingForEditor(...)` (`MarkdownEngine.kt:535`) — a **second, edit-mode-oriented** styling function (bold/italic/code/strikethrough/link `SpanStyle`s applied by safe character-offset ranges) — **exists but is called from nowhere else in the repo** (confirmed via grep — zero call sites outside its own file). This means SteleKit has already built, but never wired up, the groundwork for "WYSIWYG-lite" in-place decoration while editing (i.e., rendering `**bold**` as actually bold text with the markers dimmed, instead of the current always-raw-markdown-text `BasicTextField`). If the journey audit finds that raw-markdown-visible-while-editing is a discoverability/efficiency problem (plausible — it's exactly the kind of thing a "step-count and heuristic audit" turns up), **this function is the reuse target, not something to build from scratch.** It would need to be threaded through as a `VisualTransformation` (Compose Foundation text API) wrapping `BlockEditor`'s `BasicTextField` — no `VisualTransformation` usage exists yet in `BlockEditor.kt`/`BlockItem.kt` (the only current uses are `PasswordVisualTransformation` in unrelated settings/vault/git screens), so this would be new integration work, but the styling logic itself is already written.
- No table-insert or code-block-language-picker rendering exists yet in `MarkdownEngine.kt` — `renderNode()`'s handled node types (from the grep) cover bold/italic/strikethrough/highlight/link/wiki-link/tag/task-marker/inline-code, but not fenced code blocks with language or tables. Any table/code-block-language toolbar affordance is new rendering work, though it should extend `MarkdownEngine.kt` rather than create a parallel file, to keep the single-renderer invariant the rest of the app already depends on.

## 5. Toolbar/shortcut extension pattern (confirmed reusable)

`FormatAction` enum (`ui/screens/JournalsViewModel.kt:26-37`, despite the filename) defines `(prefix, suffix)` pairs: `BOLD("**","**")`, `ITALIC("*","*")`, `STRIKETHROUGH("~~","~~")`, `HIGHLIGHT("^^","^^")`, `CODE("`","`")`, `LINK("[[","]]")`, plus line-prefix actions `QUOTE("> ","")`, `NUMBERED_LIST("1. ","")`, `HEADING("# ","")`. `applyFormatAction()` (`BlockEditor.kt:491`) is the single function both the hardware-keyboard shortcut cascade (`handleKeyEvent()`) and (per `MobileBlockToolbar.kt`'s formatting-overflow row) the mobile toolbar buttons call. **Any new formatting affordance (e.g. a table-insert or code-block-with-language action) should be added as a new `FormatAction` entry + `applyFormatAction` branch**, which automatically makes it available to both hardware shortcuts and the toolbar with one code path — this is the existing, idiomatic extension point and should be preserved rather than introducing a separate mechanism for "new" toolbar actions.

## 6. iOS hardware-keyboard gotcha

No iOS-specific keyboard-shortcut code exists in this repo at all: `iosMain` has no key-event handling, no `UIKeyCommand` usage, and no hardware-keyboard-specific `actual`/`expect` declarations (confirmed via grep across `kmp/src/iosMain` and the whole repo for `UIKeyCommand`/`hardwareKeyboard`/`externalKeyboard`). This means:
- All existing `onPreviewKeyEvent`/`onKeyEvent` handlers in `BlockEditor.kt`/`CommandPalette.kt`/`App.kt` are **common code** and should theoretically also fire on iOS when a hardware (Bluetooth/Smart) keyboard is attached to an iPad/iPhone, since Compose Multiplatform's `KeyEvent` API is exposed uniformly on iOS as of Compose 1.10.x (JetBrains has stabilized hardware keyboard input event dispatch on iOS in recent 1.7+/1.10.x releases) — but **this has never been tested in this repo** (no iOS-target key-event tests exist; `businessTest`/`commonTest` key-event tests, if any, run on JVM only via Robolectric/desktop, not the iOS target).
- Known category of gotcha to budget test time for in planning (Phase 3/4): iOS Compose Multiplatform's key-event dispatch historically lagged behind Android/Desktop for **modifier-key state** (`isMetaPressed` mapping to Cmd on an external keyboard is a newer addition and version-sensitive) and for intercepting **system-reserved shortcuts** (Cmd+Tab, Cmd+H, screenshot combos are never delivered to the app, which is expected OS behavior, not a bug to fix). Given "iOS is Gradle-only (no Bazel KMP yet)" per requirements.md, any iOS keyboard-shortcut verification in this project must run through the existing Gradle iOS toolchain, not Bazel, and should be manually verified on a simulator/device with an attached hardware keyboard — there is no automated CI coverage for this today, which is a testing-plan gap the validation phase (Phase 4) should call out explicitly rather than assume "if it compiles for iosMain it works."
