# Research: Build vs. Buy — Toolbar / Shortcut / Command-Palette Overhaul

**Project**: rich-editing-experience
**Question**: Should the toolbar/shortcut/command-palette overhaul be built from scratch (extending `BlockEditor.kt`'s hand-rolled `BasicTextField` approach) or sourced from an existing library?

## Baseline: what already exists in-repo

- `kmp/src/commonMain/kotlin/dev/stapler/stelekit/ui/components/BlockEditor.kt` (646 lines) — hand-rolled `BasicTextField` wrapper. `handleKeyEvent()` (line 219) is a plain `when`-based keyboard dispatch table: Ctrl/Meta+B/I/S/H/E map to `FormatAction` (bold/italic/strikethrough/highlight/code), Ctrl+Arrow does word navigation, Tab/Enter/Backspace handle outliner indent/split/merge. Regex-based autocomplete (`WIKI_LINK_AUTOCOMPLETE_REGEX`, `HASHTAG_AUTOCOMPLETE_REGEX`) drives `[[` and `#` triggers, matching the requirements.md baseline description.
- `kmp/src/commonMain/kotlin/dev/stapler/stelekit/ui/components/CommandPalette.kt` (225 lines) — already implements a hand-rolled **subsequence fuzzy match** (lines 41-57): a query is accepted if its characters appear as an ordered subsequence in the label (the same core matching primitive fzf/VS Code Go-to-anything use), then bucket-sorted by `startsWith` (0) / `contains` (1) / subsequence-only (2). This is a legitimate, well-known algorithm already in place — not naive substring search.
- `kmp/src/commonMain/kotlin/dev/stapler/stelekit/editor/RichTextEditor.kt` (281 lines) plus `editor/text/`, `editor/format/` packages — a **second**, separate hand-rolled rich-editor stack (`ITextOperations`, `IFormatProcessor`, custom `TextRange`) built on `BasicTextField`. This confirms the "hand-rolled Compose editor" pattern is already the repo's established, repeated convention, not a one-off.
- Project memory (`project_render_all_markdown.md`, 83 days old, verify-before-citing): the team explicitly decided custom Compose renderers > third-party libraries for markdown/block rendering, with LaTeX/math as the sole documented exception, specifically calling out **`multiplatform-markdown-renderer` and `compose-richtext` as libraries not to reach for**.
- No copy of the original ClojureScript/React Logseq source exists in this repo. The `./logseq` directory at repo root is the user's personal Logseq **graph data** (markdown journals/pages), not the Logseq application source — grep for "logseq" hits only README prose and this data directory.

## 1. Existing OSS library or framework

| Library | KMP targets | Editable or display-only | Maturity/license | Fit |
|---|---|---|---|---|
| `mikepenz/multiplatform-markdown-renderer` | Android, JVM, iOS (Kotlin/Native), Wasm/JS — genuinely broad coverage | **Display-only** — renders markdown to Compose UI, no text-input/editing model | Active, mature, Apache-2.0 | Not applicable — this project needs an editable toolbar/shortcut/palette layer, not a markdown viewer. Also the exact library the repo's own memory says not to use. |
| `halilozercan/compose-richtext` | Android + Desktop only — **no iOS target**, no stated Wasm support | Mix of display (`richtext-ui`) and a printer/markdown renderer | Pre-1.0, unclear release timeline | **Fails the hard requirement** (iOS/Web coverage) outright, regardless of other merits. Also named in project memory as a library to avoid. |
| `MohamedRejeb/compose-rich-editor` (`richeditor-compose`) | Android, iOS, Desktop, Web (Wasm) — the only candidate with real cross-target editing support | **Editable** WYSIWYG editor with `RichTextState`, span-style toggling, trigger-based popups for `@`/`#`/custom triggers | Apache-2.0, active, but still pre-1.0 (`1.0.0-rc14` at time of writing) — API surface not yet frozen | The only library that structurally fits "editable, cross-platform, has trigger popups." But: its core data model is `RichTextState` — an annotated-string/span-based WYSIWYG document, not a plain markdown-`String`/`TextFieldValue`. SteleKit's architecture treats markdown text as the source of truth end-to-end (files → `OutlinerPipeline` parser → `TextFieldValue` in `BlockEditor` → `GraphWriter` writes raw markdown back to disk). Adopting this library would mean maintaining a lossy bidirectional `RichTextState <-> markdown string` converter at every block boundary, on top of the existing custom parser — likely *more* code than extending the current `BasicTextField` dispatch table, not less. It also does not provide a command-palette component or a generic keyboard-shortcut dispatch table — those are app-level concerns this library doesn't address at all. |
| Cascade Editor / other block-based WYSIWYG editors (search hits, not deeply vetted) | Varies | Editable, Notion/Craft-style block model | Early-stage, low adoption signal | Same architectural mismatch as above (own block/document model, not markdown-source-of-truth), plus materially less community validation than `compose-rich-editor`. Not investigated further — didn't clear the first filter. |

**Verdict: Not recommended.** Nothing on the market provides toolbar + shortcut-dispatch + command-palette infrastructure that operates on plain markdown text the way this repo's architecture requires. The one credible editing library (`compose-rich-editor`) would require replacing the text-model layer (`TextFieldValue` + regex autocomplete + `OutlinerPipeline`) with an adapter to its own `RichTextState`, for a pre-1.0 dependency, to gain feature surface (WYSIWYG span toggling, mention popups) that largely duplicates what `BlockEditor.kt`'s `handleKeyEvent`/autocomplete regexes and `CommandPalette.kt`'s fuzzy matcher already do by hand. This confirms and extends the repo's existing documented decision (`project_render_all_markdown.md`) to prefer hand-rolled Compose over third-party rich-text libraries — the toolbar/shortcut/palette layer is not the LaTeX-style "genuinely unsolvable without a library" exception.

## 3. LLM-generated implementation vs. battle-tested library/algorithm, per sub-problem

| Sub-problem | Risk level | Recommendation |
|---|---|---|
| Keyboard shortcut dispatch table (Ctrl/Meta+key → `FormatAction`, navigation keys, indent/outdent) | **Low** — already hand-written as a flat `when` block in `handleKeyEvent()`. No hidden algorithmic complexity; correctness is enumerable and testable via direct `KeyEvent` unit tests. | Safe to hand-write / extend directly. No library needed. |
| Command-palette fuzzy search (if new palette entries are added) | **Medium in principle, already low here** — fuzzy string matching *does* have known-correct algorithms (subsequence matching as in fzf/`fuzzysort`; Smith-Waterman-style scoring for ranked highlighting) that are easy to get subtly wrong (e.g., not preferring contiguous/prefix matches, O(n·m) blowups on large lists). But this repo's existing `CommandPalette.kt` implementation already uses the correct core primitive (ordered-subsequence matching) plus a reasonable 3-tier bucket sort, and the corpus size is a command list (tens of entries), not a fuzzy-file-search index over thousands of paths. | Safe to extend the existing implementation as-is. If the palette grows to hundreds+ entries or needs highlighted match spans, adopt scoring conventions from a reference algorithm (e.g., prefer contiguous runs, boost matches at word boundaries — the approach fzf/VS Code's `fuzzaldrin-plus` document) rather than inventing new heuristics from scratch, but do not pull in a runtime dependency for this — porting the *scoring conventions* is enough. |
| Markdown auto-formatting on typed patterns (` ``` `, `1. `, etc.) | **Low-medium** — pattern-triggered auto-formatting (detecting a typed prefix and transforming it) is a well-trodden UX pattern (same shape as Markdown editors in Notion, Obsidian, GitHub's textarea auto-list-continue). The risk is in edge cases (cursor position after transform, undo/redo interaction, IME composition) rather than the algorithm itself, which is simple prefix/regex matching similar to what `BlockEditor.kt` already does for `[[`/`#`. | Safe to hand-write, following the same regex-trigger-on-`onValueChange` pattern already established in `BlockEditor.kt`. Test edge cases explicitly (cursor placement post-transform, undo, and IME composition on Android/iOS soft keyboards — see `detectSoftKeyboardBracketWrap()` at line 633 for the existing precedent of handling soft-keyboard quirks separately from hardware `KeyEvent`s). |

No sub-problem in this overhaul rises to the level of "reinventing a solved hard algorithm" (e.g., a real diff/merge algorithm, a CRDT, or a general parser) — the existing conventions in `BlockEditor.kt` and `CommandPalette.kt` are the right reference points to extend, not replace.

## Final Recommendation

**Build from scratch, extending the existing hand-rolled patterns in `BlockEditor.kt` and `CommandPalette.kt`.** This is consistent with the repo's already-established and twice-repeated convention (the render-all-markdown decision, and the existence of a second hand-rolled `editor/RichTextEditor.kt` stack) of preferring custom Compose code over third-party rich-text libraries. No candidate library clears the bar of "genuinely unsolvable without a library" the way LaTeX rendering did:

- Display-only libraries (`multiplatform-markdown-renderer`) don't address editing at all.
- The one library with real cross-platform *editing* support and trigger popups (`compose-rich-editor`) is architecturally incompatible with this repo's markdown-source-of-truth model (it owns its own `RichTextState` document model) and is still pre-1.0 — adopting it would mean building a lossy converter layer on top of a moving-target dependency, which is more risk and more code than extending the existing `TextFieldValue`-based dispatch table.
- `compose-richtext` fails the hard iOS/Web coverage requirement outright.
- None of the sub-problems (shortcut dispatch, palette fuzzy search, auto-formatting) are algorithmically risky enough to justify a dependency; the existing in-repo implementations already use the correct conventions (ordered-subsequence fuzzy matching, regex-trigger auto-formatting) and just need extension, not replacement.

The one qualifier: if the command palette's entry count grows substantially (hundreds+), borrow *scoring conventions* (contiguous-run and word-boundary bonuses) from established fuzzy-matcher literature (fzf/fuzzaldrin-plus) rather than inventing new heuristics — but implement it as local code, not a new dependency, per the "no new dependencies" constraint.
