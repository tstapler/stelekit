## Blockers (RE-REVIEWED 2026-07-05 — both RESOLVED)

- [x] **RESOLVED — Epic C.1 shape problem (three-state cycle vs. `FormatAction`'s `(prefix, suffix)` shape).** The plan no longer adds `FormatAction.TOGGLE_TODO`. It now specifies a genuinely separate sum type, `TodoState` (`NONE`/`TODO`/`DOING`/`DONE`), with `TodoState.parse(content): TodoState` and `TodoState.next(): TodoState` (Domain Glossary row, plan.md:49; Story C.1.1, plan.md:388-401; Task C.1.1a, plan.md:403-405). `FormatAction` itself is confirmed unchanged in the real code (`JournalsViewModel.kt:26-37`: `BOLD/ITALIC/STRIKETHROUGH/HIGHLIGHT/CODE/LINK/QUOTE/NUMBERED_LIST/HEADING`, no `TOGGLE_TODO` member) — the type-level separation is real, not just described. The stated cycle (`NONE→TODO`, `TODO→DONE`, `DOING→DONE`, `DONE→TODO`) is verified against `EssentialCommands.kt:441-446`'s actual (discarded) `newContent` computation — `content.startsWith("TODO ") -> DONE`, `"DOING " -> DONE`, `"DONE " -> TODO`, `else -> "TODO $content"` — and matches exactly, edge case included (`DONE` cycles back to `TODO`, not `NONE`, correctly called out in the plan's C.1.1 acceptance criteria at plan.md:395). No leftover `FormatAction.TOGGLE_TODO` references remain anywhere in plan.md except as the explicitly-rejected alternative in the Pattern Decisions table (plan.md:68) and one historical note in the Domain Glossary ("Replaces the originally-planned `FormatAction.TOGGLE_TODO` case," plan.md:49) — both correctly framed as superseded, not live. **PASS.**

- [x] **RESOLVED — Epic C.1 cross-contamination problem (TODO state vs. `FormatAction`'s mutually-exclusive line-prefix strip-group).** Verified the real strip-group logic at `BlockEditor.kt:513-516`: `FormatAction.entries.filter { it.suffix.isEmpty() && it.prefix.isNotEmpty() && text.startsWith(it.prefix) }.fold(text) { acc, other -> acc.removePrefix(other.prefix) }`. Since `TodoState` is a standalone enum never added to `FormatAction.entries`, this fold structurally cannot see or strip `TODO `/`DOING `/`DONE ` prefixes — the two strip-groups are mutually invisible to each other by construction, not by convention. The plan's dedicated `applyTodoToggle` (Task C.1.1b, plan.md:407-409) is specified to strip only the narrow set `{"TODO ", "DOING ", "DONE "}` and explicitly states it "never touches `FormatAction`'s existing mutually-exclusive line-prefix strip-group (`BlockEditor.kt:511-514`)" (Domain Glossary, plan.md:50; Pattern Decisions, plan.md:68). The required regression test now exists as its own task, C.1.1d ("Add cross-contamination regression test (HEADING vs. TODO orthogonality)," plan.md:415-416), with concrete Given-When-Then acceptance criteria in Story C.1.1 (plan.md:398-400): `HEADING` applied to a `TODO`-prefixed block yields `"TODO # Buy milk"` (todo marker preserved), and `applyTodoToggle` applied to a `HEADING`-prefixed block yields `"TODO # Buy milk"` (heading marker preserved) — both directions covered, with a stated composition rule (todo prefix stays outermost/leftmost) rather than an unspecified ordering. **PASS.**

## Concerns

- [ ] **New `requestStructuralAction`/`structuralActionEvents` (Domain Glossary, consumed by Epic C.2+/D/E stories)** — The plan specifies these as "new method + `MutableSharedFlow<StructuralAction>` on `BlockStateManager`" but doesn't say which of the four segregated port interfaces (`BlockEditingPort`, `BlockStructurePort`, `BlockSelectionPort`, `BlockNavigationPort` in `ui/state/BlockEditorPorts.kt`) it belongs to. `requestFormat`/`formatEvents` — the pattern this mirrors — is already a member of `BlockEditingPort`, not a loose method on the concrete `BlockStateManager` class. If `requestStructuralAction` is added only to the concrete class, any caller or test fake typed against the port abstractions (the codebase's existing ISP discipline) won't see it, quietly reintroducing a dependency on the concrete type at exactly the point the ports exist to prevent that.
  **Recommendation**: Add `requestStructuralAction: (StructuralAction) -> Unit` and `structuralActionEvents: SharedFlow<StructuralAction>` to `BlockStructurePort` (the closest existing conceptual home — it already owns indent/outdent/move), not as a bare addition to `BlockStateManager` alone.

- [ ] **Epic F.3 (`ShortcutTable`)** — `EssentialCommands.toggleTodo` (`editor/commands/EssentialCommands.kt:427-461`) already declares `shortcut = "Ctrl+Enter"`, which flows straight into the visible command-palette badge today (`StelekitViewModel.updateCommands()`: `Command(cmd.id, cmd.label, cmd.shortcut)`). That string is decorative metadata only — nothing in `handleKeyEvent` currently binds a real Ctrl+Enter handler. Epic C.1 is about to add a **real** hardware Ctrl+Enter binding for `TOGGLE_TODO` in the same project. If F.3's `ShortcutTable` reconciliation only fixes *pre-existing* badge/binding drift (per the audit finding it's designed around) and doesn't specifically re-derive `EssentialCommands.toggleTodo`'s badge from the real new binding, this project will ship a brand-new instance of the exact bug it exists to eliminate.
  **Recommendation**: Add an explicit acceptance criterion to Epic F.3 (or Epic H.2, whichever lands the repoint) that `block.toggle-todo`'s palette badge is sourced from `ShortcutTable`, not the hardcoded string in `EssentialCommands.kt`.

- [ ] **Epic H.2 vs Epic F.2 — duplicate/divergent "currently-editing-block" wiring** — `StelekitViewModel.updateCommands()` invokes `executeCommand(cmd.id)` with a default, empty `CommandContext()` (confirmed at call site, `StelekitViewModel.kt:2021-2029`) — no real block reference is threaded through today. Epic F.2 already identifies that it needs to thread "a currently-editing-block reference into StelekitViewModel's command assembly" for new `format.*` palette entries. Epic H.2's repoint of `block.toggle-todo` to `requestFormat(FormatAction.TOGGLE_TODO)` needs exactly the same reference (which block is being toggled), but the plan doesn't state that H.2 reuses F.2's plumbing — as written, they could be implemented independently and diverge (two different mechanisms for "what block does this palette action target").
  **Recommendation**: Make Epic H.2's story explicitly depend on and reuse whatever currently-editing-block mechanism Epic F.2 introduces, rather than re-deriving it.

- [ ] **`MobileBlockToolbar` callback-parameter bloat (pre-existing, touched by Epic D)** — `MobileBlockToolbar`'s signature already carries 23 parameters (confirmed by direct read: `editingBlockId`, 8 `on*` structural callbacks, format/link/tag/image callbacks, 8 selection-mode fields, `isLeftHanded`, `modifier`). The plan's containment strategy (route all *future* new structural actions through one new `onStructuralAction` callback) correctly stops the bleeding going forward and is the right call given the no-feature-flag blast-radius constraint — but it's worth naming explicitly that this is accepted, not fixed, debt, since Epic D touches this file row-by-row anyway.
  **Recommendation**: File a P3 gap-backlog.md row (or a `plan.md` footnote) noting the parameter count as acknowledged debt with a proposed future shape (e.g. a small `BlockStructureCallbacks` grouping data class) so it isn't silently re-discovered as a surprise later.

## Nitpicks
- `ToolbarButtonSpec` is described as "optional polish" with no concrete trigger condition for when it should actually be introduced — as written it could remain permanently deferred. Consider adding one concrete threshold to the Domain Glossary entry (e.g. "extract once a 4th call site needs the full icon+label+shortcut+enabled+onClick tuple").
- The FormatAction-vs-StructuralAction escalation rule ("model as FormatAction where pure text mutation suffices; escalate to StructuralAction only if a picker/dialog UI is required") is a judgment call without a worked example. Different implementers across Epics C.2/D/E could categorize a borderline case (e.g. code-block language picker) inconsistently. One worked example in the Domain Glossary would remove the ambiguity.
- `CommandRegistry`/`CommandSystem`/`CommandTypes` survive Phase H.2's trim as a thinner, purpose-narrowed bridge (per `ADR-001`'s already-accepted negative consequence, mitigated only by comments). Since Phase H is touching these files anyway, a cheap additional mitigation is renaming `CommandSystem`/`CommandRegistry` to something that signals "palette-ID bridge, not general framework" (e.g. `PaletteCommandBridge`) rather than relying on comments alone to prevent future misreading.
