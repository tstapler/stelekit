## Concerns
- [ ] Story F.1.1's focus/blur spike is gated correctly (F.2 blocked until PASS), and a FAIL does trigger "design a fix... before any Story F.2 task is started" — but no time/appetite buffer is reserved for a potentially invasive redesign (e.g., adding a replay cache to `formatEvents`, or bypassing the SharedFlow entirely) discovered this late (second-to-last phase) in an already-tight Large-appetite schedule. — Reserve explicit contingency time for Phase F in the schedule, or pre-decide a cheaper fallback (e.g., palette closes only after dispatch) so a FAIL doesn't turn into open-ended redesign.
- [ ] Epic F.3 (`ShortcutTable`) modifies `handleKeyEvent` — a file pitfalls.md itself flags as "hot," with a demonstrated same-day fix-forward history (commits `27fdc421c6`, `2b3b13a951`) — purely to fix a cosmetic shortcut-badge/binding display-drift issue that is not named anywhere in requirements.md's stated pain points. This expands blast radius in the highest-risk file in the whole plan for a nice-to-have, and contradicts the plan's own "additive-only, never destructive" risk-control claim (Task F.3.1b explicitly repoints *existing* shortcut checks, which is a modification, not an addition). — Either confirm with the user this is in-scope before Phase F, or demote it to a follow-up project; if kept, add a full existing-shortcut regression pass (not just "a case to KeyboardShortcutTest.kt") as an explicit acceptance criterion.
- [ ] Phase H (delete `SlashCommandHandler.kt`, both `RichTextEditor.kt` files, `PerformanceOptimizedEditor.kt`, two READMEs, plus trimming `CommandManager`/`CommandRegistry`/`CommandSystem`/`CommandTypes`) is pure dead-code housekeeping never requested in requirements.md — only Story H.2.1 (repointing the silently-broken `block.toggle-todo` palette entry) is tied to an actual user-facing bug. The rest rides along on this project's Large-but-bounded appetite budget without being asked for. — Confirm with the user whether Phase H's cleanup belongs in this project's scope or should be split into its own maintenance project; at minimum size it explicitly against the appetite budget rather than treating it as "free" because it's additive-safe.
- [ ] No task in Epic A.3 verifies that the ranked backlog is actually platform-balanced, despite "a prioritized, ranked gap backlog... covering every platform... with equal weight — no gap list is platform-biased" being a named success metric. Ranking is by severity only (P0 correctness > P1 > P2 > P3); iOS — which has zero existing shortcut/keyboard test infrastructure and "has never been exercised in this repo" per stack.md — is structurally more likely to be under-documented and therefore under-ranked than Desktop/Android, silently violating the equal-weighting requirement. — Add an explicit per-platform representation check to Story A.3.2/A.3.3 (e.g., "no platform contributes <20% of P0/P1 rows without a documented reason").
- [ ] `handleKeyEvent`'s if/else cascade keeps growing across Phases C, E, and F with no extraction into an independently testable dispatch structure — `ShortcutTable` (Phase F.3) only unifies *display strings*, not the dispatch logic itself. Every new shortcut's regression risk stays coupled to instantiating the full `BlockEditor` composable, and there is no story anywhere that addresses this compounding architecture risk even though it's flagged as a "hot" file. — Consider (out of this project if needed) extracting shortcut dispatch into a pure, unit-testable lookup structure, at least for the shortcuts this project adds.
- [ ] Task E.1.1a adds the `composition == null` IME guard but the plan specifies no automated regression test confirming existing (non-IME) `[[`/`#` autocomplete still fires correctly after the guard lands — only a manual CJK checklist item (Task E.1.1d) is specified. A bug in the guard itself (e.g. `composition` failing to reset on some platform/timing edge) would silently break autocomplete for every non-IME user with nothing automated to catch it before manual QA (which per pitfalls.md can't even reproduce IME behavior on JVM, so it wouldn't catch this either). — Add an automated non-IME regression case (composition == null, guard passes through) alongside the IME-blocked case in Task E.1.1a/b's test coverage.
- [ ] `StructuralAction`/`requestStructuralAction` is introduced in the Domain Glossary and Pattern Decisions as a new architectural pattern before any confirmed real use case exists — Phase C.2's own illustrative example (`FormatAction.CODE_BLOCK`) is modeled as a `FormatAction`, not a `StructuralAction`, per the Pattern Decisions table's own guidance ("only escalate to `StructuralAction` if a picker/dialog UI is required"). There's a real chance this entire new type ships with zero call sites — speculative generality introduced ahead of need. — Don't introduce `StructuralAction`/the SharedFlow pair until Phase A's real backlog confirms a concrete case that needs it (defer the type's creation to whichever story first requires it, per the glossary's own stated intent, rather than pre-declaring it as a Phase C artifact).
- [ ] The entire ranked backlog's priority ordering depends on `ux-expert` agent heuristic findings (Epic A.2.3) with no calibration or cross-check step against real usage signals or a second opinion. Since this backlog directly determines what code gets built in Phases C-G, an LLM agent's own systematic biases (e.g., over-weighting keyboard-shortcut-documented surfaces it has more training exposure to, vs. mobile/touch gaps) could shape the entire project's output with no way for a solo maintainer to detect the skew. — Add a lightweight sanity-check step (e.g., the solo maintainer manually spot-checks 2-3 of the agent's top-ranked findings against their own daily usage before backlog is finalized).
