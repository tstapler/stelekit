## Epic E.2 ŌĆö Browser-reserved-shortcut audit (Web), Task E.2.1a/b result (2026-07-06)

**Method: STATIC ANALYSIS ONLY, not empirical.** No browser session was available in the implementing environment to load a built `bazel build //kmp:web_app` artifact and press Ctrl+S/Ctrl+F/Ctrl+P directly, so this does not satisfy Story E.2.1's acceptance criteria as originally written ("tested on the WASM/Web target by pressing... in a focused block"). The findings below are grep/code-reading-based and are recorded honestly as inconclusive on the one open question, per this backlog's existing practice (cf. GAP-004's "not confirmed broken" framing) rather than being presented as a verified pass.

- **No WASM-specific key-event wiring exists.** `grep -rn "preventDefault\|addEventListener" kmp/src/wasmJsMain` finds only unrelated file-input (`WasmMediaAttachmentService.kt`) and worker-message (`SqliteWorkerInterop.kt`) listeners ŌĆö zero `keydown`/`preventDefault` handling. `browser/Main.kt` mounts via plain `ComposeViewport(document.body!!) { ... }` with no additional JS-level key interception layered on top. This is unchanged from pitfalls.md ┬¦2's original finding.
- **Ctrl+F and bare Ctrl+P are not consumed anywhere in this codebase.** Grepped `handleKeyEvent` (`BlockEditor.kt`) and `onGraphKeyEvent` (`App.kt`) for `Key.F`/`Key.P` ŌĆö only `Ctrl+Shift+P` (command palette) exists; no bare `Ctrl+P`, no `Key.F` binding at all. Since the app never calls `event.consume()`/returns `true` for these, there is nothing to suppress ŌĆö the browser's native Find/Print behavior passes through unmodified on Web. **Not a gap; no backlog row needed.**
- **Ctrl+S is consumed** by `handleKeyEvent` (mapped to `FormatAction.STRIKETHROUGH`) while a block has focus. Whether Compose Multiplatform 1.10.3's canvas actually calls the underlying DOM `preventDefault()` when `onPreviewKeyEvent` returns `true` (thereby suppressing the browser's native Save-page dialog) was **not empirically confirmed** ŌĆö a web search for JetBrains Compose Multiplatform documentation/issue-tracker guidance on this specific question did not surface a definitive answer either. This remains the same open question already tracked in plan.md's Unresolved Questions (now annotated with this static-analysis attempt, still unchecked).
- **A working fallback already exists regardless of the outcome above**: `MobileBlockToolbar.kt` renders a Strikethrough `IconButton` (`onClick = { onFormat(FormatAction.STRIKETHROUGH) }`, line 122) independent of the keyboard path, so Web users already have a non-keyboard route to the same action.
- **Verdict**: because nothing was *confirmed* unsuppressable (only left genuinely uncertain), **no new P0 backlog row is filed** ŌĆö Task E.2.1b's condition ("confirmed-unsuppressable shortcut lacking a fallback") is not met on either prong (not confirmed, and a fallback exists). This is recorded as a still-open to-verify item, not a shipped regression, and not silently dropped ŌĆö see plan.md's Unresolved Questions for the full write-up and next-owner note.
