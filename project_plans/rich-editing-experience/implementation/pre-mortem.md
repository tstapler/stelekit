# Pre-mortem: rich-editing-experience
**Date**: 2026-07-05

## Failure Modes

| # | Failure | First Symptom | Prevention | Severity |
|---|---------|--------------|------------|----------|
| 1 | Solo maintainer runs out of appetite mid-plan: Phase A alone (11 journey docs × authoring + benchmarking + heuristic review + backlog reconciliation, ~30+ tasks) consumes a disproportionate share of the 3-6 week Large appetite before a single user-facing toolbar/shortcut/palette change ships, forcing a choice between an overrun or quietly reverting to "audit-only" — which requirements.md explicitly says the user rejected. | Two-plus weeks elapsed and the project is still inside Phase A/early Phase C with zero shipped PRs touching `MobileBlockToolbar.kt`/`handleKeyEvent`/`CommandPalette.kt`; `gap-backlog.md` exists but no phase past B has a merged PR. | Add a hard time-box/checkpoint to plan.md: if Phase A + Phase B are not both merged within the first ~25% of elapsed appetite, trigger an explicit scope-cut review (using the existing cut-not-grow triage rule machinery, but applied to *phase count*, not just story count within a phase) — e.g., pre-declare that Phase G2 (voice-mode) and/or Phase H (cleanup) are the first things dropped if the schedule slips, rather than discovering this ad hoc under pressure. | P1 |
| 2 | The single `ux-expert` agent pass misjudges a journey's real-world severity (e.g., ranks a rarely-hit path P0 while a high-frequency pain point lands P2), and the cut-not-grow triage rule then structurally locks that ranking in — cutting real backlog capacity to fit an incorrect priority order. The error isn't caught until a later phase ships and real usage (or a user bug report) contradicts the audit, by which point the "wrong" fix is already merged, additive code, not a spreadsheet row. | A shipped Phase C/D/E fix (e.g., a new toolbar button or shortcut) that the audit called high-priority draws no usage/positive feedback, while users independently complain about a different, lower-ranked journey the audit under-weighted. | Add a lightweight falsifiability check to `gap-backlog.md`: for each P0/P1 row, require one concrete, checkable piece of *evidence* beyond the agent's own narrative (e.g. an existing GitHub issue, a `features.md`/`pitfalls.md` citation, or a repro step a human can verify in under a minute) before it's allowed to consume triage-rule budget — this doesn't need a second agent, just a non-agent evidence gate the solo maintainer can apply in ~1 minute per row. | P2 |
| 3 | Ctrl+Enter's new TODO-toggle behavior collides with real muscle memory: today Ctrl+Enter silently falls through to plain-Enter's "split into new block" behavior (per features.md §1), so users who habitually chord Ctrl+Enter (common in other editors as an "insert without autocomplete"/"confirm" gesture) have months of experience where it behaves just like Enter. After Phase C.1.2 ships, the exact same keystroke now toggles TODO state instead — a real, felt behavior change for existing users, even though the plan's bookkeeping calls it "additive" (no `FormatAction` was remapped/removed). | Bug reports of the shape "I pressed Ctrl+Enter expecting a new block and got a TODO marker instead" — a UX regression despite passing every acceptance criterion in Story C.1.2. | Add an explicit acceptance criterion to Story C.1.2 acknowledging this as a *user-observable* behavior change (not just a structurally-additive one), and ship it with a one-time in-app notice/toast ("Ctrl+Enter now toggles TODO — use Enter to split a block") for the first N days post-release, plus a task to dogfood the collision manually (not just the two isolated branch tests in C.1.2b) before merging. | P2 |
| 4 | Phase C.1.1c introduces `todoToggleEvents` (a `MutableSharedFlow<Unit>` with no replay cache) using the exact same shape as `formatEvents` — the shape Phase F.1's spike exists specifically to empirically distrust. But Phase C ships before Phase F's spike runs, and F.1.1's test scope is limited to `CommandPalette` + `FormatAction` (per its acceptance criteria), not `todoToggleEvents`/`structuralActionEvents`. If Phase D or G later wires a TODO-toggle or structural-action trigger through any Dialog/BottomSheet-hosted UI (e.g., a toolbar button inside a `SuggestionBottomSheet`-style overlay), the same silent-drop failure this whole project exists to eliminate (`block.toggle-todo`'s precedent) can reappear under a new name, undetected by any test in the plan. | Tapping a TODO-toggle (or structural-action) trigger from inside a Dialog/BottomSheet silently does nothing — no crash, no error, no content change — discovered only via manual testing or a user report after Phase D/G ship. | Extend Story F.1.1's spike (or add a sibling test) to cover `todoToggleEvents`/`structuralActionEvents` through any Dialog/BottomSheet-hosted trigger, not just `CommandPalette`+`FormatAction` — and add this as a blocking prerequisite to any Phase D/G task that wires a Dialog-hosted button to `requestTodoToggle`/`requestStructuralAction`, mirroring the existing "F.1 MUST PASS before F.2" gate. | P1 |
| 5 | The cut-not-grow triage rule (Story A.3.3) only specifies what happens when real P0/P1 backlog rows *exceed* 2x a phase's placeholder baseline — it says nothing about the opposite case, where a phase's real backlog turns out thin (e.g., `ux-expert` ranks most of that phase's candidates P2/P3). In that gap, Task C.2.2a's "confirm or substitute" instruction has no forcing function, so an implementer under time pressure can just leave the illustrative placeholder (e.g., `FormatAction.CODE_BLOCK`) in place unchanged — shipping audit-blind, made-up-before-the-audit-existed features while the success metric ("highest-priority gaps IMPLEMENTED") is silently unmet for that phase. | A phase's shipped PR implements exactly the illustrative placeholder from plan.md with no `gap_id` citation in the commit/PR description, even though `gap-backlog.md` exists and was populated. | Add an explicit acceptance criterion to each reconciliation story (A.3.3, C.2.2, D.1.1/D.2.1/D.3.1, E.3.1, F.2.1) requiring the merged PR description to cite the real `gap_id`(s) implemented — if the real backlog has no qualifying P0/P1 row for that phase, the task must be explicitly marked "no real backlog match — placeholder dropped" rather than silently implemented as originally written. | P2 |

## P1 Items (address before implementation)
- [x] Failure #1 — RESOLVED: plan.md's Risk Control section now has an explicit Phase A time-box (~1 week / ~20% of appetite) with a concrete fallback rule (ship partial backlog/journeys, mark rest `audit-deferred`, proceed to Phase B/C rather than stalling on audit completeness).
- [x] Failure #4 — RESOLVED: Story F.1.1's acceptance criteria and Task F.1.1a now also cover `requestTodoToggle()`/`todoToggleEvents` dispatch (not just `FormatAction`/`formatEvents`), and Epic C.1's Goal notes the shared-shape risk explicitly, pointing forward to this widened spike as the empirical verification point.

## Summary
2 P1, 3 P2, 0 P3. Top failure mode: the sequential 9-phase, 85-task plan has no time-box on Phase A's audit work, so a solo maintainer can burn most of the 3-6 week appetite on documentation/benchmarking before any user-facing toolbar/shortcut/palette change ships — directly risking the explicit requirement that gaps be *implemented*, not just audited.
