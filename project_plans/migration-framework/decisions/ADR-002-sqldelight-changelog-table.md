# ADR-002: SQLDelight Per-Graph Changelog Table

**Status**: Accepted
**Date**: 2026-04-14
**Deciders**: Tyler Stapler

---

## Context

The migration framework must persist which migrations have been applied to each graph so that re-runs are idempotent. This requires a durable store scoped per graph. Two options were evaluated:

**Option A — SQLDelight table in the existing per-graph SQLite database**

Add a `migration_changelog` table to `SteleDatabase.sq`. Each graph's `.db` file carries its own migration log, scoped by `graph_id`. Queries are generated by SQLDelight 2.3.2 (already in the project). Writes go through the existing `DatabaseWriteActor` for serialization.

```sql
CREATE TABLE migration_changelog (
    id              TEXT NOT NULL PRIMARY KEY,
    description     TEXT NOT NULL,
    checksum        TEXT NOT NULL,
    applied_at      INTEGER NOT NULL,
    execution_ms    INTEGER NOT NULL DEFAULT 0,
    status          TEXT NOT NULL DEFAULT 'APPLIED',
    applied_by      TEXT NOT NULL DEFAULT '',
    graph_id        TEXT NOT NULL,
    changes_applied INTEGER NOT NULL DEFAULT 0,
    error_message   TEXT
);
```

**Option B — JSON sidecar file per graph (`.stele-migration-log.json`)**

A `kotlinx.serialization` JSON file written alongside the graph directory. Human-readable and git-diffable. Mirrors the pattern used by SteleKit's existing op-log sidecar.

```kotlin
@Serializable
data class MigrationRecord(
    val id: String,
    val checksum: String,
    val appliedAt: String,   // ISO-8601 string (avoids Long precision loss on JS)
    val status: MigrationStatus
)
```

The sidecar file is written atomically via Okio `FileSystem.atomicMove()`. No SQLite dependency.

**Stack research finding** (from `research/stack.md`): The research initially recommended Option B to avoid a circular dependency: "if migrations run before graph DB open, SQLDelight creates a circular dependency." However, the architecture research (`research/architecture.md`) clarified the actual integration point: `MigrationRunner` runs *after* `RepositoryFactory.createRepositorySet()` inside `GraphManager.switchGraph()`, at which point the SQLite database is already open. The circular dependency concern does not apply.

---

## Decision

**Option A: SQLDelight table in the existing per-graph SQLite database.**

---

## Rationale

1. **Database is already open at migration time**: `MigrationRunner.runPending()` is called after `createRepositorySet()` returns, which means the SQLite connection and `SteleDatabase` instance are already initialized. There is no circular dependency.

2. **Transactional consistency**: SQLite transactions wrap the changelog write alongside the block changes. If the process dies between applying block changes and writing the changelog record, the `status = 'RUNNING'` guard (set before any writes begin) allows detection and recovery on restart. A JSON file write has no such atomicity guarantee.

3. **Queryability**: `SELECT * FROM migration_changelog WHERE graph_id = ? AND status = 'APPLIED'` is O(1) with an index. Reading and parsing a JSON sidecar requires full file I/O and deserialization on every startup — a performance regression for graphs with many migrations.

4. **Reuse of existing infrastructure**: `ContentHasher.sha256()` already exists in `util/`. SQLDelight generates type-safe query methods. `DatabaseWriteActor` serializes all writes. No new libraries required.

5. **Multi-graph correctness**: The `graph_id` column scopes every row to its graph. Migrations applied to Graph A cannot contaminate Graph B's changelog.

6. **Consistency with `UuidMigration` pattern**: The existing `UuidMigration` stores its completion state in the `metadata` table (also SQLDelight). The migration changelog follows the same architectural pattern.

---

## Consequences

- `SteleDatabase.sq` gains a `migration_changelog` table definition and its associated named queries (`insertMigrationRecord`, `selectAppliedMigrations`, `updateMigrationStatus`, `selectMigrationById`).
- `applied_at` is stored as epoch milliseconds (`INTEGER`). On Kotlin/JS, the `MigrationRunner` only reads this value back for display — it is not deserialized into a `Long` on the JS target via `kotlinx.serialization`, so the Long-precision issue from `pitfalls.md` §4c does not apply.
- A `status` index on `(graph_id, status)` enables efficient "find all RUNNING migrations on startup" queries.
- The JSON sidecar approach (Option B) remains available for a future "portable changelog" feature where users can export their migration history as a human-readable file alongside the graph.
