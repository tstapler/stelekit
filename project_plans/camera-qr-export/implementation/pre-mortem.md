# Pre-mortem: camera-qr-export
**Date**: 2026-07-11

## Failure Modes

| # | Failure | First Symptom | Prevention | Severity |
|---|---------|--------------|------------|----------|
| 1 | The feature ships behind an opt-in flag solving a genuinely rare problem (air-gapped, no-network device pairing) with zero existing UX convention in Logseq/Obsidian to anchor discovery — most users, including the solo developer's own day-to-day workflow (which almost always has a network path available), never have a reason to reach for it, so it ships, is "dogfooded" once, and then bit-rots unused. | No forum/issue-tracker mentions of the feature after release; the developer notices months later they haven't personally used it since the initial demo; no crash reports or bug filings either way because nobody is exercising the code path. | Add a real, sustained-use dogfooding bar to Risk Control / Definition of Done — e.g. require the developer to use the feature for actual personal transfers (not staged demos) over a period of weeks before broad enablement — and add a local (no-network, no-telemetry-service) usage log counter surfaced in a debug/settings screen so "is anyone using this" becomes an observable fact instead of an assumption, without violating the "no new alerting/oncall" NFR. | P1 |
| 2 | UQ-1's empirical max-payload spike (Task 2.1.3a) is the one task requiring physical print/scan measurement rather than code — it gets "wired" (the settings field and provisional default of `maxFragmentBytes=512`/EC-M ship) but the actual arm's-length manual scan measurement is skipped under time pressure, since nothing in the v1 Definition of Done checks that UQ-1 was actually resolved rather than left at its documented fallback. | Encoder chunk sizing and fps defaults never change from their day-1 provisional values; nobody can tell from the code whether they reflect a measurement or a guess; if a real transfer is unusually slow or unreliable, there is no evidence trail showing whether the density/fps tradeoff was ever actually tuned. | Promote UQ-1 (and its Task 2.1.3a) from "Unresolved Questions" to an explicit, checked line item in the "v1 Definition of Done" section of plan.md — sign-off should require either a recorded measurement result or an explicit, dated decision to ship the provisional default and accept the risk, not silent inheritance of the fallback. | P2 |
| 3 | Story 3.3.5 (hardware-in-the-loop reliability pass) requires two physical devices and manual labor, is explicitly called out as the one task that "isn't CI-enforceable," and is **not listed in the v1 Definition of Done** (which only cites the *simulated* Story 3.3.1 round-trip test) — so it is the single most likely task to be silently dropped under time pressure, shipping a feature whose entire value proposition is "works when you have no other option" without ever having validated it against real camera artifacts (glare, motion blur, autofocus hunting, correlated/bursty frame loss) that the synthetic random-drop model in 3.3.1/1.2.4 does not represent. | The feature demos fine in the developer's own controlled lighting/distance setup, but the first time someone actually needs it under real, imperfect conditions (dim room, phone held at an angle, screen glare), the transfer stalls or never converges — precisely the moment the "no network fallback" feature is supposed to be trustworthy and isn't. | Add Story 3.3.5 (or at minimum "N≥20 real transfers across varied lighting/distance/angle recorded") as an explicit, checked entry in the "v1 Definition of Done" list in plan.md, and make clear that Story 3.3.1's CI-green simulated test does NOT satisfy requirements.md's "hardware-in-the-loop testing pass, not just unit tests" ask on its own — the flag should not be enabled beyond the developer's own device until 3.3.5 is done or its omission is an explicit, dated, accepted risk. | P1 |
| 4 | ADR-003's "Android-round-trip-first, Desktop/iOS send-only" sequencing assumes the primary JTBD is "beam a page onto my phone," but a very plausible dominant real trigger for a note-taking app is the reverse — capturing a quick note on a phone and wanting it on the desktop to edit/paste/publish from — which v1 architecturally cannot serve at all (Desktop/iOS receive is deferred to Phase 4), frustrating exactly the users most motivated to use an air-gapped transfer feature in the first place. | The developer's own first real (non-demo) use case, or first user feedback, is "I need this the other way around" — a request to flip the direction that the v1 architecture has no answer for, revealing the ADR-003 sequencing bet was wrong about which direction people actually need. | Before committing to the full Phase-4 deferral, explicitly test ADR-003's "primary JTBD satisfied phone-side" assumption as a hypothesis, not a settled fact: have the developer use v1 send-only for a real reverse-direction (phone→desktop) task during dogfooding, and record in plan.md whether Desktop-receive (Epic 4.3) needs to be pulled forward into the v1 appetite rather than deferred, before broad rollout. | P2 |
| 5 | The ≤3 fps WCAG ceiling (ADR-004) forces transfer duration up, which means a human has to hold a phone steady, handheld, at arm's length, pointed at another screen for the entire 30–60s+ transfer — a physical ergonomics problem no story or acceptance criterion addresses. Hand tremor from fatigue causes motion blur, which drops frames, which lengthens the transfer further, which increases fatigue — a vicious cycle never modeled by the synthetic (random-drop) or even the planned hardware-in-the-loop tests if those tests use a stabilized/braced phone rather than genuinely handheld use. | Real-world transfers reliably run much longer than the 30–60s quality bar specifically for handheld (non-propped) use, even though tripod/braced hardware-in-the-loop tests looked fine — a gap between "measured reliable" and "actually usable one-handed." | Add an explicit "handheld, unsupported, held by hand for the full duration" condition to Story 3.3.5's real-world test matrix (not just varied lighting/distance with an implicit stable mount), and add a UX tip to Story 3.2.3's decoder screen suggesting the user prop the phone against something stable during longer scans, so the 30–60s bar is validated against how people will actually hold the device, not just how the developer holds it during a careful test. | P2 |

## P1 Items (address before implementation)
- [x] #1 — Added a real sustained-use dogfooding requirement and a local (non-telemetry) usage counter to plan.md's "v1 Definition of Done" section, so adoption is measured rather than assumed before the flag is ever enabled broadly.
- [x] #3 — Added Story 3.3.5 (hardware-in-the-loop reliability pass, ≥20 real transfers across lighting/distance/angle, including a handheld/unpropped condition) as an explicit, required line in plan.md's "v1 Definition of Done" section; clarified that Story 3.3.1's simulated CI test does not substitute for it.
