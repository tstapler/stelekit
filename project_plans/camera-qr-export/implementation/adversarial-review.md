## Minors
- **Citation over-correction on the `QrEncodeUiState`/`QrDecodeUiState` sealed-state precedent (Domain Glossary line 75, Pattern Decisions line 92).** The prior review's minor said the plan should say `DepthModelDownloader.ModelState`, not `DepthModelUiState`. The fix did exactly that — but this trades one imprecision for another. `DepthModelDownloader.ModelState` is `androidMain`-only (`kmp/src/androidMain/kotlin/dev/stapler/stelekit/platform/ml/DepthModelDownloader.kt`). The actual **commonMain** analog — the layer `QrEncodeUiState`/`QrDecodeUiState` actually live in — is `DepthModelUiState` (`ui/annotate/AnnotationEditorViewModel.kt`), whose own KDoc calls itself "Platform-independent mirror of `DepthModelDownloader.ModelState` for common UI." Worse, Pattern Decisions line 92 attributes the `DepthModelDownloader.ModelState` citation to "(features research)," but `project_plans/camera-qr-export/research/features.md` (lines 33, 77, 90) only ever names `DepthModelUiState` — the plan's citation now contradicts the research doc it claims to cite. Recommend: cite both, e.g. "per `DepthModelUiState` (commonMain mirror of `DepthModelDownloader.ModelState`)."
- **Task 2.1.2b Fallback B ("sequence Gradle-first") sits in tension with this repo's own stated build policy.** CLAUDE.md reserves Gradle for "iOS (no Bazel KMP support yet), screenshot tests (Roborazzi), and benchmarks... until those are migrated" — it does not name JVM/Android feature work as a sanctioned Gradle-first use case. The fallback is reasonable as a narrowly-scoped, explicitly-tracked contingency (fast-follow task named), so this isn't a functional problem, just worth a one-line acknowledgment in the task notes that it's a temporary, tracked deviation from the Bazel-canonical rule, not a new precedent.
- **requirements.md's gate wording was not reconciled.** The original blocker's recommendation was "add an explicit Out-of-Scope entry to plan.md **and** reconcile requirements.md's gate wording, or add a minimal multi-page path." The fix did the first (plan.md now has an explicit, prominent "v1 scope — explicit cuts" section) but requirements.md's Success Metrics line 19 still reads "a page (or set of pages)" with no v1 pointer back to the cut. Not a blocker — plan.md is the authoritative Phase 3 artifact and the cut is now fully visible there, and plan.md itself explains how requirements.md's framing is satisfied — but a reader of requirements.md alone still sees the original ambiguous wording.

### Concerns (6/6)

- **No hardware-in-the-loop reliability test**: new Story 3.3.5 / Task 3.3.5a (`plan.md` lines 615–623) — ≥20 real transfers across varied lighting/distance/angle, measuring frame-loss/false-decode rates and feeding back into `maxFragmentBytes`/`framesPerSecond` defaults, explicitly distinguished from the density spike (2.1.3a) and the synthetic lossy-channel tests (1.2.4b, 3.3.1). Also now resolves UQ-2 (steady-state frame-interval floor), which previously had no owning task.
- **`Paused` declared but never wired**: Story 3.1.2 (`plan.md` lines 427–439) now specifies explicit `pause()`/`resume()` triggers, exact-frame-position preservation, and a dedicated AC; Task 3.1.2c adds a unit test asserting `pause()`/`resume()` preserve `frameIndex`.

Verified against the actual codebase: `tags/TagSettings.kt` — `class TagSettings(private val platformSettings: Settings)` — confirms the precedent is real, and `grep -rn multiplatform-settings kmp/build.gradle.kts MODULE.bazel` returns nothing, confirming no such dependency exists or is needed.
- **Unrealistic time estimates + no automated Desktop/iOS send test**: new Task 4.1.1b (JVM `QrCodec.encode` same-process round-trip test) and Task 4.2.1b (iOS same-process round-trip test) close the "manual demo only" gap. - **`validateName` traversal gap + `fc5661eb14` mischaracterization**: the commit citation is now explicitly corrected in-plan (line 497: "the prior citation of commit `fc5661eb14` was **incorrect**..."). A new AC (Story 3.2.1, lines 495–497) confirms the imported name is never used to build a raw filesystem path. Verified against the actual codebase: `DatabaseWriteActor.kt` has separate `savePage` (line 684) / `saveBlocks` (line 695) calls with no atomic combined method, and `deletePage` (line 873) exists for the compensating rollback exactly as claimed.
