# Stack Research: camera-qr-export

Agent 1 (Stack) — Phase 2 research, 2026-07-11.

## Codebase orientation (facts that constrain the stack choice)

- `CameraProvider` (`kmp/src/commonMain/kotlin/dev/stapler/stelekit/platform/sensor/CameraProvider.kt`) currently exposes **one method**: `suspend fun capturePhoto(): Either<DomainError.SensorError, PlatformImageFile>` — a single still-photo capture returning a file path. **There is no live-frame-stream API today.** Implementations: `AndroidCameraProvider` (CameraX), `IOSCameraProvider` (AVFoundation, comment references "CameraK or direct Swift interop"), `WebcamCameraProvider` (JVM desktop — comment in file says desktop is import-only, no live capture), `WebCameraProvider` (WASM/JS, stubbed — comment references `navigator.mediaDevices.getUserMedia()` but not implemented), `NoOpCameraProvider` (fallback/tests).
- **Implication for this feature**: the requirement to reuse `CameraProvider` "rather than introducing a second camera API surface" will require **extending the interface** with a continuous frame-stream method (e.g. `Flow<PlatformFrame>` or callback-based `analyzeFrames`), not just calling the existing `capturePhoto()`. This is an architecture/design decision for Phase 3, but it changes the shape of every platform's scanning integration — CameraX already supports an `ImageAnalysis` use case (frame callbacks) alongside its existing `ImageCapture` use case, so Android is the least-risk extension. iOS/AVFoundation and Desktop/JVM (Webcam) need a new frame-callback path added to their respective actuals. WASM has no live capture at all yet.
- Existing Maven deps relevant here (`kmp/build.gradle.kts`, mirrored in `MODULE.bazel` `artifacts = [...]`):
  - `com.squareup.okio:okio:3.17.0` (+ `okio-fakefilesystem` in tests) — already a first-class multiplatform dependency.
  - CameraX 1.4.1 (`androidx.camera:camera-core/camera-camera2/camera-lifecycle`) on Android only.
  - No QR/barcode library, no fountain-code library, and no explicit compression library present anywhere in the repo today (`grep` for zxing/qrcode/barcode/deflate/gzip in build.gradle.kts and MODULE.bazel returned nothing beyond CameraX).
  - No `gradle/libs.versions.toml` exists in this repo — Gradle deps are declared as raw coordinate strings directly in `kmp/build.gradle.kts`; Bazel deps are declared as raw coordinate strings in `MODULE.bazel`'s `maven.install(artifacts = [...])` block, locked via `rules_jvm_external++maven+maven_install.json`. Any new dependency must be added to **both** files, in both places, using matching versions, and the Bazel lock file regenerated per the `MODULE.bazel` comments (`bazel run @unpinned_maven//:pin` pattern implied by the `lock_file` directive).

## 3. Fountain-code / rateless erasure coding for chunked transfer

- **Blockchain Commons UR (Uniform Resources) spec** uses **Luby Transform (fountain) codes** for its "multipart animated QR" transport — exactly the prior art named in the requirements doc, and it is a mature, field-tested spec (used by Sparrow Wallet, Keystone, ColdCard, Specter Desktop/Sparrow PSBT transfer flows).
- **Reference/port landscape**:
  - `BlockchainCommons/bc-ur` — canonical C++ reference implementation.
  - `BlockchainCommons/bc-ur-java` — Java **wrapper around the C++ library** (JNI-style bridge, not a pure-JVM port) — likely awkward for a KMP `commonMain` target since it pulls in native C++ binaries per-platform.
  - **`sparrowwallet/hummingbird`** — a **pure Java, direct port** of Wolf McNally's URKit (the reference Swift/Java implementation lineage), published to Maven Central (`com.sparrowwallet:hummingbird:1.7.4`), requires only Java 8, MIT-style license (verify exact license in repo), field-proven inside Sparrow Wallet's production airgapped-signing flow. This is the strongest lead for JVM/Android reuse — plain Java, no native deps, already solves UR encode/decode + fountain-code chunking + CBOR framing together.
  - **iOS**: no Kotlin/Native-native port exists; would need either (a) a Swift binding to Blockchain Commons' own Swift `URKit` (first-party, used in `BlockchainCommons/URDemo`) exposed via cinterop/Swift export, or (b) re-implementing the LT-code logic in pure Kotlin `commonMain` from scratch, which sidesteps the interop problem entirely and gives one implementation for all 4 targets.
  - **Pure-Kotlin/JVM generic LT-code implementations** (not UR-specific, no CBOR/bitcoin baggage): `mwdchang/fountain-code` (small reference Luby Transform implementation), `smileboywtu/LubyTransformCode` (Python+Java). These are toy/reference scale, not production-hardened, but usable as an algorithmic starting point for a from-scratch pure-Kotlin `commonMain` implementation — likely the more portable option given iOS has no direct bc-ur/hummingbird port.
- **Recommendation direction**: given (a) iOS is a hard KMP-native target with no existing JVM-style library to lean on, and (b) the requirements explicitly frame full UR-spec compliance as unnecessary (this is a SteleKit-to-SteleKit protocol, not required to interoperate with hardware wallets), **writing a small pure-Kotlin `commonMain` LT-code encoder/decoder** (droplet degree distribution + XOR combination + Gaussian-elimination-style decode, following the Luby Transform algorithm description referenced in the fountain-codes literature) is likely lower long-term risk than bridging three different platform-native UR libraries. `hummingbird`/`bc-ur` remain valuable **reference implementations to study algorithmically** (chunk numbering, degree distribution choice, checksum/CRC framing) even if not linked as a dependency.

## 4. Compression / binary framing for markdown payload

- **Okio 3.17.0 is already a dependency** (`kmp/build.gradle.kts`) and — as of recent Okio 3.x — `GzipSink`/`GzipSource`/`DeflaterSink`/`InflaterSource` are **no longer JVM-only**; Okio added Kotlin/Native support for these compression types, and Okio's own docs (klibs.io) list JVM, Kotlin/Native, Wasm, and JS as supported targets, with WASM support marked "experimental but improving." **Action for Phase 3**: confirm empirically (small spike) that `okio:3.17.0`'s GzipSource/Sink actually resolves and executes correctly on the `wasmJs` and `iosX64/iosArm64` targets this project already builds for — do not assume from changelog prose alone, since WASM support is explicitly flagged experimental upstream.
- **Fallback/alternative multiplatform compression libraries** (only needed if Okio's multiplatform Gzip has gaps on a specific target):
  - `henrik242/kmp-zip` — KMP ZIP/GZIP library, JVM + iOS/macOS/Linux/Windows native + wasmJs, wraps `java.util.zip` (JVM), `platform.zlib` (Apple/Linux/Windows native), `pako` (wasmJs). Published to Maven Central, no special repo config.
  - `karmakrafts/Kompress` — Deflate/Inflate-only KMP API (`Deflater.deflate`/`Inflater.inflate`), narrower scope than kmp-zip.
- **Custom binary framing**: regardless of which gzip/deflate library is used, the protocol needs a compact **chunk header** (sequence/degree info for the fountain code, total-payload-length, CRC/checksum, page-count) ahead of the compressed markdown body — this is bespoke and language-agnostic; `kotlinx.serialization` (already a dependency, `kotlinx-serialization-json`) could serialize a small header object, though for QR payload density a hand-rolled fixed-width binary header (few bytes) is likely to be more space-efficient than JSON — worth comparing binary vs. CBOR (a natural pairing with any UR-style approach) vs. hand-rolled framing in Phase 3.
- **Base rationale**: raw markdown compresses well with deflate (structured, repetitive text), and deflating before fountain-chunking directly reduces frame count, which is the stated performance lever in the requirements ("payload density vs. frame count trade-off").

## Open questions / risks flagged for Phase 3 (planning) and Phase 4 (validation)

1. Does `qrcode-kotlin` 4.5.0 actually publish a `wasmJs` (not just legacy `js`) artifact? Web target in this repo is WASM/JS per `enableJs=true` — must confirm exact target compatibility before committing.
2. `CameraProvider` needs a new frame-stream method (interface change) — this is a Phase 3 architecture decision, not just a dependency choice, and touches all 5 current implementations (`AndroidCameraProvider`, `IOSCameraProvider`, `WebcamCameraProvider`, `WebCameraProvider`, `NoOpCameraProvider`).
3. No existing fountain-code dependency is both (a) pure-Kotlin-portable to iOS and (b) production-hardened — recommend a from-scratch pure-Kotlin `commonMain` LT-code implementation informed by, but not dependent on, Blockchain Commons' bc-ur/hummingbird prior art.
4. Web/WASM camera capture (`WebCameraProvider`) and decode (`BarcodeDetector`/jsQR) are both the least mature legs of the "any↔any" 6-combination matrix named as a rabbit hole in requirements — confirm whether v1 truly needs Web send/receive or whether it can be deferred within the "Large" appetite.
