# UX Research: camera-qr-export

Agent 5 (UX Research), Phase 2 — SDD for `camera-qr-export`.

## 0. Codebase context (existing patterns to stay consistent with)

- **Camera permission rationale**: `kmp/src/commonMain/kotlin/dev/stapler/stelekit/ui/annotate/PermissionRationaleDialog.kt` defines `CameraPermissionRationaleDialog(onNotNow, onContinue, isPermanentlyDenied)` — an `AlertDialog` with a `CameraAlt` icon, plain-language rationale ("SteleKit needs camera access to photograph construction sites directly from the app. You can also import existing photos without camera access."), a **non-punitive escape hatch** ("Not now" + an alternative path that doesn't require the permission), and a permanently-denied variant that changes the confirm button to "Open Settings". `BlePermissionRationaleDialog` in the same file follows an identical shape (icon, rationale text branching on `isPermanentlyDenied`/`requiresLocation`, dismiss = fallback method, confirm = grant). **This is the template to reuse** for camera-qr-export's permission ask on the *decode* side (the *encode* side needs no permission — it only displays QR frames).
- **Capture confirmation pattern**: `CapturePreviewDialog.kt` (`kmp/src/commonMain/kotlin/dev/stapler/stelekit/ui/components/`) shows a full-screen `Dialog` with a preview, a disabled-during-work `Save` button that swaps its label for a `CircularProgressIndicator`, and a `Discard` escape. The "action is in flight, button disables + spinner replaces label, dialog stays up so the user knows work is happening" pattern should carry over to the import-confirmation step after a fountain-code decode completes.
- **`CameraProvider` interface** (`kmp/src/commonMain/kotlin/dev/stapler/stelekit/platform/sensor/CameraProvider.kt`) is **single-shot photo capture only** (`suspend fun capturePhoto(): Either<DomainError.SensorError, PlatformImageFile>`) — there is no live-frame/continuous-scan abstraction anywhere in the codebase yet (`AndroidCameraProvider` uses CameraX, `IOSCameraProvider` uses CameraK/AVFoundation, JVM/WASM are `NoOpCameraProvider` with `isAvailable = false`). **This means the decode side of camera-qr-export needs a net-new "live scan" platform abstraction** — it cannot reuse `CameraProvider` as-is. This is an architecture/planning concern, but it has a direct UX consequence: JVM desktop's existing camera story is entirely no-op (`isAvailable` false), so desktop "Import via camera" either needs `WebcamCameraProvider` wired to a live preview, or desktop must be positioned as *encode-only* in the UI (a real UX/scope decision Phase 3 needs to make explicitly, since requirements.md asks for all 6 send/receive combinations).
- Accessibility idiom already in use: `Modifier.semantics { contentDescription = "..." }` on custom-drawn canvases (e.g. `AnnotationEditorScreen.kt:892`, `"Annotation canvas — tap to place points"`) — the QR display surface and the camera viewfinder surface should follow the same pattern with live status text, not just a static label.

## 1. Comparable UX patterns — air-gapped hardware wallets

Animated/multi-part QR transfer is a solved problem in the Bitcoin hardware-wallet world via the **Blockchain Commons UR (Uniform Resources) / BC-UR v2** standard, built on Luby-transform **fountain codes** — the same problem shape as camera-qr-export (large payload, small QR capacity, tolerant of missed/out-of-order frames). This is the closest prior art available and should be treated as a reference implementation for both the wire format and the UX:

- **Sparrow Wallet ↔ Specter-DIY / SeedSigner / Keystone / ColdCard**: the receiving side shows a **fill-up progress bar** that advances only on *successfully decoded new* frames — not on frames merely "seen." Users are told directly: if the bar stalls, change **angle or distance** relative to the screen until scanning resumes. This is the "actionable stall" pattern: don't just show 40%, tell the user what physical action un-sticks it.
- **Known failure mode** (from a real Sparrow GitHub issue, #1814/#139): the progress bar can loop indefinitely without completing when frame decode fails to converge — this is a *silent stall*, the worst failure mode, because the UI looks like it's working (bar is filling) when it's actually cycling the same subset of frames forever without acquiring the missing ones. **camera-qr-export must design against this explicitly** (see §4 — needs a distinct "stalled, not just slow" state, detected via a timeout with no new-fragment progress, not just elapsed time).
- **Keystone**: touchscreen-driven, positioned as the more "intuitive" of the touchscreen wallets — device shows QR, phone scans it, decodes, and a **confirmation popup summarizes what was decoded** before final action (broadcast, in their case). This "human-readable confirmation of the payload before committing" step maps directly to camera-qr-export's page-name-collision / import-preview step.
- **ColdCard**: notably **does not use camera-QR at all for its primary flow** — it uses a microSD card. Reviews consistently describe it as "clunkier" and "less intuitive" than the camera-QR flow. This is a useful contrast: it validates that camera-QR is the more approachable/consumer-friendly of the two air-gap transfer mechanisms, which supports positioning it as the more "no-technical-savvy-required" option in SteleKit rather than treating it as niche/nerdy.
- **Frame pacing**: none of the wallets researched expose a raw "frames per second" slider to end users; encoding density and frame rate are typically fixed/tuned by the implementer, not user-configurable. This suggests camera-qr-export should **not** surface an FPS knob as primary UI (it's an accessibility/reliability lever, not a feature) — default to a rate proven safe (see §3) and only expose "faster/slower" as a secondary affordance if scan reliability data says users need it.

## 2. User mental models & expectations

What a user is likely to assume "Export via camera" / "Import via camera" means, based on the name alone and on adjacent product metaphors (AirDrop, "scan to pay," QR boarding passes):

- **Assumption risk — "it's a photo/screenshot"**: users' most common QR interaction is a single static code (a boarding pass, a Wi-Fi login, a payment request) scanned once. The animated, continuous nature is *not* implied by "camera export" — the UI must state up front, before the flashing starts, that this is a **continuous live scan, not a single photo**, and that both screens must stay **visible to each other and still** for the duration (this is the single most important expectation-setting the flow must do, and it should happen once, before the frames start flashing, not buried in help text).
- **Assumption risk — instantaneous, like AirDrop**: users familiar with AirDrop/Nearby Share expect near-instant, fire-and-forget transfer. camera-qr-export's ~30-60s target for a ~2KB page is an order of magnitude slower and requires *sustained attention* (holding a phone steady, pointed at a screen) rather than a passive wait. The UI needs to set this expectation ("Keep both screens visible for about 30-60 seconds") rather than let the user assume it's instant and get confused/frustrated when it isn't done in 2 seconds.
- **Assumption risk — "any QR code will do"**: because the receiving device is doing live continuous scanning, a user might believe they can point it at *any* QR code (e.g., a random poster) and get useful results, or conversely worry that scanning the wrong code is dangerous. The UI must clearly signal **what family of code is expected** and cleanly reject unrelated QR content (see §4) without alarming the user ("this isn't a SteleKit transfer code" rather than a scary generic error).
- **Positive mental model to reinforce (this is the differentiator)**: users choosing this feature over "just email yourself a file" or "use a USB cable" are explicitly choosing air-gap. The mental model to *cultivate*, not just avoid confusing, is: **"nothing left this room / no network was used."** This should be stated as a positive claim in the UI (see §5), not just an implicit property.
- **Directional confusion risk**: with 3 platforms × 2 roles = 6 combinations, users may not immediately know "which device should be the sender." The entry points ("Export via camera" on the page they want to send, "Import via camera" from wherever new pages land) are reasonably self-describing, but first-run copy should briefly confirm "this device is sending" / "this device is receiving" so a user doesn't start both devices in scan mode facing each other (a plausible, harmless-but-confusing user error).

## 3. Accessibility

**Flash/strobe safety (WCAG 2.3.1, "Three Flashes or Below Threshold", Level A)**:
- The general rule: content must not flash more than **3 times per second**, unless each flash is below the *general flash threshold* and *red flash threshold*, or the flashing area is small (≤ 25% of any 10°-visual-angle patch, i.e. ≤ 0.006 steradians at typical viewing distance).
- A full-screen animated QR sequence, as scoped ("full-screen animated QR frame sequence"), is **exactly the failure case WCAG 2.3.1 exists for** if driven above 3 Hz: it is large-area (fails the small-area exemption) and full-screen QR frame changes are high-contrast black/white transitions, which is worst-case for general flash risk. Practically:
  - **Frame rate must be capped at ≤ 3 frames/second** to categorically satisfy 2.3.1 by rate alone, without needing to run PEAT (Photosensitive Epilepsy Analysis Tool) contrast/area analysis. This is the simplest, most defensible engineering target — treat 3 fps as a hard ceiling, not a soft default, given the target audience (general consumer note-taking app, not a specialist tool where users have opted into higher risk).
  - If product/perf work later wants a materially higher frame rate for transfer speed, it must be validated with PEAT (the W3C/Trace-Center-recommended tool) rather than assumed safe — do not increase past 3 Hz without that validation step in the plan.
  - Consider **not using true full-screen high-contrast B/W** for the QR display chrome — e.g., inset the QR in a bordered card on a static background rather than the QR module pattern occupying literally 100% of the display — this both reduces the flashing area (helps the area-based exemption as a second line of defense) and gives room for a persistent status line (frame counter, "keep steady" reminder) that doesn't itself flash.
  - Provide a **"reduce motion / accessible transfer mode"** that trades speed for safety: lower frame rate further (e.g., 1-2 fps) plus a manual "next frame" advance button (tap-to-advance) as a fully user-paced alternative — this simultaneously serves photosensitivity concerns *and* motor/switch-access users who can't reliably time a camera against an auto-advancing display.
- **Low vision / camera alignment**: aligning a small phone camera to a small on-screen code is a fine-motor + visual-acuity task that will be hard for low-vision users. Mitigations, in priority order:
  1. Make the QR **as large as safely possible** on the sending screen (maximize the target size to ease aiming) — direct tension with keeping flash area small for 2.3.1, so this needs explicit tradeoff documentation in the plan, not left implicit.
  2. On the receiving/scanning side, provide **audio and/or haptic feedback** on each successfully-decoded new fragment (a soft tick/vibration), so a user who can't clearly see the on-screen progress bar still gets non-visual confirmation that alignment is working — this is the single highest-leverage accessibility addition since it turns a visual-only feedback loop into a multi-modal one.
  3. A **bounding-box/reticle overlay** with `contentDescription`/semantics text like "Point camera at the SteleKit transfer code" (consistent with the existing `"Annotation canvas — tap to place points"` semantics idiom in `AnnotationEditorScreen.kt`) helps both low-vision users (with TalkBack/VoiceOver reading it) and everyone else.
  4. Avoid relying on color alone for the "locked on" vs. "searching" scanning state — use shape/icon + text, not just a green/red border.
- **Keyboard/switch-access navigation**: the entry points ("Export via camera" / "Import via camera" menu items or buttons), the permission dialog, and any "Cancel transfer" / "Done" controls must be reachable via standard focus order and activatable without a pointer — this is standard Compose semantics hygiene, consistent with existing dialogs in the codebase (`AlertDialog` usage is already keyboard-accessible by default in Compose Material3; camera-qr-export must not regress this by building custom non-`AlertDialog` chrome for its dialogs without equivalent focus/traversal support).

## 4. Error states & edge cases

| Case | Bad pattern to avoid | Recommended UX |
|---|---|---|
| **Partial/stalled transfer** | A progress bar that keeps "moving" by re-showing already-seen fragments (the observed Sparrow bug) gives false confidence. | Track *distinct new fragments per unit time*; if zero new fragments arrive for e.g. 8-10s, switch from "in progress" to an explicit **"Not receiving new data — try moving closer / adjusting angle"** state, mirroring the Sparrow guidance almost verbatim (it's proven end-user-tested language). Never let the bar visually animate without genuine new-data progress behind it. |
| **Wrong/unrelated QR code scanned** | Generic "Invalid QR code" error reads as broken/scary. | Distinguish "not a SteleKit code" (calm, informational: "That's not a SteleKit transfer code") from "SteleKit code but wrong session/page" if that's structurally detectable — keep tone neutral, this is a common harmless mis-scan, not a security event. |
| **Permission denied (camera)** | Dead-ending the whole feature. | Reuse `CameraPermissionRationaleDialog` shape exactly: rationale text specific to *why this feature* needs the camera ("SteleKit needs camera access to scan the transfer code from the other device"), "Not now" fallback, `isPermanentlyDenied` → "Open Settings" variant. |
| **Ambient light too low** | Scanner silently fails to acquire any frames with no explanation. | Live camera-quality heuristic (CameraX/AVFoundation exposure metrics) drives a direct hint: **"Too dark to scan — try a brighter room or increase the other screen's brightness."** This is more actionable than a generic "can't find code." |
| **Page-name collision on import** | Silent overwrite (data loss) or silent duplicate-with-suffix (surprising). | Explicit choice dialog at decode-complete time — consistent with existing confirm patterns (`CapturePreviewDialog`'s Save/Discard shape): "A page named '<name>' already exists. [Keep both] [Overwrite] [Cancel import]" — never default to silent overwrite. |
| **Backgrounding mid-transfer** | Transfer state silently lost, user has to restart from 0% with no acknowledgment of the interruption. | Since fountain codes are resumable by nature (any new fragment helps, regardless of session continuity, as long as encode-side keeps looping the same page), the receiving state (fragments-so-far) should **survive** app backgrounding/foregrounding within the same in-process session at minimum, and the UI should say "Paused — reopen camera to continue" rather than silently resetting. Encode side should keep cycling frames indefinitely (or until explicitly stopped) precisely so a backgrounded-then-resumed receiver can pick back up. |
| **Communicating "% done" for a non-linear fountain scheme** | A literal linear progress bar misrepresents fountain codes — decode probability rises non-linearly and can appear "stuck near 100%" for a stretch before rapidly resolving (the classic "coupon collector" tail). | Do **not** present raw fragment-count-received as a linear 0-100% bar without qualification — it will look broken as it stalls near the end. Options, from simplest to richest: (a) show "Receiving… (N unique fragments)" as a count rather than a percentage, avoiding the false-precision of a % that isn't linear; (b) if a percentage is used, base it on decode-probability estimate, not raw fragment count, and label it approximate ("~"); (c) borrow the hardware-wallet pattern directly — a bar that only advances on genuinely new information and stalls visibly (not smoothly) when no new fragments arrive, so the *shape* of the stall itself communicates "fountain code, not deterministic countdown" without user education being required. |

## 5. Jobs-to-be-done

- **Functional job**: "Get this page (or set of pages) onto my other device, right now, with nothing else available (no shared Wi-Fi, no cloud account, no cable)." The job is fundamentally about **removing dependency on infrastructure** — the feature should be discoverable and usable precisely in the moment when other options have already failed or aren't wanted, so entry points should not be buried (e.g., available directly from a page's share/export menu, not nested under a settings submenu).
- **Emotional job — trust/control, air-gap as a feature not a workaround**: the requirements explicitly flag this ("the air-gap is a feature, not just a constraint"). The UI should **visibly assert** the no-network property rather than leave it implicit — e.g., persistent copy during the transfer like "No internet connection used" / a small offline/air-gap icon, possibly paired with a literal check that airplane-mode-or-no-network is compatible with the feature (if technically verifiable, even a passive "this works with Wi-Fi off" note reassures the security/privacy-motivated user this is designed for). This is the single biggest UX-differentiation opportunity versus competitors who'd just say "use email" — name the property being protected, don't make the user infer it.
- **Social job**: performing an offline transfer in front of another person (another Logseq/note-taking enthusiast, or a colleague) is inherently a bit of a "trick" — two screens flashing at each other, resolving into a perfectly transferred note, is visually impressive precisel* because it's legible as "look, no network." Small UX choices that serve this social/demo job: a satisfying, visible completion state (not just a dry "Import successful" toast — a moment of confirmation, e.g. brief success animation/haptic, that rewards the "trick" being watched), and keeping the flashing-QR visual itself reasonably attractive/deliberate-looking (not looking like an error state) since it will often be performed *for* an audience of one.

## Summary of concrete UX recommendations for Phase 3 (plan.md) to carry forward

1. Reuse `CameraPermissionRationaleDialog`'s exact shape/tone for the camera-qr-export decode-side permission ask; add feature-specific rationale copy only.
2. Cap animated QR frame rate at ≤ 3 fps by default to satisfy WCAG 2.3.1 without requiring PEAT validation; treat any higher rate as requiring explicit accessibility sign-off.
3. Provide a manual tap-to-advance / reduced-motion transfer mode as a first-class alternative, not an afterthought.
4. Add audio/haptic tick feedback on each new fragment decoded, independent of the visual progress indicator, for low-vision users.
5. Design the receive-progress indicator as fragment-count-based or explicitly-approximate, never a naive linear percentage; make "stalled vs. slow" a distinct, explicitly-communicated state with actionable guidance ("move closer / change angle / brighten the screen"), following the Sparrow/BC-UR precedent almost verbatim.
6. Handle page-name collisions with an explicit modal choice (Keep both / Overwrite / Cancel), reusing the Save/Discard button-disable-during-work pattern from `CapturePreviewDialog`.
7. State up front, before frames start flashing, that this is a continuous live scan (not a photo) requiring ~30-60s of both screens staying visible to each other — set this expectation once, prominently, not in buried help text.
8. Make the "no network was used" property an explicit, persistent visual claim during transfer, not an implicit side effect — this is the emotional/social differentiator the requirements call out.
9. Flag to Phase 3/architecture: `CameraProvider` is single-shot-capture only today; a live-scan abstraction is net-new work, and JVM desktop's camera story (`NoOpCameraProvider`, `isAvailable = false`) needs an explicit scope decision — either wire `WebcamCameraProvider` to live preview for desktop-as-receiver, or scope desktop as encode-only in this iteration.
