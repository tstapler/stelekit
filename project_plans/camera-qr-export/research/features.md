# Feature Landscape Research: camera-qr-export

Agent 2 (Features) — Phase 2 research, SDD workflow.

## 1. Existing patterns in this codebase worth reusing

### `CameraProvider` — critical gap: single-shot capture only, no live frame stream

`kmp/src/commonMain/kotlin/dev/stapler/stelekit/platform/sensor/CameraProvider.kt` exposes exactly one method:

```kotlin
interface CameraProvider {
    val isAvailable: Boolean
    suspend fun capturePhoto(): Either<DomainError.SensorError, PlatformImageFile>
}
```

This is a **one-shot shutter abstraction** (used today for `ImageImportService`/annotation photo capture). It has no notion of a live preview surface, a continuous frame stream, or per-frame analysis (no `ImageAnalysis`-style callback, no `Flow<Frame>`). The requirements doc's constraint — "must reuse existing cross-platform `CameraProvider` abstraction ... Android/iOS/JVM/WASM implementations already exist" — is **only partially true**: the interface exists and the per-platform camera bootstrapping/permission code exists (`AndroidCameraProvider` uses CameraX, `IOSCameraProvider` uses CameraK/AVFoundation, `WebcamCameraProvider` on JVM, `WebCameraProvider` on WASM), but the *decode* side of camera-qr-export needs a live continuous-frame API this interface does not provide today. Confirmed via direct read of `AndroidCameraProvider.kt` (only `capturePhoto()`), `WebcamCameraProvider.kt` (same), and the shared interface.

**Implication for planning**: this feature requires either (a) extending `CameraProvider` with a new `observeFrames(): Flow<PlatformImageFrame>`-style method per platform (CameraX `ImageAnalysis.Analyzer`, AVFoundation `AVCaptureVideoDataOutput`, JVM webcam frame-grab loop, WASM `getUserMedia` + canvas frame sampling), or (b) a parallel, purpose-built `QrScanProvider` abstraction that reuses the *permission/availability* plumbing but not the `capturePhoto()` contract itself. Either way this is a new platform abstraction, not a drop-in reuse — flag this explicitly in the architecture/plan phase since it affects the "Large 3-6 week" appetite and touches all 4 platform source sets.

### `ImageImportService` — pipeline shape and transactional-write ordering to imitate

`kmp/src/commonMain/kotlin/dev/stapler/stelekit/db/ImageImportService.kt` is the closest existing analog to "receive external content → integrate into a page/graph." Reusable patterns:
- Numbered pipeline with an explicit comment stating the invariant order (reserve path → copy bytes → build domain object → write sidecar **before** DB row → save to repository → create block → optional journal insert), with a one-line justification of *why* the order matters ("If the sidecar write fails, the DB row is never written"). The decoder's markdown-reconstruction pipeline should have an equivalently explicit invariant order (e.g. validate full reassembly checksum → parse via `OutlinerPipeline` → only then offer page-name-collision resolution UI → only then write via `GraphWriter`/repository), and it should be commented the same way.
- `Either<DomainError, T>` return type end-to-end, consistent with the rest of the repo's error-handling convention (see CLAUDE.md's Arrow rules) — the decode/reassembly result should follow the same `Either` idiom, e.g. a `DomainError.TransferError` sealed hierarchy analogous to `DomainError.SensorError`.
- Constructor takes optional collaborators (`journalService`, `writeActor`, `measurementAnnotationRepository` all nullable with defaults) — same shape recommended for a new `QrTransferService`/`QrImportService` so it's easy to unit test in isolation without a live camera.

### `AnnotationEditorViewModel` — ViewModel lifecycle / scope-ownership pattern (mandatory)

Owns its own `CoroutineScope(SupervisorJob() + Dispatchers.Default)` internally and exposes an explicit `close()` for cleanup — this is the exact pattern CLAUDE.md mandates for any long-lived class instantiated via `remember {}`. A `QrEncodeViewModel` (send side, drives the animated-frame timer) and `QrDecodeViewModel` (receive side, owns the live scan collector) must follow this same rule: own their own scope, never accept `rememberCoroutineScope()`, expose `close()`/`stop()` to cancel in-flight frame timers or camera collectors when the screen is dismissed mid-transfer. This directly matters for the "transfer cancellation mid-stream" edge case in the requirements — the scope-cancellation contract is exactly the mechanism that should back "Cancel" cleanly.

Also worth reusing: the `StateFlow`-based state pattern (`AnnotationEditorState` immutable data class + `MutableStateFlow`/`asStateFlow()`), and the "UI-facing sealed state enum mirroring a platform-download/readiness state machine" pattern used for `DepthModelUiState` (`Absent` / `Downloading(progress)` / `Ready` / `Failed`). This is a strong template for both:
- **Encoder-side UI state**: `Idle` / `Encoding` / `Displaying(frameIndex, totalFrames)` / `Paused` / `Complete` / `Cancelled`.
- **Decoder-side UI state**: `Idle` / `Scanning(framesReceived, framesExpected)` / `Reassembling` / `Success(page)` / `Failed(reason)`.

### `CapturePreviewDialog` — confirm-before-commit UX pattern to reuse for import collision

Full-screen `Dialog` with `DialogProperties(usePlatformDefaultWidth = false)`, a preview of the captured artifact, `Save`/`Discard` buttons, and an `isImporting: Boolean` flag that disables the Save button and swaps it for a `CircularProgressIndicator` while the async write is in flight (dialog stays open so the user knows work is happening, and `onDismissRequest` is guarded by `!isImporting` so a stray tap can't abandon an in-progress write). This is the right template for the decoder's **"page already exists — overwrite / keep both / cancel" confirmation dialog** (an unstated need called out in the assignment) and for the general "importing… please wait" spinner state during final markdown parse + write.

### `OutlinerPipeline` and `GraphWriter`/`GraphLoader`

`OutlinerPipeline` is confirmed as the existing markdown → block-tree parser that the decoded payload must ultimately be run through (per requirements: "parsed via existing outliner pipeline") — reuse it as-is rather than writing a parallel parser. `GraphWriter`/`GraphLoader` constructors take `fileSystem`, repositories, and an optional `writeActor`/callback — the same DI shape should back a new `QrImportService` so it composes with `DatabaseWriteActor` per the mandatory write-gating rule (`@DirectSqlWrite`/`RestrictedDatabaseQueries`) rather than writing directly to `SteleDatabaseQueries`.

### No existing QR/barcode dependency

`kmp/build.gradle.kts` has no ZXing, no `com.google.mlkit:barcode-scanning`, no QR generation library today. ML Kit is already a dependency for other features (`image-labeling`, `text-recognition`, `genai-prompt` — Android-only, `androidMain` source set), which is a data point for platform-parity risk: ML Kit's barcode module would be free to add on Android but has no equivalent on iOS/JVM/WASM, so a cross-platform decode library (or per-platform native barcode APIs behind a common interface) will be needed regardless. No feature-flag framework was found in a repo-wide grep beyond ad hoc `Settings`-backed toggles (see `TagSettings.kt` using `com.russhwolf:multiplatform-settings`) — the "feature-flagged, opt-in" requirement should follow that same `Settings`-backed boolean toggle pattern rather than introducing a new flagging system.

## 2. Industry prior art

### Direct precedent: Blockchain Commons UR + animated QR (named in requirements, confirmed via research)
- Uses **Luby Transform (fountain) codes** — rateless erasure coding where the receiver can begin at *any* frame and reconstruct from *any* sufficiently large subset of frames, tolerating drops/misses without requiring a specific frame to be re-shown. This is the direct technical precedent for the "tolerant of missed/out-of-order frames" requirement.
- `FountainEncoder`/`FountainDecoder` (Swift, in URKit) is a reference implementation shape: encode → sequence of mixed/pure fragments → decoder XORs incoming fragments against previously solved ones until the full message resolves. Worth pulling this open-source reference during planning/architecture phase for the actual encoding algorithm rather than reinventing fountain codes from scratch.
- URs encode binary data as CBOR wrapped in a well-formed URI text scheme, which is then what's rendered into each QR frame — a reusable idea for framing our own chunk header (sequence id, total count, payload, checksum) into a compact wire format before QR encoding.
- Source: https://developer.blockchaincommons.com/animated-qrs/, https://github.com/BlockchainCommons/URKit, https://divan.dev/posts/fountaincodes/

### Adjacent open-source "sneakernet via QR" tools (general file transfer, not fountain-coded)
- `cpqr` (github.com/jmalenko/cpqr) — display sequence of QR frames on source, target device (typically phone) records/decodes, reconstructs file. Simple sequential (non-fountain) approach — useful as a "simplest possible baseline" reference, and a cautionary example of what fountain coding avoids (strict frame-by-frame completeness requirement, brittle to drops).
- `qrxfer` (github.com/leonjza/qrxfer), `qrcode-file-transfer` (ganlvtech), `airgapped-qr-code-transfer` (mohankumarelec, Vue.js + pako compression + qrcode.js + zbar-wasm decode) — all confirm the same architecture shape: chunk → compress → per-chunk QR → animate at a fixed interval → camera/webcam scan loop → dedup by chunk index → reassemble. `airgapped-qr-code-transfer`'s use of `pako` (zlib) compression pre-chunking is worth adopting: compressing markdown text before chunking measurably reduces frame count for a ~2KB page.
- None of these found tools implement fountain/rateless coding — they are all sequential-with-retransmission, meaning SteleKit's fountain-coded approach (as scoped in requirements) is a meaningfully more robust design point than the median open-source prior art, not merely matching it.

### Hardware wallet UX precedent (Keystone / ColdCard / Sparrow / Specter)
- Confirms the general workflow pattern (device shows animated QR → other device's camera captures) is a shipped, well-understood UX in a different high-stakes domain (signing transactions). Detailed UX telemetry (glare handling, progress-bar specifics) was not surfaced in search results — treat this as a research gap; recommend a follow-up targeted look at COLDCARD's and Sparrow's actual scan-screen implementation/source if the architecture phase wants concrete progress-UI reference designs, rather than relying on this pass's search results alone.
- One incidental finding: COLDCARD's newer "Q" hardware uses a physical scanner with an active illumination indicator (strobe) so the user knows where the camera is aimed — a physical-hardware-specific affordance not transferable to phone/webcam cameras, but the *underlying need* (visual confirmation of scan alignment, e.g. an on-screen viewfinder reticle/bounding box overlay on detected QR position) is directly transferable UX guidance for the decoder screen.

### No Logseq/Obsidian community precedent found
Targeted search turned up no existing discussion, plugin, or tool for QR-based note transfer in the Logseq or Obsidian ecosystems. This appears to be a genuinely novel feature for the note-taking app category — there is no existing user mental model or competing UX convention to match or diverge from. This raises the discoverability risk noted in section 4 below (nothing to "look like" for users coming from other apps) but also means there's no incumbent UX to be inconsistent with.

### Library/tooling landscape for implementation
- ZXing: fully open source, no ML model assets, deterministic decode, full user-owned camera pipeline (you control autofocus/UI) — good fit for a from-scratch cross-platform (Android/iOS/JVM/WASM) implementation where consistency across platforms matters more than best-in-class decode speed. Currently in maintenance mode (bug fixes only, no new features) — stable but not actively evolving.
- ML Kit barcode scanning: faster / more robust to real-world conditions (motion blur, angle) out of the box, but Android-only in this codebase's current dependency footprint (no ML Kit on iOS/JVM/WASM) — would require a second decode library for other platforms, undermining cross-platform consistency. Given ML Kit is already a build dependency for other Android features, there may be temptation to reach for it on Android only and ZXing elsewhere — flag this as a "why-not-consistent-libraries-everywhere" risk for the architecture/ADR phase, since asymmetric decode robustness across platforms directly contradicts the "6 send/receive combos to validate" rabbit hole named in requirements.

## 3. Edge cases and failure modes to design for

Beyond what's already named in the requirements' "Rabbit Holes" section, concrete cases surfaced by comparing to prior art and the codebase's existing resilience patterns:

- **Partial/stalled scans**: decoder must show live progress (frames captured / frames needed, fountain codes make "needed" probabilistic not exact — surface as a percentage/confidence estimate, not a hard fraction) so the user knows whether to keep holding the camera steady, per the `DepthModelUiState`-style progress pattern above.
- **User moves phone away mid-transfer**: decoder must persist partially-reassembled fountain-decode state across transient loss of any single frame (fountain codes tolerate this by design) but must also detect prolonged total signal loss (e.g. no new frame captured in N seconds) and surface a distinct "lost signal — realign camera" state rather than silently hanging.
- **Ambient light changes / glare / motion blur**: camera decode libraries (ZXing/ML Kit) already handle some of this, but the encoder side has a lever too — QR error-correction level (L/M/Q/H) and frame display duration are tunable; higher EC level trades payload density for resilience, directly interacting with the "payload density vs. frame count trade-off" rabbit hole.
- **QR frame rate vs. camera frame rate mismatch**: if the sender cycles frames faster than the receiver's camera can capture+decode a frame, frames are silently skipped — fountain coding tolerates missed frames by design, but the *display duration per frame* still needs a floor tuned empirically per the requirements' own risk callout ("empirical"). Recommend the sender expose a configurable/adaptive display interval, defaulting conservatively (e.g. slower on first release) rather than hard-coding a rate optimized only for the primary dev's test devices.
- **Duplicate/replay frames**: fountain decoders must treat repeated frames as expected/idempotent (looping display is *how* frames get resent) — dedup by encoded fragment identity, not by simple frame count, since the encoder loops indefinitely until the user stops it.
- **Content exceeding target frame count** (page too large for "~2KB in 30-60s" quality bar): needs an explicit UX signal *before* starting transfer — e.g. estimated payload size / frame count / time shown on the encoder screen before the user commits to holding two devices up to each other, so oversized pages fail fast with guidance ("try exporting to git sync instead") rather than silently taking minutes.
- **Transfer cancellation mid-stream**: both encoder and decoder need a clean `Cancel` control that tears down the camera session / frame-display timer without corrupting state — directly ties to the mandatory `CoroutineScope` ownership/`close()` pattern above.
- **Concurrent multi-page sessions**: requirements scope allows multiple pages per session; decoder needs a session/transfer identifier baked into the chunk header (not just chunk index) so it can distinguish "new session started" from "more frames of the same session," and should reject/warn on receiving frames from a second concurrent sender rather than silently interleaving two sessions' fragments.
- **Reassembled markdown that fails to parse**: `OutlinerPipeline` parse failure after a "successful" fountain-decode (payload arrived intact but is not valid markdown, e.g. sender bug, encoding mismatch, or truncated payload that happened to pass a checksum on a bug) needs a distinct terminal failure state ("received data but could not parse as a page") distinguishable from "still scanning" — do not conflate reassembly-complete with import-succeeded.
- **Checksum/integrity validation**: given "lossless markdown reconstruction" is the hard gate (per Success Metrics), the chunk header must carry a strong per-page checksum (not just per-fragment CRC used internally by the fountain code) validated *before* the reassembled markdown is ever handed to `OutlinerPipeline` or `GraphWriter` — mirrors the `ImageImportService` principle of validating/writing the source-of-truth artifact before touching the DB.

## 4. Unstated user needs

- **Visual progress feedback during transfer** — explicitly expected by the task description and directly supported by reusable state-machine patterns already in the codebase (`DepthModelUiState`). Users transferring a page by holding two phones up to each other for up to a minute need continuous confirmation that *something is happening* (percentage/frame count, not just a spinner) — a bare spinner risks users assuming the feature is broken partway through a legitimate multi-frame decode.
- **Pause/resume** — not in explicit requirements scope, but fountain coding's design (any-frame-restart) makes this nearly free architecturally: pausing is just "stop displaying/stop scanning," resuming just re-enters the same session without losing already-decoded fragments. Recommend surfacing this as a real button even though requirements didn't ask for it explicitly, since the underlying protocol gives it away for free and users will expect a "pause" affordance any time a modal, multi-second, camera-driven flow exists (matches the `CapturePreviewDialog` precedent of never leaving the user stuck mid-operation with no control).
- **Confirmation before overwriting an existing page of the same name on import** — explicitly named in the assignment as a likely unstated need; directly analogous to the `CapturePreviewDialog` `Save`/`Discard` confirm-before-commit pattern, and should follow the same "importing… disable controls, keep dialog open" pattern while the write actually happens through `GraphWriter`.
- **Discoverability** — since no comparable feature exists in Logseq/Obsidian (section 2), and the requirement is "feature-flagged, opt-in," there is a real risk this ships and nobody finds it. The existing app has precedent for feature entry points via explicit UI affordances (e.g. camera capture button in the annotation flow) — recommend the plan phase explicitly design an entry point (e.g. a "Send via QR" action in the page context menu, discoverable the same way page export/share actions already are) rather than only a hidden settings toggle, and consider a one-time contextual hint/tooltip the first time a user opens a graph on a device with no configured sync method (a proxy signal that they may need an alternative transfer path).
- **Trust/verification signal for a "zero-trust, no-network" feature** — the requirements frame this as air-gapped/zero-trust; users adopting it for that reason may want an explicit visual/textual confirmation of what specifically is about to be transmitted (e.g. a pre-flight summary: page title, size, block count) before broadcasting it as an optically-readable QR stream on screen, since anyone with line-of-sight to the screen (not just the intended receiving device) can capture the data. This is a soft privacy expectation not stated in requirements but implied by the "air-gapped" framing.
- **Recovery from an abandoned partial transfer** — if a user cancels or the app is killed mid-scan, is there any expectation that a next attempt resumes rather than restarts from zero? Given no persistence of partial fountain-decode state is mentioned in requirements/scope, recommend explicitly deciding (and documenting) whether partial progress survives app backgrounding/kill, since users may assume "resume" persists further than it actually will.
