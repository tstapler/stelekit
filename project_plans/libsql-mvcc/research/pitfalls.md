# Pitfalls: Rust JNI + libsql Failure Modes

Research for the `libsql-mvcc` feature. Covers the six critical risk areas identified in the
requirements.

---

## 1. Rust JNI Panics: `panic = "abort"` vs Unwind

### What happens without `panic = "abort"`

If a Rust function defined as `extern "C"` panics with the default `unwind` panic strategy,
the unwind crosses an FFI boundary into the JVM. The Rust Reference and the Rustonomicon both
classify this as **undefined behavior**: the C++ unwind ABI is not guaranteed to be the same as
Java's, and DWARF/exception-table entries generated by the Rust compiler are not understood by
the HotSpot JIT. In practice this manifests as:

- A native crash dump (SIGSEGV or similar) that the JVM cannot catch as a Java exception.
- Silent heap corruption if the JVM stack frame holding local JNI references is left in an
  inconsistent state before the unwind escapes.
- A `FatalError: hs_err_pid` file rather than a Java `RuntimeException`.

### Effect of `panic = "abort"`

Setting `panic = "abort"` in the `[profile.release]` (and `[profile.dev]`) section of
`Cargo.toml` **does prevent undefined behavior** from a panicking unwind, but at the cost of
immediately calling `abort(3)`. On the JVM that surfaces as:

- The process dies with `SIGABRT`. Android: the app process is killed; desktop: daemon exits.
- **No `JNIException` is thrown.** No `catchDbError()` or Kotlin `try/catch` block can catch
  a native abort. The coroutine running the query is not resumed — the entire JVM terminates.

### Recommended mitigation

Do **not** rely on `panic = "abort"` as the sole safety net. Every `extern "C" fn` entry point
must wrap its body in `std::panic::catch_unwind(|| { ... })` and translate the caught panic into
a JNI exception throw:

```rust
#[no_mangle]
pub extern "C" fn Java_..._nativeQuery(env: JNIEnv, ...) -> jlong {
    let result = std::panic::catch_unwind(std::panic::AssertUnwindSafe(|| {
        // actual logic
    }));
    match result {
        Ok(v) => v,
        Err(_) => {
            let _ = env.throw_new("java/lang/RuntimeException", "Rust panic in JNI call");
            -1  // sentinel error value
        }
    }
}
```

`catch_unwind` only works when the panic strategy is `unwind`. With `panic = "abort"` it is a
no-op (panics immediately abort before the closure can return). The recommended approach for
a cdylib consumed by the JVM is therefore:

1. Keep `panic = "unwind"` (the default).
2. Wrap every public JNI entry point in `catch_unwind`.
3. On the caught panic, throw a Java `RuntimeException` via `env.throw_new(...)`.
4. The Kotlin call site receives a thrown exception, which surfaces as an `IOException` or
   propagates to `catchDbError()` in the normal flow.

The `jni` crate (0.21+) provides `JNIEnv::catch_unwind` helper and `JavaException` as
alternatives to the manual pattern above.

## 2. Box-as-Handle Memory Safety: Double-Free and Use-After-Free

### The pattern

```rust
// Allocate: return handle to Kotlin
let handle: jlong = Box::into_raw(Box::new(MyStruct { ... })) as jlong;

// Use: cast back to reference
let s: &mut MyStruct = unsafe { &mut *(handle as *mut MyStruct) };

// Finalize: reclaim ownership and drop
let _ = unsafe { Box::from_raw(handle as *mut MyStruct) };
```

### Double-free risk

Once `Box::from_raw` is called, Rust drops the allocation. If `finalizeStatement` (or any other
handle-consuming JNI function) is called a second time with the same `jlong` value:

- The pointer is dangling. `Box::from_raw` reinterprets the old address as a live `Box` and
  drops it again.
- This is **undefined behavior**: the allocator may corrupt its free-list, producing heap
  corruption that surfaces as a crash or silent data corruption elsewhere.
- Rust's ownership system **does not protect raw pointers**; the compiler cannot see across the
  JNI boundary to detect the second call.

### Use-after-free risk

Kotlin retains a `jlong` handle value in a field. If `close()` drops the object while another
thread holds the same handle value and is mid-query:

- The query thread dereferences a freed allocation.
- Again UB — the allocation may have been reused by a different `Box::new` call.

### Mitigations

1. **Null-sentinel after free.** After `Box::from_raw`, zero the field on the Kotlin side so any
   subsequent call sees `0L` and throws instead of dereferencing. Enforce this via a private
   `AtomicLong` in `LibsqlJni.kt` that is CAS'd from handle → 0 before being passed to the
   native finalizer.

2. **`AtomicBool` closed flag on the Rust side.** Each handle struct carries a flag set by the
   finalizer. Any other JNI entry that receives the same handle checks the flag before
   dereferencing. If set, throw `IllegalStateException` immediately.

3. **Connection pool discipline.** The 8-connection `JvmLibsqlDriver` pool must ensure that
   `close()` is not called while a connection borrowed from the pool is still in use. A
   `ReentrantReadWriteLock` write-locked during close is the idiomatic JVM pattern.

4. **Separate finalization path.** Prefer explicit `AutoCloseable` / `use { }` over finalizers
   (`finalize()` / `Cleaner`) for resource release. JVM finalizers run on a GC thread with no
   ordering guarantee relative to other threads.

**References:**
- [Understanding Box and Raw Pointers in Rust](https://knowledge.buka.sh/understanding-box-and-raw-pointers-in-rust-a-guide-to-safe-and-unsafe-memory-access/)
- [jni crate docs — JNIEnv](https://docs.rs/jni/latest/jni/struct.JNIEnv.html)

## 3. Does libsql Local Mode Support `BEGIN CONCURRENT`?

### Critical finding: local mode does NOT have a working MVCC implementation

This is the highest-risk finding for the libsql-mvcc feature.

**`BEGIN CONCURRENT` in the libsql C fork (`tursodatabase/libsql`) exists only as an
experimental, non-production-ready extension that is NOT available in the standard local
file-based mode offered by the `libsql` Rust crate.**

The architecture is:

| Mode | Description | BEGIN CONCURRENT |
|---|---|---|
| `Database::open_local(path)` | Pure local SQLite-compatible file | **No.** Single-writer WAL only. |
| Embedded replica (`open_with_local_replica`) | Local + remote sync via `sqld` | Partial / experimental. |
| Remote (`open_remote`) | HTTP to Turso cloud sqld | Handled server-side. |

Evidence:

- The `libsql` Rust crate documentation distinguishes the `local`, `remote`, and `embedded_replica`
  `Builder` modes. Concurrent write is not listed as a feature of the local mode builder.
- Multiple sources confirm: "libSQL is an open-source fork of SQLite that extends SQLite with
  features like embedded replicas and remote access, but **inherits SQLite's fundamental
  limitations such as the single-writer model**."
- The search result from the BetterStack Turso explainer specifically attributes concurrent
  write throughput to the **sqld server** (the remote/embedded-replica path), not the local
  embedded library.
- Turso's own 2025 blog posts describe a ground-up rewrite ("Turso Database") in Rust — the
  fact that they are rewriting from scratch is strong evidence that the C-fork libsql never
  achieved working local MVCC.
- The virtual WAL interface (`walx`) in the C fork is the plumbing that *could* support
  `BEGIN CONCURRENT`, but as of 2025–2026 it is used for replication, not local concurrent
  writes. Issue #1553 ("Use WAL as default everywhere") remains open.

### Impact on the feature

**The core premise of the libsql-mvcc plan — using `BEGIN CONCURRENT` in local embedded mode
to reduce write-actor tail latency — is not achievable with the `libsql` Rust crate's local
mode as it exists today.** Attempting to send `BEGIN CONCURRENT` to a local connection backed by
the standard SQLite WAL will likely:

- Be silently treated as a regular `BEGIN` (if the C fork ignores unknown syntax).
- Return an error (if the extension is not compiled in).
- Fall back to single-writer semantics in any case.

### Required design change

The implementation plan must choose one of:

1. **Use the sqld embedded-replica path** with a local sqld instance, accepting the complexity
   of running a separate server process (defeats the embedded goal).
2. **Use the Turso Rust rewrite** (`tursodatabase/turso` — in beta as of 2025), which does
   implement native concurrent writes. API compatibility with the `libsql` crate is not
   guaranteed.
3. **Reframe the goal**: do not use `BEGIN CONCURRENT`; instead use the Rust JNI bridge to
   drive the existing SQLite WAL with a custom connection pool that more aggressively interleaves
   reads and writes — without MVCC.
4. **Wait for Turso Database beta stabilization** before committing to this feature.

Option 3 is the lowest risk for the current milestone and preserves the architecture (Rust JNI
bridge, connection pool, Kotlin driver shim) without depending on an unstable C extension.

**References:**
- [libsql Rust crate docs](https://docs.rs/libsql)
- [tursodatabase/libsql GitHub](https://github.com/tursodatabase/libsql)
- [BetterStack — How Turso Eliminates SQLite's Single-Writer Bottleneck](https://betterstack.com/community/guides/databases/turso-explained/)
- [tursodatabase/libsql issue #1553](https://github.com/tursodatabase/libsql/issues/1553)
- [Turso Experimental Features docs](https://docs.turso.tech/sql-reference/experimental-features)

## 4. Bazel + `rules_rust` Cross-Compilation for Android (aarch64-linux-android)

### Known pitfalls (2024–2025)

#### 4a. Bzlmod cross-compilation is still incomplete

`rules_rust` issue #2518 ("Cross-compilation with Bzlmod not working") was open as of early
2024 and describes linker failures when targeting multiple architectures simultaneously under
Bzlmod. Issue #2529 reports `UnknownOperatingSystem` errors when parsing the
`aarch64-unknown-linux-gnu` target triple under Bzlmod. The cross-compile documentation for
`rules_rust` is listed as a tracked issue (#276) and not fully written.

Implication: the `MODULE.bazel` setup for `aarch64-linux-android` may require manual
`register_toolchains()` calls, custom platform constraints, and a `--platforms` flag at build
time. The hermetic Zig-linker path used by some `rules_rust` users does not apply cleanly to
Android because Android requires the NDK Clang toolchain.

#### 4b. `libgcc` replaced by `libunwind` in NDK r23+

Starting with Android NDK r23-beta3, `libgcc.a` was removed and replaced by `libunwind.a`.
Rust 1.68+ started passing `-lunwind` as a linker flag for Android targets. This causes:

- "ld: library not found for -lgcc" on older NDK versions.
- Possible duplicate symbol errors on newer NDK versions if both are pulled in.

The workaround frequently cited is: copy `libunwind.a` and symlink/rename it as `libgcc.a`.
This is a build-system-level hack that must be encoded into the Bazel `BUILD.bazel` or the
`android_ndk_repository` setup.

Recommended NDK version: **r25c or r26b** (LTS). NDK r27 is the latest as of 2025 but has
not been widely validated against `rules_rust`-based Rust cross-compilation.

#### 4c. `android_ndk_repository` vs `rules_android_ndk`

Bazel 7.4+ recommends migrating from the built-in `android_ndk_repository` to the standalone
`rules_android_ndk` Bzlmod extension. The built-in rule is being removed. The Bzlmod extension
syntax differs:

```python
# Old (WORKSPACE)
android_ndk_repository(name = "androidndk", path = "/opt/ndk/r25c")

# New (MODULE.bazel with rules_android_ndk)
bazel_dep(name = "rules_android_ndk", version = "...")
android_ndk_repository = use_extension("@rules_android_ndk//:extensions.bzl", "android_ndk_repository")
android_ndk_repository.remote(api_level = 30)
```

Mixing the old and new styles in Bazel 9.1.1 (the project's pinned version) is unsupported.

#### 4d. Tokio and `async-std` pull in `pthread` dependencies

On Android, POSIX thread primitives are in `libc` (no separate `libpthread`). Older Rust
toolchains and crates that explicitly link `-lpthread` will fail the Android linker. Tokio's
`mio` backend uses `epoll` (Linux) which is fine, but any crate that links `-lpthread` or
`-lrt` explicitly will need a NDK-specific override.

**References:**
- [rules_rust issue #2518](https://github.com/bazelbuild/rules_rust/issues/2518)
- [rules_rust issue #2529](https://github.com/bazelbuild/rules_rust/issues/2529)
- [rust-lang/rust issue #108943 — -lunwind on Android](https://github.com/rust-lang/rust/issues/108943)
- [Bazel Android NDK docs](https://bazel.build/docs/android-ndk)

## 5. Database Closed Mid-Query: Replicating `catchDbError()`

### Existing pattern (Kotlin side)

The current `DbFlowExtensions.kt` centralizes the closed-DB guard:

```kotlin
internal fun <T> Flow<Either<DomainError, T>>.catchDbError(): Flow<Either<DomainError, T>> =
    catch { e ->
        if (e is CancellationException) throw e
        emit(DomainError.DatabaseError.ReadFailed(e.message ?: "database closed").left())
    }
```

This catches any `Throwable` emitted by a `Flow` — including `IllegalStateException` thrown by
the SQLDelight driver when the underlying JDBC connection pool is closed.

### How the libsql JNI driver differs

The `JvmLibsqlDriver` does not use JDBC. Its connection pool is managed entirely in Kotlin
(or Rust). When `GraphManager.shutdown()` calls `JvmLibsqlDriver.close()`, in-flight operations
can fail in several ways:

1. **Rust side returns error code.** The JNI function can detect the closed state (via the
   `AtomicBool` closed flag recommended in §2) and return a sentinel indicating "driver closed".
   The Kotlin JNI wrapper converts this to a thrown `IOException`.

2. **Tokio future cancelled.** If `GraphManager.shutdown()` shuts down the Tokio runtime before
   all in-flight `block_on` calls complete, those calls will block until the runtime drains.
   Do **not** call `Runtime::shutdown_background()` or `Runtime::shutdown_timeout()` from the
   JNI `close` function while a `block_on` call is still live on another thread — this deadlocks
   on the current-thread runtime.

3. **Pool connection returned after close.** A connection checked out before `close()` may
   complete its query and attempt to return the connection to a closed pool. The pool must
   handle this gracefully (return without panic).

### Required replication strategy

The `JvmLibsqlDriver` must:

1. Maintain an `AtomicBoolean closed` field. Set it to `true` in `close()` before waiting for
   in-flight calls.
2. Every public `SqlDriver` method (`execute`, `executeQuery`, `newTransaction`) checks `closed`
   before acquiring a pool connection and throws `IllegalStateException("driver is closed")` if
   set. This propagates through the `asDbFlowList` / `asDbFlowOrNull` chains to `catchDbError()`
   exactly as the JDBC driver does today.
3. The Rust-side JNI entry points for `dbQuery` and `dbExecute` also check the closed flag on
   their handle struct and return an error code rather than a result, so the Kotlin layer can
   throw before any `Box::from_raw` dereference.
4. `close()` must coordinate with the connection pool: drain all checked-out connections before
   returning, using a `CountDownLatch` or `Semaphore` to wait for in-flight queries to complete.

### `CancellationException` preservation

The existing `catchDbError()` correctly re-throws `CancellationException` without wrapping.
The libsql driver must not swallow coroutine cancellations either. Any `InterruptedException`
or `CancellationException` observed in the Kotlin wrapper layer must propagate upward unchanged.

## 6. Tokio Runtime Thread Safety with Concurrent JNI Calls

### Architecture recap (from requirements)

The plan specifies a global `Runtime` with 2 worker threads, created once with
`Runtime::new()` and stored in a `static OnceLock<Runtime>`. Each JNI function calls
`runtime.block_on(async { ... })`.

### Is `block_on` safe from multiple OS threads?

`Runtime::block_on` is **not** callable concurrently from multiple threads on the same
`Runtime` instance when using the `current_thread` scheduler. The first call takes ownership of
the I/O and timer drivers; concurrent callers block waiting for the driver to be released.

With the **multi-thread** scheduler (the default for `Runtime::new()`):
- `block_on` can be called from multiple threads concurrently.
- Each call runs the supplied future to completion on the calling OS thread, using the shared
  worker-thread pool for async work.
- The global runtime is `Send + Sync`, so sharing via `static OnceLock<Runtime>` is safe.

However, two subtle problems arise:

#### 6a. `block_on` on a multi-thread runtime from 8+ JVM pool threads

With an 8-connection pool and potentially 8 concurrent JVM threads each calling `block_on`,
the Tokio runtime's 2 worker threads become a bottleneck: the 8 JVM threads are each blocked
waiting for their future to complete, but only 2 Tokio workers are making progress on the async
futures. This does not cause unsafety but significantly degrades throughput — the pool's
concurrency is effectively limited to 2.

Mitigation: size the Tokio worker thread count to match or slightly exceed the JVM connection
pool size. For an 8-connection JVM pool, use `Runtime::new()` with
`.worker_threads(8)` in the builder, or use `tokio::runtime::Builder::new_multi_thread()`.

#### 6b. `Handle::block_on` vs `Runtime::block_on`

If the global is stored as a `Handle` (from `Runtime::handle().clone()`), note that
`Handle::block_on` does **not** drive the runtime — there must be at least one live
`Runtime::block_on` call active for the current-thread scheduler to make progress. For the
multi-thread scheduler, `Handle::block_on` works correctly because background worker threads
are always running.

The safe pattern: store `Arc<Runtime>` (not just `Handle`) in the `OnceLock` and call
`runtime.block_on(...)` from each JNI function.

#### 6c. No async across JNI boundary — no `spawn` leaks

JNI functions are called on JVM-managed threads that are not Tokio tasks. Calling
`tokio::spawn(...)` inside a JNI function is safe (the spawned task runs on Tokio workers)
but the task must not capture `JNIEnv` or any JNI local references — those are thread-local
and the task may be polled on a different thread. The requirements' design choice of
`block_on` + `Vec<Vec<Value>>` eager collection sidesteps this entirely. Maintain this design:
**never return a `JoinHandle` or spawn a detached task from a JNI function**; always use
`block_on` to completion.

#### 6d. Runtime destruction order

If the global `Runtime` is stored in a `static`, Rust's static destructors run after `main`
returns. On Android, the process is typically killed by the OS rather than allowed to exit
cleanly, so this is not a concern. On JVM desktop, `System.exit()` also skips Rust static
destructors. The `close()` JNI call must explicitly shut down the runtime (or drop it) before
the JVM exits to avoid resource leaks.

## Summary Risk Table

| # | Risk | Severity | Mitigation |
|---|---|---|---|
| 1 | Rust panic crashes JVM with `panic=abort`, uncatchable | **Critical** | Use `catch_unwind` at every JNI entry; keep `panic=unwind` |
| 2 | Double-free / use-after-free on Box handle | **Critical** | CAS null-sentinel in Kotlin; closed flag in Rust struct |
| 3 | `BEGIN CONCURRENT` not available in libsql local mode | **Critical / Feature-Blocking** | Reframe to non-MVCC or switch to Turso Rust rewrite |
| 4 | Bazel rules_rust Android Bzlmod cross-compilation gaps | **High** | Pin NDK r25c; use `rules_android_ndk`; manual toolchain wiring |
| 5 | Driver close while mid-query bypasses `catchDbError()` | **High** | `AtomicBoolean closed` + drain latch; closed check before `Box::from_raw` |
| 6 | Tokio worker thread count mismatch with JVM pool size | **Medium** | Set `worker_threads` = pool size; use `Arc<Runtime>` not `Handle` |
