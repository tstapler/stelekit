# Fix line Editing in Stelekit sometimes will select the line instead of entering edit mode (Priority 3 | Status: in_progress)

## Description
When I'm editing my graph on stelekit web I'm encountering that it will enter the "multi select" mode instead of editing an existing line when I click on it using the track pad we need to figure out and fix this to make sure that our selection features and editing do not conflict and cancel each other out

## Acceptance Criteria
0. [✓] A fast tap on an existing block's text enters edit mode (not selection mode) on stelekit web
1. [✓] A genuine long-press (>500ms, no movement) on a block row enters selection mode when not already selecting
2. [✓] Repeated fast trackpad clicks on stelekit web reliably enter edit mode under CPU-throttled conditions (bug reproduction eliminated)
3. [✓] Shift+click on a block still extends the selection range instead of entering edit mode
4. [✓] Shift+ArrowUp/Down still extends selection by one block (regression check)
5. [✓] Ctrl/Cmd+A select-all and Ctrl/Cmd+C/X copy/cut still work while in selection mode (regression check)
6. [✓] Gutter checkbox still toggles selection when already in selection mode (regression check)
7. [✓] Gutter drag-handle still reorders blocks on desktop/Android without unexpected mode entry (regression check)
8. [✓] Right-click on block text still opens the link/URL context menu (regression check)
9. [✓] Tapping a wiki-link/tag/URL span inside block text still navigates/opens the link instead of entering edit mode
10. [✓] Android's suppression of row-level long-press-to-select (in favor of the gutter's drag-after-long-press) is unaffected by this fix
11. [✓] Non-default block types (heading, code fence, image annotation, etc.) enter edit mode on tap as reliably as plain text blocks
